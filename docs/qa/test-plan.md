# PMS 실사용 테스트 계획서 (Master Test Plan)

> 환자 관리 시스템(PMS) — 기능 명세(PRD FR-001~243 · NFR) 전수 기반 E2E/수동 테스트 계획.
> 8에픽 / 60스토리 구현·코드리뷰 완료 후 **첫 실사용 검증**을 위한 체계적 시나리오 집합.
>
> - **상세 시나리오:** [`docs/qa/scenarios/epic-1.md` ~ `epic-8.md`](./scenarios/) — 총 **423개** 시나리오
> - 각 시나리오는 명세 AC ↔ 실제 구현 코드를 전수 대조해 작성됨(추측 아님).

---

## 1. 목적과 범위

| 항목 | 내용 |
|---|---|
| 목적 | 명세된 모든 기능(FR/NFR)이 **실제 화면·API·DB에서 의도대로 동작**하는지 빠짐없이 검증 |
| 범위 | 8에픽 전 기능 + 횡단 역량(RBAC·RLS·암호화·감사·상태머신) + 6역할 통합 흐름 |
| 검증 계층 | ① 자동 통합/단위 테스트(이미 통과: API 973 · web 541) ② **수동 E2E(본 계획서의 초점)** |
| 산출물 | 본 계획서(전략·환경·골든패스·추적) + 에픽별 상세 시나리오 8종 |

**자동 테스트와의 역할 분담** — 통합테스트(`api/tests`, `web`)가 RLS·IDOR·산정 공식·상태전이를 이미 코드 레벨에서 검증한다. 따라서 수동 E2E는 **자동 테스트가 못 보는 영역**에 집중한다: 실제 화면 UX·역할별 셸 노출·브라우저 인쇄(영수증/처방전)·실시간 대기판 갱신·APK 셸·환자 계정 연결 플로우·6역할이 한 내원을 가로지르는 통합 흐름.

## 2. 테스트 환경 & 사전 준비

### 2.1 환경 선택

| 환경 | 주소 | 용도 | 비고 |
|---|---|---|---|
| 로컬 | API `:8060` · web `:3002` | 전체 시나리오(DB 직접 조작·상태 셋업 가능) | `supabase db reset` 로 클린 시드 |
| 클라우드 | https://kuntae802.mooo.com/patient_management_system | 배포 형상 그대로 검증 | `db push`(마이그만) — **seed/demo_seed 미적용**, 데모 계정·환자 없음 주의 |

> **권장:** 기능 전수 검증은 **로컬**에서 한다(상태 셋업·DB 확인 자유). 클라우드는 배포 정합성(라우팅·HTTPS·셸) 스모크 테스트용.

### 2.2 클린 시드 절차 (로컬)

```bash
# 1) DB 초기화 + 마이그(0001~0055) + seed.sql(마스터·계정) 적용
supabase db reset

# 2) 임상 더미 데이터(환자·예약·내원·오더·수가) 적재 — 재실행 가능
docker exec -i <supabase_db> psql "$SUPABASE_DB_URL" -v ON_ERROR_STOP=1 -f - < supabase/demo_seed.sql

# 3) API·web 기동, ⚠️ db reset 직후 auth/kong 재시작 지연으로 502/토큰실패 가능 → 30초 대기 후 로그인
```

> ⚠️ **절대 금지:** `supabase db reset --linked` (클라우드 DB 전체 초기화됨). seed.sql은 로컬 reset에서만 실행된다.

## 3. ⚠️ 실행 전 필독 — 알려진 제약·함정

> 명세-구현 전수 대조에서 발견된, **테스트 셋업/판정에 직접 영향**을 주는 항목. 모르고 실행하면 "정상인데 버그로 오판" 하기 쉽다.

| # | 함정 | 영향 / 대응 |
|---|---|---|
| F1 | **reception 에 `patient.create/read/update` 권한 없음** (시드 누락 · Finding#1) | 원무가 환자 등록/검색을 못 함(메뉴는 보이는데 403). → **골든패스 사전 스텝에서 admin이 RBAC 화면으로 부여**(FR-211 즉시반영 검증 겸). 실제 가능 역할은 admin(전부), doctor(read+reveal만) |
| F2 | **데모 환자 20명 전원 `auth_uid` = NULL** | 환자 포털(Epic 8)은 시드 환자로 바로 못 들어감. → `/signup` 자가가입 → `/onboarding` self-link(주민번호 blind_index + 성명 매칭)로 한 환자를 연결한 뒤 그 계정으로 검증 |
| F3 | **시드에 scheduled/cancelled/no_show 내원 없음** | register(예약→접수)·cancel·no_show 경로는 psql 직접 INSERT 또는 예약→접수 흐름으로 셋업 필요 |
| F4 | **lab(검체) 검사 실 수행 경로 부재** | `perform_examination` 는 imaging 전용(lab=422 `not_imaging`), 간호 측 검체수행 UI 없음. demo_seed는 lab을 직접 UPDATE로 전이. → 명세("검체→간호 워크리스트")와 구현 갭으로 **확인 후 기록** |
| F5 | **오버부킹 정책 토글 미구현** | FR-013 "오버부킹 정책 시 허용"은 코드에 없음(더블부킹 하드 차단이 유일 동작). 갭으로 기록 |
| F6 | **약제비(약가) 미발생** | 처방 발행해도 수가 0(drugs에 약가 없음·원외 약국·설계상 제외). 수납 금액에 약값 미반영이 정상 |
| F7 | **진찰료 초진/재진 동적 판정** | 과거 완료 내원 유무로 AA154(초진)/AA254(재진) 자동 분기. 신규 환자=초진. 같은 환자 2회차=재진 |
| F8 | **demo finalized 6건(e01~06)은 total=copay=paid 하드코딩** | 실제 본인부담 산정값이 아님. 산정 정확성 검증은 e07/e08 또는 신규 내원을 build→finalize |
| F9 | **비급여·자보(0%) 오더 demo에 없음** | 비급여 100%/자보 0% 본인부담 검증은 해당 수가(F6310/MM070 등)·자보 환자 수동 오더 필요 |
| F10 | **세션당 활성 내원 1개 가드 = 브라우저 localStorage** | 서버 락 아님. 충돌/superseded 재현은 **같은 브라우저 두 탭**에서만. 오환자 차팅 차단의 핵심 안전선 |
| F11 | **실시간 대기판 = postgres_changes 구독 + 30초 폴링 백스톱** | db reset 후 realtime 끊김 가능. ≤5초 실시간 갱신과 30초 폴링 복구를 구분해 판정 |
| F12 | **미들웨어 = `web/src/proxy.ts`** (Next 16 변경, `middleware.ts` 아님) | 미들웨어 가드/RBAC UI 게이트 검증 시 참조 위치 |
| F13 | **문서 출력 = 브라우저 인쇄(`window.print`)** | 인쇄 시 `beforeprint`가 export 감사 RPC 호출. APK 웹뷰에서는 print 미동작(한계) |
| F14 | **APK 런타임 테스트는 사용자 수용 단계** | 빌드·analyze·단위테스트는 검증 가능. 뒤로가기·로딩/오류·세션지속은 Android 기기 필요(`~/patient-portal-app-release-v1.0.0.apk`) |

## 4. 핵심 E2E 골든패스 — "한 번의 내원, 6역할 가로지르기" (UJ-1)

> 가장 중요한 통합 시나리오. **신규 환자 1명이 예약/접수 → 진료 → 수행 → 수납 → 포털 조회**까지 6역할을 순서대로 흐른다. 각 단계는 에픽별 상세 시나리오로 연결된다. 이 흐름이 끊김 없이 완주되면 시스템의 척추(내원 파이프라인)가 검증된 것이다.

**사전 준비 (1회):**
- `db reset` + `demo_seed.sql` 적용, API·web 기동, 30초 대기
- **(admin)** 로그인 → `/admin/permissions` → **reception 역할에 `patient.create`·`patient.read`·`patient.update` 체크 → 저장 없이 즉시 반영 확인** (F1 우회 + FR-211 검증 → [TC-E1-16~20])

|---|---|---|---|---|
| 1 | admin | `/admin/masters` 마스터 확인(IM 진료과·의사 근무표), `/admin/users` 직원 확인 | 마스터·직원 정상 노출 | TC-E2-01·, TC-E1-25· |
