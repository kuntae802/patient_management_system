# 환자 관리 시스템(PMS) — 요구사항 및 설계 명세

> **이 문서에 대하여.** 이 문서는 "무엇을 만들었는가(요구사항)"와 "무엇을 고려해 그렇게 만들었는가(설계 결정)"를 함께 담은 제품 설명서다. 기능 목록을 건조하게 나열하는 대신, 각 결정의 **배경·근거·트레이드오프**를 드러내 — 모호한 한 줄 요청을 현실의 결을 가진 시스템으로 구체화한 과정 — 을 보이는 데 목적이 있다.
>
> 구성: ①제품 개요 → ②문제와 접근 → ③기능 요구사항(도메인별) → ④비기능 요구사항 → ⑤핵심 설계 결정 → ⑥범위와 한계 → ⑦요구사항 추적표.

---

## 1. 한눈에 보기

수십 년 업력의 지역 중소병원이 **세대교체**를 맞고 있다. 외래 운영을 떠받쳐 온 베테랑들의 워크플로우는 어디에도 적혀 있지 않고, 그들의 머릿속과 손버릇 — 암묵적 관성 — 위에서만 돌아갔다. 고참이 빠지고 신규 인력이 채워지자 접수·처치·수납·예약 전반이 삐끗하기 시작했다. 클라이언트의 요청은 "EMR 하나 깔아 달라"가 아니라, **흩어진 암묵적 워크플로우를 사람이 바뀌어도 무너지지 않는 명시적·표준 운영 체계로 굳혀 달라**는 것이다.

**환자 관리 시스템(PMS)** 은 그 요청에 답하는 외래 운영 관리 앱이다. 핵심 아이디어는 단순하다 — 환자의 한 번의 내원(진료, Encounter)을 예약부터 수납까지 시스템이 단계별로 안내하는 **명시적·역할별 파이프라인**으로 만들어, 워크플로우가 사람의 기억이 아니라 시스템 안에 살게 하는 것이다.

| 항목 | 내용 |
|---|---|
| **사용자** | 직원 5역할(원무·의사·간호사·방사선사·관리자) + 환자 — 총 6역할 |
| **플랫폼** | 직원 = 데스크톱 웹 / 환자 = Android 앱(APK·웹뷰 셸) |
| **백엔드** | Supabase(PostgreSQL·Auth·Storage·RLS·pgcrypto·Vault) + FastAPI(오케스트레이션) |
| **프런트** | Next.js(직원 웹) + Flutter(환자 앱 셸) |
| **합격선** | 6역할을 가로지르는 데모 골든패스(예약→접수→진료→수행→수납→관리)가 끊김 없이 동작 |

**골든패스(이 시스템이 통한다는 증거).** 환자 앱 예약 → 원무 접수 → 의사 진료(SOAP)·처방·검사 오더 → 간호 활력·처치 수행 → 방사선사 촬영·판독 → 원무 수납(표준 진료비 문서 출력) → 관리자 권한 토글·감사로그 확인.

## 2. 문제와 접근 — 왜 이렇게 만들었나

### 2.1 문제: 세대교체 앞에서 무너지는 암묵적 워크플로우

이 병원의 외래 워크플로우는 어디에도 적혀 있지 않다. 접수 순번, 대기 호출, 오더 인계, 수납 코드, 예약 조정까지 거의 모든 절차가 "원래 그렇게 해 왔으니까"로 굴러간다. 사람이 바뀌지 않는 동안엔 놀랄 만큼 잘 작동했지만, **지금 사람이 바뀌고 있다.** 신규 인력 위에서 암묵 체계가 삐끗하는 장면은 동선 전체에서 동시다발로 터진다:

- **접수·대기** — 순번·호출 규칙을 몰라 환자를 두 번 부르거나 건너뛴다.
- **간호·처치** — 구두 오더가 떠돌아 처치를 **중복하거나 누락**한다(환자 안전 직결).
- **수납·정산** — 베테랑만 알던 수가 코드를 틀려 정산 오류가 난다.
- **예약** — 장부·구두로만 존재해 더블부킹·노쇼 관리가 무너진다.
- **조직 리스크** — 핵심 업무가 한 사람에게 묶여(버스 팩터 1) 그의 부재가 곧 마비다.

### 2.2 접근: 워크플로우를 시스템에 새겨 넣는다 — 일곱 기둥

한 번의 내원을 **추적되는 워크플로우 한 건**으로 만든다. 각 기둥은 위 고통 하나에 정확히 대응한다.

| 기둥 | 무엇 | 겨냥하는 고통 |
|---|---|---|
| ① 명시적 진료 파이프라인 | 내원이 정해진 상태(예약→접수→진행중→완료)를 따라 흐르고, 화면이 늘 "다음 할 일"을 보여줌 | 접수·대기 혼선 |
| ② 역할별 가이드 화면 + RBAC | 6역할이 각자 자기 단계만 봄. 권한은 관리자가 코드 수정 없이 토글 | 역할 혼선·구두 의존 |
| ③ 오더 → 수행 추적 | 처방·검사·처치가 "지시(의사)→수행(약국·방사선사·간호사)"로 흐르고 누가·언제·무엇을 기록 | 처치 중복·누락 |
| ④ 단일 진실 마스터 | 수가·약품·진단코드는 표준 마스터에서만 선택(자유 입력 차단) | 수납·정산 오류 |
| ⑤ 표준 기록 자동화 | SOAP 임상기록 + 한국 표준 「진료비 계산서·영수증」·「세부산정내역서」를 데이터에서 자동 생성 | 기록 분실·추적 불가 |
| ⑥ 멀티플랫폼 + 환자 포털 | 직원은 웹, 환자는 앱에서 직접 예약·본인 내역 조회 | 예약 혼선·원무 과부하 |
| ⑦ 보안·신뢰 | 환자는 본인 데이터만(RLS), 주민번호 암호화(pgcrypto+Vault), 주요 작업 감사로그 | 민감정보 안전 |

**관통하는 한 줄:** 일곱 기둥은 모두 같은 일을 한다 — 사람 머릿속의 **암묵지를 시스템 안의 명시지로** 옮긴다.

## 3. 기능 요구사항 — 무엇을 (도메인별)

> 전체 70개 기능 요구사항(FR)을 14개 도메인으로 묶었다. 각 도메인은 "핵심 기능 + 까다로웠던 점"을 함께 적었다. 번호 추적은 §7 참조.

### 3.1 신원·등록·인증 (FR-001~006)
환자는 모바일 앱에서 자가가입(본인인증 시뮬)하거나 원무가 직접 레코드를 생성한다. **까다로운 점:** 앱을 안 쓰는 환자(전화·고령자)도 시스템에 등록되어야 해서 `auth_uid`를 nullable로 두고, 나중에 앱 가입 시 **주민번호 blind index(HMAC)로 기존 레코드를 자동 매칭**해 중복 환자를 막는다. 임상 프로필(혈액형·알레르기·기저질환·복용약)·보호자 정보를 함께 관리한다.

### 3.2 예약·일정 (FR-010~016)
환자는 근무표·휴진이 반영된 가능 슬롯(30분 단위)을 조회해 예약하고, 원무는 전화·방문 환자를 대리 예약한다. **까다로운 점:** 더블부킹은 DB 차원에서 하드 차단(409)하고, 노쇼 횟수를 추적해 임계치(기본 2회) 초과 시 예약을 제한한다. 관리자가 휴진·예외를 등록하면 영향받는 예약을 검출해 재배정을 돕는다. SMS 리마인더(D-3·D-1)는 시뮬레이션·발송 이력으로 기록한다.

### 3.3 접수·대기·호출 (FR-020~023)
원무가 예약 환자나 walk-in을 접수하면 내원이 생성되고 진료과 대기열에 등록된다. 실시간 대기 현황·순번을 제공하고, "다음 호출" 안내와 호출 기록으로 중복·누락을 막는다. **개선 반영:** 원무 대기 현황판은 진료과별 + **전체 진료과** 보기를 지원하며(원무 기본=전체), 전체 보기 시 각 환자 행에 진료과 배지를 붙여 어느 과 대기인지 한눈에 보인다.

### 3.4 진찰·진료·기록 (FR-030~042)
의사는 진료 대기열에서 환자를 선택해 진찰을 시작(진행중)하고, **진료 허브(3-pane: 좌 컨텍스트·중앙 SOAP·우 오더)** 에서 과거 이력·활력을 한 화면에서 본다. SOAP 진료기록(주관적·객관적·평가·계획)을 쓰고, 평가에 KCD 진단을 부착해 주/부상병을 구분한다. **까다로운 점:** 한 내원에 복수 기록이 가능하고, **주상병이 없으면 완료가 불가**하다(422). 진료 완료 자체는 의사가 누르지 않고 수납 finalize가 트리거한다(§5.7 참조).

### 3.5 오더·처방 (FR-050~052)
의사가 약품 마스터에서 약을 골라 처방전을 발행(약품·용량·횟수·일수·용법)하고, 처방↔진단을 연결해 근거를 남긴다. 동일 성분 중복 처방은 경고하고, **환자 기록 알레르기와 약품명이 매칭되면 사유 입력 없이는 발행을 차단**한다. 처방은 '발행→발급(원외 약국)' 생명주기를 따른다.

### 3.6 검사·영상·처치 오더 (FR-060~061, 070, 090~094, 100~103)
의사가 검사·영상·처치를 오더하면 '지시(ordered)' 상태로 생성되어 직역별 워크리스트로 분기된다 — 영상→방사선사, 처치→간호. 간호사는 활력징후(혈압·맥박·체온·호흡·SpO₂)를 기록하고 처치를 수행하면 오더가 '수행'으로 전환된다. **재수행 차단**으로 중복·누락을 막고, 방사선사가 촬영·업로드(영상은 Storage·URL만 DB)하면 의사가 판독해 검사를 완료한다. **구현 정리:** 진단검사(검체)는 채취 수행 주체가 시스템 범위 밖(외부 의뢰)이라 의사 진료허브의 "검사" 탭을 제거하고 **영상·처치만 수행 경로**로 운영한다(검사 인프라는 향후 확장 위해 유지).

### 3.7 오더 생명주기·자동 청구 (FR-080~081)
모든 오더는 유형별 상태전이(처방: 발행→발급, 검사·영상: 지시→수행→판독/완료, 처치: 지시→수행→완료)를 따르며 지시자·수행자·시각이 기록된다. **상태머신이 역행·건너뛰기를 막고**, 수행 시점에 수가가 자동 발생한다. **개선 반영:** 잘못 낸 **미수행 오더는 취소(cancelled)** 할 수 있다 — 수행분은 정산에 영향을 주므로 차단하고, 미수행만 취소되며 수가에는 영향이 없다.

### 3.8 수납·정산·청구 (FR-110~119)
수행된 오더에서 자동 적재된 수가를 모아 수납 건을 만들고, 급여/비급여를 구분해 본인부담을 산정한다. 원무가 결제(수단·금액)를 기록하면 내원이 완료된다. 시스템은 표준 「진료비 계산서·영수증」·「세부산정내역서」·원외처방전을 **법정 서식(바탕체)** 으로 출력하고, **재출력을 위한 수납 내역 검색**(환자명·차트·영수증번호·기간)을 제공한다. 후/선수납을 지원하고, 취소·노쇼는 수가 미발생, 부분 수행은 수행분만 정산한다.

### 3.9 환자 포털 (FR-120~122)
환자는 앱에서 본인 내원 이력·처방·검사 결과·수납 영수증을 조회한다(RLS로 본인만). **까다로운 점:** 화면은 임상 용어 대신 **쉬운 말로 풀어** 보여주고(예: 검사 결과를 "정상/주의"로), 모바일 전용 UI·12시간 표기를 쓴다.

### 3.10 마스터 데이터 (FR-200~203)
관리자가 진료과·진료실·KCD 진단·EDI 수가·약품 마스터를 관리하고, 모든 임상·정산 입력은 마스터 선택을 강제해 자유 입력을 막는다. 마스터는 **soft delete(비활성)** 로 처리해 과거 기록의 참조 무결성을 지킨다.

### 3.11 역할·권한·신원 (FR-210~215)
관리자가 역할↔권한(N:M, `리소스.동작`)을 **체크박스 토글**로 바꾸면(코드 수정 없이) 즉시 적용되고 감사에 남는다. 직원/환자는 분리 프로필로 구분되어 로그인 시 소속 테이블로 분기된다. 접근 제어는 **3계층**(화면 노출 / FastAPI 명령 / RLS 행)으로 겹쳐 강제되며, 직원 재직상태(재직·휴직·퇴사)가 접근에 정확히 반영된다.

### 3.12 근무표·스케줄 (FR-220~221)
관리자가 의사별 근무표(요일·시간·진료실)와 휴진·예외를 등록하면, 예약 슬롯이 **근무표 − 예외 − 기예약**으로 동적 계산된다.

### 3.13 운영 대시보드 (FR-230)
관리자가 운영 현황(내원·대기·매출·노쇼율)을 본다. **개선 반영:** 당일 스냅샷에 더해 **조회 기준일 선택(일별)** 과 **기간 합계(월간 수치)** · 추세 그래프를 제공한다.

### 3.14 보안·감사 (FR-240~243)
RLS가 "환자는 본인만, 직원은 역할별만"을 DB 차원에서 강제한다. 주민번호는 pgcrypto로 암호화(키는 Vault)하고, 주요 작업은 행위자·시각·대상·전후 스냅샷으로 감사된다. 감사로그는 **append-only**(UPDATE/DELETE 전역 차단)이며, 주민번호는 화면 기본 마스킹 + 권한 게이트 reveal(조회 자체가 감사 이벤트)로 다룬다. **개선 반영:** 감사 목록의 대상에 비-PII 식별명(역할명·직원명+사번 등)을 함께 보여 가독성을 높였다.

## 5. 무엇을 고려해 이렇게 구현했나 — 핵심 설계 결정

> 이 절이 이 문서의 핵심이다. 각 결정을 **고려 → 결정 → 트레이드오프** 순으로 적는다. 관통하는 테제는 하나다 — **"불변식은 DB가 소유하고, 앱은 오케스트레이션만 한다."** 상태머신·권한·수가·감사가 모두 DB에서 강제되므로, 개발자가 실수해도 DB가 최종 방어선이 된다.

### 5.1 스택 — Supabase · FastAPI · Next.js · Flutter
**고려.** 직원 데스크톱·환자 모바일·백엔드를 모두 다뤄야 했고, "실무 감각을 익히는 것"이 학습 목표였다.
**결정.** 관리형 데이터·인증(Supabase) + 비즈니스 로직(FastAPI) + 반응형 웹(Next.js) + 네이티브 셸(Flutter). Supabase가 RLS·pgcrypto·Storage·Auth를 한 묶음으로 제공해 "DB가 불변식을 소유한다"는 아키텍처를 가능케 한다.
**트레이드오프.** ORM을 쓰지 않아(Pydantic + 직접 SQL) 타입 동기화 책임이 개발자에게 있지만, **스키마 단일 소유(마이그레이션)** 와 명확한 경계(쓰기=FastAPI / 읽기=Supabase 직접·RLS)를 얻는다.

### 5.2 분리 프로필(직원/환자)
**고려.** 직원과 환자는 권한·데이터 접근 모델이 완전히 다르다. 한 계정을 양쪽에 쓰면 RLS·감사 경계가 흐려진다.
**결정.** `users`(직원·`id`=auth uid)와 `patients`(`auth_uid` nullable)를 분리하고, 로그인 후 uid가 어느 테이블에 매핑되는지로 직원/환자 라우트를 가른다.
**트레이드오프.** 환자 가입 시 기존 레코드를 찾아 연결하는 로직이 필요하지만, 겸용 계정의 권한 충돌(같은 내원을 의사로도 환자로도 접근)을 원천 차단한다.

### 5.3 DB 기반 RBAC + 행수준보안(RLS) — 왜 앱이 아니라 DB인가
**고려.** 인가를 앱이 담당하면 쿼리 재구현·우회 시 논리 구멍이 누적된다. 의료는 보안을 구부릴 여지가 없다.
**결정.** `role_permissions`(N:M)를 DB가 소유하고 `has_permission()` 헬퍼가 진실 원천. RLS 정책이 행 접근을 강제하고, FastAPI의 쓰기 위임은 항상 `SECURITY DEFINER` RPC를 경유한다. **3계층**(화면 노출 / FastAPI 권한 / RLS 행)이 겹쳐 한 층이 뚫려도 나머지가 막는다.
**트레이드오프.** DB 로직이 복잡해지고 TOCTOU(권한 평가↔쓰기 분리)를 이해해야 하지만, 인가가 새지 않는다.

### 5.4 주민번호 암호화 · 마스킹 · reveal 게이트
**고려.** 주민번호는 최고위험 PII다. 평문 저장은 DB 침해 시 통째 노출이고, 조회를 추적하지 못하면 무단 열람을 알 수 없다.
**결정.** `pgcrypto`로 컬럼 암호화(키는 Vault·코드에 평문 없음), **HMAC blind index**로 복호 없이 중복 매칭, 복호 호출 자체를 감사(`read`)로 기록. 화면은 기본 마스킹(`710314-2******`)하고 권한 게이트로만 reveal한다.
**트레이드오프.** Vault 미설정 시 조용히 실패할 수 있어 함수에 키 가드를 두고, 주민번호 정규화를 소비처가 책임지도록 명시했다.

### 5.5 진료 허브 + 내원 상태머신
**고려.** 내원은 `scheduled→registered→in_progress→completed`를 따라야 하고, 역행·건너뛰기가 절대 안 된다 — 이 전이가 수가·수납의 근거다.
**결정.** `status`에 CHECK 제약을 걸고, **전이 트리거**가 허용 매트릭스를 강제하며, 각 전이는 권한 게이트가 달린 **전이 RPC**로만 수행된다(직접 UPDATE 금지). 전이 시각도 RPC가 기록한다.
**트레이드오프.** enum 대신 text+CHECK로 통일해(상태 추가/제거가 쉬움) 타입 안전을 약간 양보하고 마이그레이션 유연성을 택했다.

### 5.6 오더 → 수행 분리 + 재수행 차단
**고려.** 의사가 지시(ordered)하고 다른 직역이 수행(performed)한다. 실수로 두 번 수행되면 수가가 중복 청구된다.
**결정.** `ordered_by`(지시자)와 `performed_by`(수행자)를 분리 기록하고, 수행 RPC가 **소스 상태가 `ordered`임을 선검사**해 재수행을 막는다(전이 트리거가 최종선).
**트레이드오프.** 미수행 오더가 대기열에 떠 있어야 해서 UX에서 "다음 수행 대상"을 명확히 해야 한다. 대신 누가·언제 수행했는지 완전히 추적된다.

### 5.7 수가 자동발생 트리거 + billing-completes
**고려.** 진찰료는 진찰 시작 시, 검사·처치료는 수행 시점에 결정돼야 한다. 수동 입력은 실수·누락·중복이 잦다.
**결정.** DB 트리거가 상태 전이(진찰 시작·수행)에 반응해 `fee_items`를 **멱등하게**(unique(source_type, source_id)) 적재한다. 진찰료는 초진/재진을 동적으로 판정하고, 본인부담은 보험유형별로 산정한다. 결제 finalize가 같은 트랜잭션에서 `complete_encounter`를 호출 — **"결제 완료 = 내원 종결"(billing-completes)** 이라, 의사는 진료 완료 버튼을 누르지 않는다(정산 누락 방지).
**트레이드오프.** 임상 이벤트와 청구가 강하게 결합된다. 수가를 안 내는 항목은 매핑에서 빼고, 약제비(약가 데이터 없음)는 원외 처방으로 분리해 정직하게 명시했다.

### 5.8 표준 문서 자동 생성 — 브라우저 인쇄
**고려.** 의료기관 청구는 국가 표준 서식(진료비 계산서·세부산정내역서·영수증)을 써야 한다. 수기 작성은 오류·불일치가 많다.
**결정.** 수납 집계 데이터(`payment_details`)에서 문서를 **브라우저 인쇄**(`window.print`)로 렌더링한다 — 별도 PDF 엔진 의존 없이, 인쇄용 영역에 **바탕체(법정 서식 관례)** 를 적용하고 화면 UI는 Pretendard를 쓴다. **인쇄/내보내기 자체가 감사 이벤트**로 기록된다.
**트레이드오프.** 서버 PDF 캐시가 없어 매번 렌더링하지만, 의존성이 가볍고 단순하다. 캐시는 후속 성능 작업으로 이월.

### 5.9 감사 append-only — 3중 잠금
**고려.** 의료는 규제 산업이다. 모든 작업이 추적되어야 하고, 감사로그 자체가 조작되면 안 된다.
**결정.** 트리거가 INSERT/UPDATE/DELETE를 자동 기록(행위자·동작·전후 스냅샷)하고, 변조 방지를 **3계층**으로 건다 — ① RLS가 UPDATE/DELETE를 deny ② service_role 포함 전 역할에서 UPDATE/DELETE 권한 REVOKE ③ SECURITY DEFINER 트리거만 쓰기. PII는 서버·렌더 양쪽에서 마스킹한다.
**트레이드오프.** 쓰기마다 행이 늘어 실운영에선 파티션·아카이빙이 필요하다(데모 범위에선 무problem).

### 5.10 디자인 언어 — 음영 비의존·중복 인코딩
**고려.** 의료 UI는 정보 밀도가 높고, 모든 모니터(저가·미보정)에서 읽혀야 한다.
**결정.** 액션색(임상 틸)과 5개 상태 기능색을 분리하고, **색 + 도형(○●◐✓✕) 중복 인코딩**으로 색약·저대비 환경에 대비한다. **음영(그림자)에 의미를 싣지 않고** 색 채움·테두리·굵기로만 신호한다. 타이포는 Pretendard 자체호스팅으로 OS 무관 일관성을 보장한다.
**트레이드오프.** 다크모드는 보류(라이트 전용). 색 선택은 진료과 중립성을 고려했고, 조직 컬러와 충돌 시 토큰 교체로 대응한다.

### 5.11 그 외 굵은 결정
- **KST 시간 일관.** 모두 UTC로 저장하되 표시·영수증 일자·예약 슬롯은 `Asia/Seoul`로 변환 — 멀티 타임존을 원천 차단하고 로컬 규정에 정렬.
- **soft-delete 일관.** 직원·환자·마스터를 물리 삭제하지 않고 `is_active`로 비활성 — 과거 기록 참조 보존 + 누가 언제 비활성화했는지 추적.
- **사이드바 best-match 하이라이트.** 하위 경로 진입 시 부모 메뉴까지 중복 강조되지 않도록, 더 구체적인 메뉴가 매칭하면 부모는 양보한다(작지만 신규 직원 혼선 방지).

## 6. 범위와 한계 — 정직하게

**포함 (외래 전체):** 환자·진료(접수 흡수)·SOAP·진단, 처방·검사·처치 오더와 수행, 활력, 예약·근무 스케줄, 수납·정산·표준 문서, 마스터, 감사 — 외래 도메인 26개 테이블 + 6역할 + RBAC/RLS/pgcrypto.

- **결제(PG) 실연동** — 결제는 수단·금액 기록까지.

**솔직한 한계.**
- 소프트웨어는 만병통치가 아니다 — 워크플로우를 강제하면 현장 유연성이 일부 줄고, 신규 정착엔 교육이 함께 필요하다.
