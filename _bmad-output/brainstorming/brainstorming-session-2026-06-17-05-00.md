### 기법 ① Mind Mapping — 엔티티(테이블) 발산 결과

**중심 노드:** 환자 관리 시스템 DB

**발굴된 핵심 엔티티 (8개):**

1. **환자 (Patient)** — 시스템의 중심 주체
2. **사용자 / 계정 (User/Account)** — Supabase 로그인 주체. 직무별로 테이블을 쪼개지 않고 **계정 + 역할(Role)** 모델로 결정
3. **역할 (Role)** — `원무과(접수/수납)` · `의사(진료기록·처방전 작성)` · `방사선사(검사 촬영)` · `관리자(의사/진료실 추가·수정 등 권한 관리)`
4. **진료 / 내원 (Encounter)** — 환자가 한 번 내원한 "진료 건". 진료기록·처방전·검사를 묶는 **허브**
5. **진료기록 (MedicalRecord)** — 진료 한 건당 1개
6. **처방전 (Prescription)** — 진료 한 건당 1개(또는 0개)
7. **검사 (Examination)** — 한 진료에 여러 건 가능. 종류: X-ray / CT / MRI
8. **진료실 (Room)** — 진료 장소

**핵심 관계(카디널리티):**

- 환자 (1) ─< 진료 (N) — 방문할 때마다 진료 1건 누적
- 진료 (1) ─ 진료기록 (1)
- 진료 (1) ─ 처방전 (1 또는 0)
- 진료 (1) ─< 검사 (N)

**주요 설계 결정 (대화 중 도출):**

- **계정 vs 사람 분리:** 의사·방사선사를 별도 테이블로 두지 않고, `사용자(계정) + 역할`로 통합 → 과제 규모에 적합
- **판독 소견의 위치:** 진료기록이 아니라 **검사(Examination)** 에 부착. 소견은 "그 영상에 대한 해석"이라 영상과 1:1로 붙는 게 자연스럽고, 진료기록은 검사를 FK로 참조만 → 데이터 중복 제거. 촬영자(방사선사)와 판독자(영상의학과 의사)는 다를 수 있어 FK 분리(과제 범위면 단순화 가능)
- **검사 종류(X-ray/CT/MRI) 모델링:** 종류별 테이블 분리 대신 **검사 테이블 1개 + `검사종류` enum 컬럼**으로 확정 → 단순·확장 용이

**추가 엔티티 — 원무과(행정) 흐름 (전부 채택, "이왕 하는 거 다 넣자"):**

9. **예약 (Appointment)** — 진료 전 사전 예약. 환자·대상 의사·예약일시·상태
10. **접수/대기 (Reception)** — 당일 내원 환자의 대기 순번·접수 상태 (진료와 연결)
11. **수납/결제 (Payment)** — 진료비·검사비 청구 항목·금액·결제 상태 (진료와 연결)

**추가 엔티티 — 마스터/참조 데이터 + 자산 (전부 채택):**

12. **약품 마스터 (Drug)** — 약품코드·약품명·성분명·제형·규격·단가. 처방전이 FK로 참조
13. **처방상세 (PrescriptionDetail)** — 처방전 1장에 약 여러 개(N) → 약품·용량·복용법·일수를 담는 중간 테이블 (처방전 1 : N 처방상세)
14. **진료과 (Department)** — 과코드·과명(내과/정형외과/영상의학과 등). 의사·예약·진료실이 참조
15. **진단/질병코드 (Diagnosis)** — 표준 코드(KCD/ICD-10)로 관리. 질병코드·진단명·분류
16. **진료_진단 (연결테이블)** — 진료기록 ↔ 진단코드는 N:M(한 진료에 진단 여러 개) → 중간 연결테이블
17. **검사 장비 (Equipment)** — X-ray/CT/MRI 기계 본체. 장비명·종류·모델명·설치위치·상태(가동/점검중). 검사가 선택적 FK로 참조

**추가 엔티티 — 시스템/보안 (DB 기반 권한 관리 RBAC + 감사 로그, 전부 채택):**

18. **역할 (Role)** — enum이 아닌 **테이블로 승격**. 역할명(원무과/의사/방사선사/관리자)·설명
19. **권한 (Permission)** — 권한코드(예: `patient.read`, `doctor.create`, `room.update`)·설명
20. **역할_권한 (RolePermission)** — 역할 ↔ 권한 N:M 매핑. **관리자 페이지에서 체크박스로 토글** → 과제 "관리자 페이지" 요구와 직결
21. **감사 로그 (AuditLog)** — user_id·action(CRUD)·대상테이블·대상_id·시간·상세(변경내용/IP). 민감한 의료정보 접근 추적

**설계 메모:** 사용자(User)는 `role_id` FK로 역할 하나를 부여(필요 시 사용자_역할 N:M로 확장). 권한을 코드 하드코딩이 아닌 DB로 관리 → 관리자 페이지에서 역할별 권한을 동적으로 켜고 끌 수 있음.

| 축 | 컬럼 |

| 연락 | `휴대전화`, `주소`, `이메일`, `보호자명`, `보호자_연락처`, `보호자_관계` |
| 의료 | `혈액형`, `알레르기`, `기저질환`, `복용약`, `특이사항`(메모) |
| 행정 | `등록일시`, `최근방문일`, `보험유형`(건강보험/의료급여/자보), `보험번호`, `상태`(활성/비활성), `auth_uid`(→ auth.users, **nullable** — 포털 가입 환자만) |

**결정 사항:**
- `id`(내부 PK) + `환자번호`(사람이 보는 차트번호) **둘 다 유지**
- **주민등록번호:** `pgcrypto`의 `pgp_sym_encrypt`로 컬럼 암호화 저장, 암호화 키는 **Supabase Vault**(또는 앱 시크릿)에서 관리 — DB에 키를 두지 않음. 화면 표시는 마스킹(`900101-1******`)
- 로그인 비밀번호는 Supabase Auth가 bcrypt 해시 처리 → 직접 구현 불필요

#### ② 진료 / 내원 (Encounter)

| 축 | 컬럼 |
|---|---|
| 식별 | `id`(PK), `진료번호`(접수번호, 일자별) |

- **진료(Encounter)와 진료기록(MedicalRecord)을 분리 유지.** 결정 근거 = **확장성(향후 한 내원에 진료기록 1:N 가능 — 협진·추가 소견)**. 부수 이점: 상태 흐름(접수만 한 상태) 표현, 작성 주체(원무과 vs 의사) 분리, 검사·처방·수납의 FK 허브가 명확
- `진료상태`를 enum 컬럼으로 → **대기 현황판·통계**의 기반

#### ⑥ 사용자 (User / 직원 프로필) + 신원 모델

**신원(Identity) 모델 — 환자 포털 + 직원, 분리 프로필 패턴:**

```
auth.users (Supabase 인증: 이메일/비번/세션 — 환자·직원 공통)
   ├─(직원 로그인)→ 사용자(직원 프로필).id = auth uid → role_id 로 화면 분기(원무과/의사/방사선사/관리자)
   └─(환자 로그인)→ 환자.auth_uid = auth uid → 환자 포털(예약·내역 조회)
```

**사용자 (직원 프로필):**

| 축 | 컬럼 |
|---|---|
| 식별 | `id`(PK, = auth uid uuid), `직원번호`(사번), `이름` |
| 소속/권한 | `role_id`(FK → 역할), `진료과_id`(FK, 의사·방사선사) |
| 자격 | `면허번호`, `면허종류`(의사/방사선사) |
| 연락/상태 | `연락처`, `재직상태`(재직/휴직/퇴사), `입사일`, `생성일시` |

**결정 사항:**
- **환자 포털 + 환자 셀프 예약 + 역할별 화면 분기(5종: 원무과/의사/방사선사/관리자/환자)** 채택
- **분리 프로필 패턴:** 환자와 직원은 정보가 달라 한 테이블에 섞지 않음. 직원 = `사용자`(id=auth uid), 환자 = `환자.auth_uid`(nullable, 워크인 환자는 계정 없이 데이터로만 존재 가능)
- **로그인 분기:** uid가 `사용자`에 있으면 직원, `환자`에 있으면 환자 포털로 라우팅
- **RLS(Row Level Security) 적용:** Supabase 정책으로 **환자는 본인 데이터만** 조회 가능하게 강제(의료정보 보안 정석, 데모 어필)
- 환자 셀프 예약 → `예약` 테이블에 `예약경로`(직원접수/환자앱) 컬럼 예정

#### ⑦ RBAC — 역할 / 권한 / 역할_권한

**역할 (Role) — 직원 역할 5종 (간호사 추가):**

| 축 | 컬럼 |
|---|---|
| 식별 | `id`(PK), `역할코드`(reception/doctor/nurse/radiologist/admin), `역할명`(원무과/의사/간호사/방사선사/관리자) |
| 설명 | `설명`, `생성일시` |

**권한 (Permission):**

| 축 | 컬럼 |
|---|---|
| 식별 | `id`(PK), `권한코드`(`리소스.동작` 형식, 예: `patient.read`, `prescription.write`) |
| 분류 | `권한명`(한글), `대상리소스`, `동작`(read/create/update/delete) |

**역할_권한 (RolePermission) — N:M:**

| 축 | 컬럼 |
|---|---|
| 관계 | `id`(PK), `role_id`(FK), `permission_id`(FK), `(role_id, permission_id)` unique |

**결정 사항:**
- 직원 역할 **5종**(원무과/의사/간호사/방사선사/관리자), 화면 분기 **6종**(+환자 포털)
- 권한 코드 네이밍 `리소스.동작` 통일 → `hasPermission('patient.read')` 식 검사
- **관리자 페이지 핵심 화면**: 역할 × 권한 체크박스 매트릭스 → 토글 시 `역할_권한` 행 추가/삭제

#### ⑧ 간호 도메인 — 처치오더 / 처치기록 / 활력징후 (의사 오더 → 간호사 수행, 풀세트)

**처치오더 (TreatmentOrder) — 의사 지시:**

| 축 | 컬럼 |
|---|---|
| 식별/관계 | `id`(PK), `진료_id`(FK), `지시의사_id`(FK) |
| 오더 내용 | `처치종류`(주사/수액/채혈/드레싱/투약), `지시내용`(약물·용량·방법), `지시일시` |
| 상태 | `오더상태`(오더→수행완료→취소) |

**처치기록 (NursingRecord) — 간호사 수행:**

| 축 | 컬럼 |
|---|---|
| 식별/관계 | `id`(PK), `진료_id`(FK), `처치오더_id`(FK, nullable — 일상 간호는 오더 없이도), `수행간호사_id`(FK) |
| 수행 | `처치종류`, `수행내용`(메모), `수행일시` |

**활력징후 (VitalSign) — 전용 테이블:**

| 축 | 컬럼 |
|---|---|
| 관계 | `id`(PK), `진료_id`(FK), `측정자_id`(FK, 간호사) |
| 측정값 | `측정일시`, `수축기혈압`, `이완기혈압`, `체온`, `맥박`, `호흡수`, `산소포화도(SpO2)`, `체중`, `신장` |

**결정 사항:**
- 풀세트 채택: 의사가 `처치오더` → 간호사가 `처치기록`으로 수행. 오더 없는 일상 간호는 `처치오더_id` nullable로 허용
- 활력징후는 **전용 테이블**(한 진료에 여러 번 측정 + 시계열 그래프), 진료기록 O축에서 분리

#### ⑨ 예약 (Appointment)

| 축 | 컬럼 |
|---|---|
| 식별 | `id`(PK), `예약번호` |
| 관계(FK) | `환자_id`, `의사_id`, `진료과_id`, `진료_id`(nullable — 실제 내원 시 생성된 진료 연결) |
| 예약 정보 | `예약일시`(희망 진료 시각), `예약상태`(확정/취소/노쇼/완료), `예약경로`(직원접수/환자앱), `증상메모` |
| 처리 | `생성일시`, `생성자_id`(환자 본인 or 원무과) |

**결정 사항:**
- `예약경로`로 환자 셀프 예약 / 직원 접수 구분
- 예약 → 진료는 **1:0..1** (`진료_id` nullable, 노쇼·취소면 진료 미생성), `노쇼` 상태 포함(노쇼율 통계)
- **중복 방지:** `(의사_id, 예약일시)` unique 제약

#### ⑩ 의사 스케줄 — 의사근무스케줄 / 의사일정예외

**가용 슬롯 공식:** `예약 가능 = 의사 근무시간 − 휴무/예외 − 이미 잡힌 예약` (별도 저장 X, **동적 계산**)

**의사근무스케줄 (DoctorSchedule) — 주간 정기 근무(반복):**

| 축 | 컬럼 |
|---|---|
| 관계 | `id`(PK), `의사_id`(FK), `진료실_id`(FK, 선택) |
| 근무 | `요일`(월~일), `시작시간`, `종료시간`, `점심시작`, `점심종료`, `예약단위_분`(15/30분) |

**의사일정예외 (DoctorTimeOff) — 특정 날짜 예외:**

| 축 | 컬럼 |
|---|---|
| 관계 | `id`(PK), `의사_id`(FK) |
| 예외 | `시작일시`, `종료일시`, `사유`(휴가/수술/학회/병가/공휴일), `종일여부`, `비고` |

**결정 사항:**
- 가용 슬롯은 동적 계산(슬롯 캐시 테이블 불필요)
- **관리 주체:** 근무 스케줄 = **관리자**(병원 정책성) / 휴무·예외 = **의사 본인 + 관리자**(의사 포털에서 직접 등록 → 의사 화면 풍부화)

#### ⑪ 수납 (Payment) + 수납상세 (PaymentDetail) + 수가마스터 (FeeSchedule)

**목표:** 한국 표준 문서 **「진료비 계산서·영수증」**(항목분류별 합계)과 **「진료비 세부산정내역서」**(항목별 상세)를 데이터에서 자동 생성. → 둘 다 새 테이블이 아니라 `수납상세`를 표준에 맞게 설계하면 쿼리로 출력.

**수납 (헤더, 진료당 1건):**

| 축 | 컬럼 |
|---|---|
| 식별/관계 | `id`(PK), `수납번호`, `진료_id`(FK), `환자_id`(FK) |
| 금액 | `총진료비`, `공단부담금`, `본인부담금` (수납상세의 SUM) |
| 결제 | `결제상태`(미수납/부분수납/수납완료), `결제수단`(현금/카드/계좌이체), `수납일시`, `수납직원_id`(FK) |

**수납상세 (라인, 수납 1:N):**

| 축 | 컬럼 |
|---|---|
| 관계 | `id`(PK), `수납_id`(FK), `수가코드`(FK → 수가마스터) |
| 표준 분류 | `항목분류`(진찰료/검사료/영상진단료/주사료/처치료/투약조제료/치료재료/비급여…), `항목명` |
| 산정 | `단가`, `수량`, `횟수`, `금액` |
| 보험 분담 | `급여구분`(급여/비급여/전액본인), `본인부담금`, `공단부담금` (**줄마다** 분리) |
| 추적 | `발생출처`(진찰/검사/처치/처방), `출처_id`(nullable) |

**수가마스터 (FeeSchedule):**

| 축 | 컬럼 |
|---|---|
| 식별 | `id`(PK), `수가코드`(EDI코드, unique), `명칭` |
| 분류/단가 | `항목분류`, `기준단가`, `급여구분`(급여/비급여/전액본인), `사용여부` |

**결정 사항:**
- 마스터-디테일 + 수가마스터(약품 마스터와 동일 패턴). 단가는 수가마스터에서 참조
- `본인부담금`/`공단부담금`을 **줄(line)마다** 분리 → 세부산정내역서가 항목별 본인/공단을 표시. 헤더 합계는 SUM
- `발생출처`+`출처_id`로 "어느 검사·처치에서 나온 비용"인지 추적 가능
- **더미(시드) 데이터 관점**: 수가·약품·진단코드 마스터를 미리 채워두면 처방·수납 화면 시연이 실제처럼 동작

#### ⑫~⑰ 마스터 / 참조 / 로그 테이블

**⑫ 진료과 (Department):** `id`(PK), `과코드`, `과명`(내과/외과/정형외과/영상의학과…), `위치/내선`, `사용여부`

**⑬ 진료실 (Room):** `id`(PK), `호실번호`, `진료실명`, `진료과_id`(FK), `유형`(진료실/검사실/처치실), `위치`(층), `사용여부`

**⑭ 약품 마스터 (Drug):** `id`(PK), `약품코드`(보험/EDI), `약품명`, `성분명`, `제형`(정/캡슐/주사/시럽), `규격`, `단위`, `단가`, `보험구분`(급여/비급여), `사용여부`

**⑮ 진단코드 (Diagnosis):** `id`(PK), `질병코드`(KCD/ICD-10), `진단명`, `진단명_영문`, `분류`, `사용여부`

**⑯ 진료_진단 (EncounterDiagnosis):** `id`(PK), `진료기록_id`(FK), `진단코드_id`(FK), `주상병여부`(주진단/부진단), `진단일시` · `(진료기록_id, 진단코드_id)` unique

**⑰ 감사로그 (AuditLog):** `id`(PK), `사용자_id`(FK, nullable), `action`(create/read/update/delete/login), `대상테이블`, `대상_id`, `변경전`(jsonb), `변경후`(jsonb), `ip주소`, `발생일시`

**결정 사항:**
- `진료_진단`에 `주상병여부`(주진단 1 + 부진단 N) → 주상병 기준 통계·계산서 집계
- 감사로그 `변경전`/`변경후`를 PostgreSQL `jsonb`로 전후 스냅샷 저장(Supabase=Postgres)

### 🚫 범위 결정: 입원(Inpatient) 관리 — 향후 확장으로 보류

- **이번 범위는 외래(Outpatient)에 집중.** 입원은 별도 허브(`입원/Admission`)가 필요하고, 처방·검사·처치·수납이 외래/입원 두 갈래로 매달리며 구조가 한 단계 복잡해짐(거의 두 번째 시스템)
- 입원 시 추가 필요: 병동/병실/병상, 입퇴원·재원일수·퇴원요약, 회진, 입원료·식대 수가
- **판단:** 외래를 완성도 높게 끝내는 것이 평가·구현 측면에서 유리. 입원은 외래 완성 후 확장 후보(경량 → 풀세트 순)

### ERD 관계도 개요

```
auth.users ─1:1─ 사용자(직원) ─N:1─ 역할 ─N:M(역할_권한)─ 권한
auth.users ─1:1─ 환자(포털, nullable)

환자 ─1:N─ 예약 ─0..1─ 진료
환자 ─1:N─ 진료(Encounter) ★허브★
              ├─1:1─ 진료기록(SOAP) ─N:M(진료_진단)─ 진단코드
              ├─0..1─ 처방전 ─1:N─ 처방상세 ─N:1─ 약품마스터

              └─0..1─ 수납 ─1:N─ 수납상세 ─N:1─ 수가마스터

진료과 ─1:N─ {사용자(의사), 진료실, 예약, 진료}
의사(사용자) ─1:N─ {의사근무스케줄, 의사일정예외}
사용자 ─1:N─ 감사로그
```

### 핵심 설계 결정 (Key Decisions)

1. **계정 vs 사람 분리** — 직무별 테이블 대신 `사용자(계정) + 역할(Role)`
2. **분리 프로필 신원 모델** — 직원=`사용자`(id=auth uid), 환자=`환자.auth_uid`(nullable); 로그인 시 분기
3. **DB 기반 RBAC** — 역할 N:M 권한, 관리자 페이지에서 체크박스 토글
4. **RLS(Row Level Security)** — 환자는 본인 데이터만 조회
5. **pgcrypto 컬럼 암호화** — 주민번호, 키는 Supabase Vault
6. **진료/진료기록 분리** — 근거: 향후 1:N 확장성 (+상태 흐름, 작성 주체 분리)
7. **SOAP 진료기록** + **활력징후 전용 테이블**(간호 도메인, 시계열 그래프)
8. **검사·처치의 오더→수행 흐름** — 지시(의사) / 수행(방사선사·간호사) FK 분리
9. **마스터-디테일 + 마스터 테이블**(약품/수가) — 처방·수납을 표준 단가 참조로
10. **표준 진료비 문서 대응** — 수납상세를 「계산서·영수증」「세부산정내역서」 양식에 맞춰 설계
11. **입원 관리 보류** — 외래 완성도 우선, 향후 확장

### 세션 회고 (Reflections)

- **기법 흐름이 효과적이었음**: 마인드 매핑으로 엔티티를 발산 → 형태 분석으로 컬럼을 수렴. "요소 정리"라는 목표에 정확히 부합
- **"실제처럼" 기조**가 일관된 의사결정 기준이 되어, SOAP·RBAC·pgcrypto·표준 진료비 문서 등 현실적 디테일을 반영
- **좋은 갈림길 질문들**(계정 vs 사람, 진료 vs 진료기록, 입원 범위)이 구조 품질을 끌어올림
- **사용자의 핵심 인사이트**: "어떤 진료에 자료를 이용했나" 한마디에서 진료(Encounter) 허브를 도출, 간호사·입원 범위를 능동적으로 제기
