# Story 4.3: 대기 현황판 — 실시간 · 다음 호출

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a **원무·의사**,
I want **진료과·진료실별 실시간 대기 현황과 "다음 호출 환자"를 한 화면에서 보고, 다음 환자를 호출하기를**,
so that **신규 직원도 순번을 외우지 않고 정확히 다음 환자를 부르며, 중복 호출·누락을 방지한다.**

## Acceptance Criteria

1. **AC1 — 대기 현황판 표시 (FR-022, UX-DR6·UX-DR7·UX-DR8):** 접수된 내원이 있을 때 대기 현황판을 열면, **상태별 그룹 섹션**(섹션 헤더=점+컬러 상태명+카운트 pill)이 **활성도 순**(`in_progress → registered → scheduled → completed → cancelled → no_show`)으로 표시되고, **종결 섹션(`completed`/`cancelled`/`no_show`)은 접힘+muted**(클릭 확장)이며, 각 행은 **status-badge A3**(8px 점 + 글리프 ○●◐✓✕ + 상태색 라벨, 색·글리프·굵기 다중 인코딩, `registered` 라벨=`status-received-ink`·`cancelled` 라벨=`line-through`)로 진료상태를 보이고, 상단 **"다음 호출" 히어로**(다음 호출 대상 환자 = 가장 오래 대기한 미호출 `registered` 내원)와 **행별 상태-적합 다음-액션 ghost 버튼**(이 스토리 배선: `registered`=`호출`(.key) / `scheduled`=`접수`)을 제시한다.

2. **AC2 — 실시간 갱신 ≤5초 + 신선도 가드 (NFR-002, UX-DR18·UX-DR21⑪):** 다른 단말이 상태를 변경(접수·전이·호출)하면 `encounters` 테이블의 **`postgres_changes` 구독**(진료과 필터)이 수신되어 보드가 **≤5초 내** 갱신되고, 채널이 끊기거나 신선도 임계(≤5초)를 초과해 **stale** 가 되면 **stale 배너**("연결 지연 · 실시간 갱신 멈춤" + 마지막 갱신 시각 + "다시 연결") 가 뜨고 **호출 등 중요 동작 버튼이 비활성(가드 — "권장"이 아닌 강제)** 된다. 실시간 수신 시 해당 행을 즉시 갱신/잠금한다(UX-DR21⑩).

3. **AC3 — 다음 호출 + 호출 상태 기록 + 중복 호출 방지 (FR-023, UX-DR8):** 호출 버튼을 누르면 `POST /api/v1/encounters/{id}/call` 가 `record_encounter_call` RPC 를 호출해 **`called_at`·`call_count`·`last_called_by` 를 기록**(감사 트리거가 자동 append-only 기록)하고, **mutation 진행 중 버튼이 disable**(이중 제출 차단 = 중복 호출 방지 1차선)되며, 호출된 행은 보드에 **"호출됨 · {시각} (N회)"** 로 표시되어 다른 단말에도 실시간 전파됨으로써 중복 호출·누락을 방지한다. 잘못된 상태(미접수/종결) 호출은 **409** `invalid_transition`, 권한 미보유는 **403**, 미존재 내원은 **404** 다.

### 🎯 핵심 설계 결정 (이 스토리가 새로 확정 — 산출물에 선결정 없음)

1. **"호출 상태(call state)" = 상태머신과 분리된 비-전이 마커.** 상태 enum 6값(`scheduled|registered|in_progress|completed|cancelled|no_show`, 0010:38)에 **`called` 상태는 없다** — 환자를 호출해도 내원은 `registered` 에 머문다(진찰 시작=`start_consult`가 `in_progress`로 전이, 4.4). FR-023 "호출 상태를 기록"의 목적은 **중복 호출·누락 방지**이므로, 전이가 아닌 **`called_at`/`call_count`/`last_called_by` 컬럼 + `record_encounter_call` RPC**로 모델링한다. 근거: ① 호출은 상태 전이가 아님(트리거의 same-status 통과 경로 0010:79 활용) ② 감사 트리거가 UPDATE 를 자동 기록(FR-023 "기록") ③ 중복 방지 1차선=UX disable + 실시간 cross-terminal 가시성(호출됨 표기), 하드 차단은 재호출 자체가 정상 동작이라 불요. **대안(기각)**: 별도 `encounter_calls` 이벤트 테이블(정규화·이력 풍부하나 realtime 2테이블 구독·MVP 과설계) / broadcast-only(비영속 → 새로고침 시 호출상태 소실, "기록" 요건 미충족).

2. **실시간 = Supabase `postgres_changes`(트리거) → FastAPI 보드 refetch(데이터).** architecture.md:204 패턴("postgres_changes 구독 → 캐시 무효화/패치")의 실현. **분담**: 실시간 구독(Supabase)은 "진료과 X에 변화 발생"만 알림(payload=encounters 행, **비-PII**: patient_id FK·encounter_no·status·called_at — 환자명 없음, UX-DR22 충족) → 보드는 `GET /encounters`(FastAPI, denormalized 조인)를 **refetch**. ⚠️ 코드베이스 **최초 realtime 도입** — `web/src` 에 `.channel`/`postgres_changes` 0건, **DB publication 0건**(전 마이그레이션 미설정). 둘 다 이 스토리가 신설.

3. **목록 = FastAPI `GET /encounters`(직접 Supabase SELECT 아님).** 보드는 환자명·진료과명·진료실·담당의 **다중 조인 denormalized** 집합("복잡 집계"=FastAPI, architecture:193) + 일관 `{data,meta}` + 권한 게이트. 직접 Supabase 조인은 patients PII 컬럼 노출·RLS 컬럼 스코프 부재(UX-DR22) 리스크라 비채택. **realtime payload(비-PII) ≠ 보드 표시 데이터(조인된 PII 포함)** 를 분리해 실시간 페이로드에 환자명 유입을 차단.

4. **다음-액션 버튼 스코프(엄수, 은폐 금지):** 이 스토리 배선 = **`호출`(신규 RPC)** + **`접수`(scheduled→registered, 4.2 `POST /{id}/register` 재사용·MVP엔 appointments 부재로 희귀)**. `진료 시작`(start_consult)=**Story 4.4**(의사 진료 대기열·진찰 시작·진료 허브 진입 — 4.4의 심장, start_consult 엔드포인트도 4.4 신설)·`완료 처리`(complete_encounter)=**Epic 7**(엔드포인트 미존재). WaitingBoard 는 **상태별 액션 슬롯 구조**로 작성해 4.4/Epic7 이 동일 공용 컴포넌트에 버튼을 증분 배선. AC1 "버튼 제시"는 이 스토리 스코프 액션에 대해 충족(에픽의 4.3/4.4 경계 자체가 이 분할을 전제 — 4.2 가 보드 전체를 4.3로 이월한 선례와 동일).
