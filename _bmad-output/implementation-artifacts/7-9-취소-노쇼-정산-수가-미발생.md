# Story 7.9: 취소 · 노쇼 정산 (수가 미발생)

Status: done

> ⚠️ **설계 결정 4건(사용자 확정 2026-06-24, AskUserQuestion)** — Dev Notes §설계 결정 참조.
> ① **정산 진입점 = 수납 화면 통합 액션** — billing-detail(워크리스트 registered draft 건)에 "내원 취소·환급" 버튼. 신원 confirm → `settle_cancelled_visit` RPC가 한 트랜잭션에서 `cancel_encounter`(registered→cancelled) + payment void(status='cancelled') + 선납 환급. 결제 도메인을 수납 영역에 일원화(7.8 선례). cancel_encounter RPC를 7.9가 처음 배선.
> ② **선납 환급 기록 = 신규 컬럼 `refunded_amount_krw`** — `paid_amount_krw`(총 수령)는 보존, `refunded_amount_krw`(반환액) 별도 → 순납부 = paid − refunded. "10,000 받고 10,000 환급" 이력 영속(리포팅·감사 근거). 신규 컬럼 1개 + cancelled 일관성 CHECK(0048 이월).
> ③ **노쇼 수수료 = 정책 OFF 기본·메커니즘 예비만** — 취소·노쇼 = 수가 미발생·환급만 구현. 노쇼료 미부과(한국 외래 관행·단순화 선). AC2 "별도 항목 부과 가능"은 `payment_details.fee_item_id` nullable 수기 라인 스키마(0045 예비)로 충족 — 실제 부과 UI/RPC 미구현(이월).
> ④ **환급 확인 = 신원 confirm + 원결제수단 자동** — 환급도 고위험 비가역 → 신원 재진술 confirm(이름·차트번호·환급액·원결제수단, 7.4/7.8 선례). 환급수단 = 원 선결제수단(`payment_method`) 자동·별도 입력 없음. 환급 확인증/영수증 미출력(법정 서식에 환급 개념 없음·7.8 defer 청산).

## Story

As a 시스템,
I want 취소·노쇼로 종결된 내원을 수가 미발생으로 처리하고 선납액을 환급하기를,
so that 발생하지 않은 진료가 청구되지 않고 미리 받은 금액이 정확히 반환된다.

## Acceptance Criteria

**AC1 (FR-118 수가 미발생 — 구조적 + draft void):**
**Given** registered(접수 후·진찰 전) 또는 scheduled 내원이 취소될 때
**When** 정산을 수행하면
**Then** ① 진찰료(`fee_on_encounter_start`는 `registered→in_progress` 트리거)·검사처치료(perform 시)가 **구조적으로 미적재**(취소는 in_progress 진입 전)되어 `fee_items` 0이고, ② 존재하는 draft 수납 건은 `status='cancelled'`로 void되어 청구 0이 된다. **finalized 수납(완료 내원)은 이 경로로 취소 불가**(cancel_encounter가 비-registered/scheduled를 PT409 차단).

**AC2 (settle_cancelled_visit RPC — 취소·void·환급 원자):**
**Given** registered 내원의 draft 수납 건에 대해
**When** 원무가 "내원 취소·환급"을 확정하면
**Then** `settle_cancelled_visit(encounter_id, reason)` RPC가 한 트랜잭션에서 ① `perform cancel_encounter(encounter_id, reason)`(registered→cancelled·상태머신 재사용·재구현 0) ② draft payment가 있으면 `status='cancelled'`·`cancelled_at=now()`·`cancel_reason=reason` ③ 선납액 환급(`refunded_amount_krw = paid_amount_krw`) 한다. 중간 실패 시 전체 롤백(NFR-041). `payments`·`encounters` 감사 트리거가 actor와 함께 자동 기록한다.

**AC3 (선납 환급 — refunded_amount_krw 단일 전액):**
**Given** 선결제된(`paid_amount_krw > 0`·`billing_type='prepaid'`) draft 수납 건이 취소될 때
**When** void하면
**Then** `refunded_amount_krw = paid_amount_krw`(선납 전액 환급)로 기록되고 `paid_amount_krw`(총 수령)는 보존된다 — 순납부 = `paid − refunded = 0`. 후수납(선납 0) 취소는 `refunded = 0`(환급 없음). 환급수단은 원 선결제수단(`payment_method`)을 따른다(별도 입력 없음).

**AC4 (cancelled 일관성 CHECK + 환급 불변식):**
**Given** payments 행 일관성에 대해
**When** 제약을 적용하면
**Then** ① `payments_cancelled_consistency` CHECK(0048 이월): `status<>'cancelled' OR cancelled_at is not null`(취소 행은 취소시각 필수) ② `refunded_amount_krw >= 0` ③ `refunded_amount_krw <= paid_amount_krw`(받은 것보다 더 환급 불가)가 강제된다. 기존 draft/finalized 행은 전부 통과(무회귀).

**AC5 (결제상태 A3 — 취소 배지·UX-DR14):**
**Given** cancelled 수납 건의 결제 상태에 대해
**When** 결제상태 배지를 렌더하면
**Then** `cancelled`=취소(로즈 `status-cancelled`·✕·취소선)로 표시된다(기존 `PaymentStatusBadge`의 cancelled 분기 재사용 — **무변경**). 색비의존(글리프 ✕ + "취소" 라벨 병기).

**AC6 (취소·환급 패널 — billing-detail):**
**Given** draft 수납 건 상세 화면에서
**When** 화면을 열면
**Then** 결제 섹션에 "결제·완료"와 나란히 **"내원 취소·환급" 버튼**이 노출되고, 누르면 **신원 재진술 confirm**(이름·차트번호 + 선납 시 "환급액 X원 (원결제수단)" 표시)을 거친다(UX-DR21). 확정 후 화면이 cancelled 상태로 갱신되어 **취소 패널**(취소 사유·`refunded_amount_krw>0`이면 "환급 X원 (원결제수단)" + `cancelled_at`)이 표시된다. 금액=KRW 정수·`tabular-nums`·"원".

**AC7 (진입점 — 워크리스트 연속성):**
**Given** 수납 워크리스트(registered+in_progress·7.8)에서
**When** registered 건을 선택해 취소·환급을 수행하면
**Then** 동일 `billing-detail`에서 취소가 처리되고, 취소 후 내원이 `cancelled`로 전이되어 워크리스트(registered/in_progress만)에서 자연히 사라진다(별도 제거 로직 불요).

**AC8 (노쇼 수수료 — 메커니즘 예비·미부과):**
**Given** 노쇼 수수료 정책에 대해
**When** AC2(스토리 원문 "노쇼 수수료 정책이 적용될 때 부과하면 별도 항목으로 부과 가능")를 평가하면
**Then** `payment_details.fee_item_id` nullable 수기 라인(0045 예비·"노쇼료 7.9" 주석)으로 **부과 가능성**이 스키마에 충족되어 있고, 실제 부과 경로(UI/RPC)는 **미구현(정책 OFF 기본)**임을 명시한다(한국 외래 노쇼료 미부과 관행·단순화 선). 신규 노쇼료 부과 RPC·UI 추가 **금지**(스코프 누수 방지).

**AC9 (권한·경계):**
**Given** 취소·환급 쓰기에 대해
**When** 권한을 평가하면
**Then** 라우터 게이트 = `payment.manage`(기존·7.2 reception grant), RPC 내부 `cancel_encounter`는 `encounter.cancel`(0010 기존 권한)를 평가하므로 **reception → `encounter.cancel` 시드 grant 신규**(7.4 `encounter.complete` grant 선례·신규 *권한* 0). nurse·doctor는 403(baseline). **신규 마이그 1건(0052)·신규 컬럼 1개(`refunded_amount_krw`)·신규 권한 0·신규 테이블 0.**

**AC10 (테스트):**
**Given** 변경 표면에 대해
**When** 테스트를 실행하면
**Then** API(settle void·선납 환급 refunded=paid·후수납 취소 refunded=0·cancel_encounter 전이·draft 없는 내원 취소·finalized/비-registered 취소 거부·cancelled CHECK·refunded≤paid CHECK·권한 reception 통과/nurse·doctor 403)와 web(취소·환급 버튼·신원 confirm 환급액 표시·취소 패널)이 통과하고, **7.4 후수납 finalize·7.8 선결제·차액 정산 회귀가 깨지지 않는다.**

### 경계 (Story 7.9 — 무엇을 하고 무엇을 안 하나)

### 핵심 통찰 — 왜 "수가 미발생"이 대부분 구조적인가 (취소·노쇼 상태머신 분석)

```
내원 상태머신(0010·glossary):
(INSERT) → scheduled | registered
scheduled  → registered | cancelled | no_show
registered → in_progress | cancelled        ← 7.9 취소는 여기(진찰 전)
in_progress→ completed (only)                ← 부분수행도 completed 후 정산(7.10)
종결(completed·cancelled·no_show) = 이탈 전이 없음
```

- **진찰료 fee_item**은 `registered→in_progress`(start_consult) 트리거(`fee_on_encounter_start`)로 적재. **검사·처치료**는 perform(in_progress 중) 시 적재. **취소·노쇼는 in_progress 진입 *전*** → `fee_items`가 **구조적으로 0** = "수가 미발생"은 자동 보장(별도 삭제·미발생 처리 로직 불요).
- 따라서 7.9의 실제 능동 작업 = ① draft 수납 건 void(특히 선납으로 생성된 prepaid 헤더) ② **선납 환급**(핵심) ③ cancelled 일관성 강제. fee_item 삭제/0 처리 코드는 **작성 금지**(존재하지 않는 것을 삭제하려 하지 말 것).
- **노쇼는 거의 예약(appointment) 도메인 개념**(0036: `encounters.no_show`는 카운트 안 함·노쇼 슬롯 낭비는 *예약* 미방문). 환자 미방문 → 내원 미생성 → 청구 대상 0(구조적). `mark_no_show`(scheduled→no_show)는 방어적 능력이고, scheduled 내원은 Option A(0033)에서 사실상 미사용(예약은 도착 시점에만 registered 내원 생성). ⇒ **실무 핵심 케이스 = registered+선납 내원 취소 → 환급**. settle RPC는 cancel_encounter(registered/scheduled→cancelled) 경로를 배선한다.

### 핵심 메커니즘·불변식 (project-context 준수)

- **불변식·정산·상태전이 = DB 소유.** settle의 취소 전이는 `cancel_encounter` perform(재사용)·void/환급은 단일 UPDATE — 상태머신/정산 로직을 Python/TS에 재구현 금지. FastAPI=오케스트레이션(존재검사 + RPC 호출 + 매핑)만.
- **쓰기=FastAPI(service_role)·SECURITY DEFINER RPC** / 조회=라우터 게이트. RLS(0045)는 web 직접조회 방어심층(유지).
- **권한 2계층:** ① 라우터 `payment.manage`(billing 쓰기 일관) ② RPC 내부 `cancel_encounter`의 `has_permission('encounter.cancel')`. 둘 다 통과 필요 → reception에 encounter.cancel grant. `settle_cancelled_visit`은 EXECUTE 회수 + service_role 호출(authenticated 직접호출 차단·prepay/finalize 동형)·db 계층 `_require_payment_manage`가 manage TOCTOU 재평가.
- **⚠️ 비자명 정합성 — actor 해석:** `has_permission`(0003)은 **`auth.uid()`로 actor 해석**(GUC 아님). `_run_authed`가 `authenticated_conn(sub)`(JWT claims=sub) 위에서 실행 → `auth.uid()=sub` 해석됨. 기존 `_require_payment_manage`(`has_permission('payment.manage')`)가 이 연결에서 정상 동작하는 것이 증거. ⇒ `cancel_encounter`(SECURITY DEFINER)의 내부 `has_permission('encounter.cancel')`도 **동일 연결·동일 메커니즘**으로 reception actor를 평가 → encounter.cancel grant만 있으면 통과(신규 코드 경로지만 검증됨). nurse·doctor(미보유) → 42501→403. SECURITY DEFINER 함수는 owner 권한으로 RLS 우회 쓰기·has_permission는 호출자 컨텍스트 평가(prepay/finalize 동형).
- **금액 = KRW 정수**. `refunded_amount_krw int ≥ 0`·`≤ paid_amount_krw`(CHECK). 환급 전액(`refunded = paid`) — 부분 환급 미지원(취소=전부 미발생·전액 반환).
- **상태 전이는 액션 엔드포인트**(`POST .../payment/cancel`·status PATCH 아님). settle는 내원(registered→cancelled) + payment(draft→cancelled) 동시 전이(원자).
- **감사 append-only**: `trg_payments_audit`(0045)·encounters 감사 트리거가 settle UPDATE를 actor와 자동 기록. raw PII 미적재(status·금액·사유=저민감 운영 텍스트). `cancel_reason`은 임상/PII 자유텍스트 금지(저민감 운영 사유만·`encounters.cancel_reason` 정합).
- **에러 봉투**: `code`=영문 기계용·`message`=한국어. PT409→`invalid_transition`(409)·42501→`forbidden`(403)·PT404→`not_found`(404)는 `_map_pg_sqlstate`(0010) 기존 매핑.
