## Story

As a **원무 직원**,
I want **전화·방문 환자를 대신해 예약을 생성·변경·취소하기를**,
so that **앱 미사용 환자도 예약된다.**

## Acceptance Criteria

**[AC1] 예약 변경·취소·노쇼 — 상태머신·감사 일관 (FR-011)**
- **Given** 원무가 캘린더의 확정(booked) 예약에 대해
- **When** 변경(reschedule)·취소(cancel)·노쇼(no_show)를 수행하면
- **Then** 예약 상태가 갱신되고(booked→cancelled/no_show, 변경=시각 재배치), **모든 전이는 DB 상태머신(enforce_appointment_transition)·감사로 일관 처리**된다. 잘못된 전이(종결 재전이 등)는 차단(409)된다.

**[AC2] 가용 슬롯 반영**
- **Given** 예약을 취소·노쇼·변경하면
- **When** 슬롯 계산/캘린더를 다시 조회하면
- **Then** 취소·노쇼된 슬롯은 다시 **가용(available)**으로, 변경된 예약은 새 슬롯에 **확정**으로 반영된다(슬롯 계산·EXCLUDE가 `status='booked'` 기준이므로 자동). 생성·변경은 **가용 슬롯만** 허용된다(서버 슬롯-윈도우 검증, 6.3 이월 청산).

**[AC3] 대기 흐름 반영 — 예약 환자 도착 접수**
- **Given** 확정 예약 환자가 내원해 원무가 "도착 접수(check-in)"하면
- **When** 접수를 수행하면
- **Then** `visit_type='reserved'`·`reservation_id`로 연결된 **registered 내원이 생성**되어 대기 현황판(4.3)에 진입하고, 예약은 `completed`로 종결된다.

**[AC4] 권한 게이트**
- **Given** `appointment.update` 미보유 계정(예: nurse)으로
- **When** 변경·취소·노쇼·접수를 호출하면
- **Then** 403. 보유 계정(reception·admin)은 정상.

> **범위 노트(에픽 6.4·Option A 확정[사용자 승인]):** **appointment.status 가 예약 생명주기의 단일 진실**(booked→cancelled/no_show/completed). 내원은 **도착 시점에만** 생성(reserved·registered·reservation_id). encounters 의 `scheduled` 초기상태·`register_encounter`(scheduled→registered)는 **reserved 경로 미사용**(예약은 appointments 가 소유; 향후 정리 대상·이월 노트). 6.7 노쇼 카운트 = `appointment.no_show` 집계. 6.2/6.3 이월(전이 트리거·`reservation_id`→내원 배선·슬롯-윈도우 검증)을 본 스토리가 청산.

  ```sql
  -- ⚠️ INSERT 가드 없음(BEFORE UPDATE 만): 6.3 test_double_booking_adjacent_and_cancelled_allowed 가
  --    cancelled 를 직접 INSERT(EXCLUDE 부분 술어 검증) → INSERT 초기상태 가드 추가 시 회귀. 초기상태는
  --    0031 status CHECK + default 'booked' 가 담당. UPDATE 전이만 매트릭스 강제.
  create or replace function public.enforce_appointment_transition() returns trigger
  language plpgsql as $$
  begin
    if new.status = old.status then return new; end if;  -- 비-상태 컬럼 갱신(reschedule 등) 통과
    if not (old.status = 'booked' and new.status in ('cancelled', 'no_show', 'completed')) then
      raise exception 'invalid appointment transition: % -> %', old.status, new.status
        using errcode = 'PT409';
    end if;
    return new;
  end; $$;
  drop trigger if exists trg_appointments_transition on public.appointments;
  create trigger trg_appointments_transition before update on public.appointments
    for each row execute function public.enforce_appointment_transition();
  ```

## Dev Notes

### 확정 설계 요약 (Option A — 사용자 승인 2026-06-22)
**예약↔내원 모델 = appointment-중심.** `appointment.status` 가 예약 생명주기 단일 진실(booked→cancelled/no_show/completed). 내원은 **도착 시점에만** 생성(visit_type='reserved'·status='registered'·reservation_id→예약)되어 대기 현황판 진입. encounters 의 `scheduled` 상태·`register_encounter`(scheduled→registered)는 **reserved 경로 미사용**(예약 생성이 scheduled 내원을 만들지 않음). 근거: 6.3 예약 생성이 appointment만 만들었고(내원 없음), 이중 상태머신 동기화 회피·예약(부킹)과 내원(방문)의 깔끔한 분리. **트레이드오프**: encounters 의 reserved 전용 설계(scheduled·register_encounter)가 사장 → 이월 노트(향후 정리 또는 환자앱/대시보드에서 재검토).
