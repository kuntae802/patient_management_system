## Story

As a 원무 직원,
I want 진찰·오더에서 자동발생한 수가(`fee_items`)를 집계한 **수납 건(draft `payments` 헤더 + `payment_details` 라인)** 을 수납 화면에서 보기를,
so that 수기 계산 없이 정산을 시작하고, 7.3(본인부담 산정)·7.4(finalize·내원 완료)가 이 영속 집계 위에서 동작한다.

> **Epic 7 첫 소비 레이어 스토리.** 7.1(`0045_payments.sql`)은 순수 DB 토대(스키마+초진재진+이월흡수, 운영코드 0)였다. 7.2는 **DB→FastAPI→Web 전환점**: 집계 명령(DB 함수)·읽기/빌드 엔드포인트(FastAPI)·수납 워크리스트+상세 화면(Next.js)을 처음 구현한다.

**AC1 — 집계 DB 함수 `build_payment(encounter_id)`(멱등·원자적 — 집계 로직 = DB 소유):**
`0046`에서 `build_payment(p_encounter_id uuid) returns uuid`(SECURITY DEFINER·`set search_path=public`)가 생성된다. 한 내원의 자동발생 수가를 draft 수납 건으로 집계한다:
- **헤더 upsert(1:1)**: `insert into payments (encounter_id) values (p_encounter_id) on conflict (encounter_id) do nothing` → 그 후 `select id, status` 룩업(내원당 단일 헤더 — 0045 `encounter_id` UNIQUE).
- **상태 가드**: 헤더 `status`가 `'draft'`가 아니면(`finalized`/`cancelled`) **집계 변경 없이** payment_id만 반환(finalize 후 라인 불변 — 7.4 결정 보존).
- **라인 적재(멱등)**: `insert into payment_details (payment_id, fee_item_id, fee_schedule_id, code, name, category, quantity, unit_amount_krw, amount_krw, coverage_type) select <payment_id>, fi.id, fi.fee_schedule_id, fs.code, fs.name, fi.category, fi.quantity, fi.unit_amount_krw, fi.amount_krw, fi.coverage_type from public.fee_items fi join public.fee_schedules fs on fs.id = fi.fee_schedule_id where fi.encounter_id = p_encounter_id on conflict (payment_id, fee_item_id) do nothing` — `unique(payment_id, fee_item_id)`로 **재집계 시 신규 fee_item만 추가**(기존 라인 불변·중복 0). ⚠️ `code`·`name`은 `fee_items`에 **없음** → `fee_schedules` 조인으로 스냅샷(나머지 금액·category·coverage는 `fee_items` 스냅샷 직접 복사 — 청구 시점 고정).
- **헤더 금액 롤업**(deferred-work L366 흡수 — 7.2 책임분): `update payments set total_amount_krw = Σ(amount_krw), covered_amount_krw = Σ(amount where coverage='covered'), non_covered_amount_krw = Σ(amount where coverage='non_covered'), updated_at = now() where id = <payment_id>`. **copay/insurer 롤업은 7.3**(본 함수는 0 유지). 결과: `total = covered + non_covered` 항등 성립.
- 반환: payment_id.
- 함수 EXECUTE posture(0045 `insert_fee_item` 동형): `revoke all on function public.build_payment(uuid) from public, anon, authenticated`(service_role/FastAPI만 호출 — authenticated 직접 호출 차단).

**AC3 — 집계 빌드 엔드포인트 `POST /v1/encounters/{encounter_id}/payment`(진입 시 자동 집계 — 멱등):**

**AC7 — 수납 집계 상세 화면 `/reception/billing/[encounterId]`(헤더 + 라인 + "자동" 마커):**
`page.tsx`(RSC·`requirePermission("payment.read", STAFF_HOME)`)가 `BillingDetail` 클라이언트 컴포넌트를 렌더한다. **진입 시 `buildPayment(encounterId)`(POST·payment.manage) 호출 → draft 집계 영속 + 반환 헤더/라인 렌더**(진입 시 자동 집계 모델):
- **헤더 요약 카드**: 총 진료비(`total_amount_krw`)·급여(`covered_amount_krw`)·비급여(`non_covered_amount_krw`) — KRW 정수·`tabular-nums`·"원" 접미. teal **"자동 산정" 마커**(order-panel 마크업 미러: `border-primary/30 bg-primary/10 ... text-primary`). 본인부담/공단부담은 "산정 전(7.3)" placeholder(또는 미표시).
- **상세 라인 목록**: 각 `payment_detail`을 행으로 — `code`(tabular-nums)·`name`·**`PayChip`(급여=그린/비급여=앰버 잉크)**·수량(quantity>1 시 "×N")·단가·금액(`formatKrw`·tabular-nums·"원")·**"자동" per-line 마커**(전 라인 fee_item_id 보유 = 자동). 분류(category)별 그룹 헤더(진찰료/검사료/영상료/처치료) 권장.
- **빈 상태**: 라인 0(아직 진찰·오더 미발생) → "집계된 수가 항목 없음" + 안내. 빌드/로드 실패 = ApiError 메시지(403 = "수납 권한 없음").
- finalize·결제수단·내원완료 **버튼 없음**(7.4 소관 — 본 화면은 집계+표시까지).

**AC8 — 응답·검증 계약(snake_case·KRW 정수·표준 봉투):**
- JSON 전 경로 `snake_case`(camelCase 변환 금지 — project-context). 금액 = **KRW 정수**(소수 없음). 날짜 = ISO 8601(timestamptz UTC → KST는 클라 `Intl`/기존 헬퍼).
- 단건(`PaymentResponse`) = 리소스 직접. 목록(워크리스트) = `{data, meta}`. 에러 = `{error:{code,message,detail}}` + HTTP(404/403/422). `code`=기계용 영문·`message`=한국어.
- 타입: Pydantic(`api/app/schemas/billing.py`) + 웹 수기 타입(`web/src/lib/billing/payments.ts` — `database.types.ts` 미생성 프로젝트, 기능별 수기 타입이 관습).

**AC9 — 감사·RLS·트랜잭션 원자성(NFR-041):**

project-context §"불변식은 DB가 소유 — 수가/정산 로직을 Python/TS에서 재구현하지 말 것. FastAPI는 오케스트레이션·호출만." → 집계(fee_items→payment_details·롤업)는 **`build_payment` DB 함수**가 소유. FastAPI는 RPC 호출 + 응답 매핑만. 5.10(`insert_fee_item`/`fee_on_encounter_start`)·7.1 선례. DB 함수 = psql 직접 테스트 가능(5.10/7.1 패턴) + API 테스트 이중 검증.
