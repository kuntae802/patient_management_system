# Story 2.1: 진료과 · 진료실 마스터 관리

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 관리자,
I want 진료과(department)와 진료실(room)을 생성·수정·비활성(soft delete)하기를,
so that 예약·접수·대기열·근무표 등 후속 화면이 정확한 조직 단위를 참조한다.

## Acceptance Criteria

**AC1 (FR-200) — 진료과·진료실 생성·수정**
**Given** 관리자 권한(`master.manage`)으로 마스터 화면(`/admin/masters`)에서
**When** 마이그레이션 `0006_masters.sql`의 `departments`·`rooms`를 기반으로 진료과·진료실을 생성·수정하면
**Then** 조직 단위가 등록되고 후속 화면(예약·대기열·근무표)에서 선택 가능해진다. 권한 없는 사용자는 사이드바 항목 미노출 + 엔드포인트 403.

**AC2 (FR-203) — soft delete(비활성) + 참조 보존**
**Given** 활성 진료과·진료실에 대해
**When** 비활성(`is_active=false`)으로 전환하면
**Then** 물리 삭제 없이 신규 선택 대상에서만 제외될 수 있도록 `is_active` 플래그가 내려가고, 행과 명칭은 그대로 보존되어 과거 기록(직원 소속·향후 예약/대기열)이 참조를 유지한다. 재활성(`is_active=true`)도 가능하다.

**AC3 — 변경 감사 + RBAC 게이트**
**Given** 진료과·진료실의 생성·수정·비활성 변경이 발생하면
**When** 그 변경이 커밋되면
**Then** `0004` 제네릭 감사 트리거가 행위자(=호출 관리자)와 전/후 스냅샷을 `audit_logs`에 자동 기록하고, 감사 뷰어(1.10)에서 조회된다. 모든 쓰기는 `require_permission('master.manage')`(403)으로 게이트되며, 평가↔쓰기는 동일 트랜잭션에서 재평가(TOCTOU 차단)된다.

**AC4 — 직원 소속 FK 무결성 확립**
**Given** `0002`가 `users.department_id`를 **FK 미부착**으로 둔 상태에서(0002:37-38 — "departments 는 0005_masters 가 생성하므로 FK 도 0005 가 추가")
**When** `0006_masters.sql`이 적용되면
**Then** `users.department_id → departments(id)` FK가 추가되어, 존재하지 않는 진료과로의 직원 배정이 DB에서 차단된다(참조 무결성). 기존 직원 행은 `department_id`가 모두 NULL이라 FK 추가가 안전하다.

> **AC 해석(중요·dev agent 필독):** 이 스토리는 **departments·rooms 마스터 CRUD + soft-delete 토글 + 감사·RLS·FK**까지다. **코드 마스터(KCD/EDI/약품)는 Story 2.2**, **재사용 검색 피커는 Story 2.3**, **참조 무결성 심화 검증은 Story 2.4**, **시드는 Story 2.5**, **직원 생성폼의 진료과 피커는 범위 밖(§Non-goals)**. 2.1은 그 다운스트림이 의존할 "정확한 조직 단위 테이블 + 관리 화면 + 단일 진실"을 세우는 일이다. 새 라이브러리·새 권한·새 nav 항목 도입 금지(모두 기존 자산 재사용).

## Dev Notes

> **이 스토리의 핵심:** Epic 1이 토대를 전부 깔아놨다 — **권한 `master.manage`는 이미 시드됨**(0002:102, admin 보유), **nav `/admin/masters`도 이미 존재**(staff-nav.ts, `requiredPermission: master.manage`), **감사 트리거 함수·관리자 화면 패턴·쓰기 권위 패턴 모두 확립**. 2.1은 사실상 **"첫 비-신원 도메인 테이블(departments·rooms) + 기존 관리자 화면 패턴으로 CRUD 화면 1개 + soft-delete 토글"**을 만들고 **FK·감사·RLS를 정석대로 거는** 일이다. retro(epic-1-retro)가 Epic 2 착수 액션을 명시했으니 그대로 따른다.

### 🧭 읽기 경로 결정 — Supabase 직접조회 (1.10 감사뷰어와 반대!)

마스터 **목록 읽기 = web의 Supabase 직접조회**(authenticated SELECT, RLS `using(true)`). **쓰기만 FastAPI**(`master.manage`). 근거(retro 액션 "마스터 읽기 = 1.7식 Supabase 직접조회 확립"):

1. **마스터는 전역 참조 데이터** — `users`처럼 본인행 RLS가 아니다. 모든 직원이 피커/조회로 읽어야 하므로 `authenticated SELECT using(true)`로 직접조회 가능 → **actor 조인·service_role 우회가 불필요**(1.10 감사뷰어가 FastAPI를 택한 이유들이 여기엔 없음).
2. **`set-state-in-effect` 마찰 회피** — RSC가 `createClient()`로 직접조회해 `initial`을 클라 컴포넌트에 주입(permissions/page.tsx 패턴). 1.8/1.10의 마운트 클라 fetch + 린트 예외가 **불필요**.
3. **쓰기는 FastAPI 단일 권위** — authenticated는 마스터에 SELECT만 갖는다(Task1 GRANT). 생성·수정·비활성은 반드시 FastAPI(service_role + `master.manage` 게이트 + 동일트랜잭션 재평가) 경로로만. RLS는 방어심층 2차선으로 유지.

> **요약:** 읽기=permissions/page.tsx 패턴(Supabase 직접·RSC 주입), 쓰기=admin.py 패턴(FastAPI·master.manage). users(1.8) 패턴(전부 FastAPI)을 복제하지 말 것 — 마스터는 본인행 RLS 마찰이 없다.

### 🗂️ 스키마 설계 (0006_masters.sql — 정확히 이 형태)

| 테이블 | 핵심 컬럼 | 비고 |
|---|---|---|
| `departments` | `id`(uuid PK) · `code`(unique) · `name` · `description` · `is_active`(default true) · `created_at`/`updated_at` | 조직 단위(정형외과 등). `code`=식별 코드 값(불변), `name`=한글 표시명 |

### 🔐 쓰기 권위 패턴 (그대로 복제 — 재발명 금지)

[Source: api/app/core/db.py:138-288 `set_role_permission`·`insert_staff_profile`]

```
async def insert_department(sub, *, code, name, description):
    async def _op(conn):
        # 1) 권한 재평가(평가↔쓰기 원자성, TOCTOU 차단) — 1.5 이월·retro 액션
        if not bool(await conn.fetchval("select public.has_permission('master.manage')")):
            raise ForbiddenError(detail={"required_permission": "master.manage"})
        # 2) INSERT returning (0004 trg_departments_audit 가 자동 감사, actor=app.actor_id)
        try:
            row = await conn.fetchrow("insert into public.departments (...) values (...) returning ...", ...)
        except asyncpg.UniqueViolationError as exc:
            raise ConflictError("이미 사용 중인 코드입니다.", code="code_taken", ...) from exc
        return row
    return await _run_authed(sub, _op)
```

- **actor 캡처:** `_run_authed` → `authenticated_conn(sub)`가 `set_config('app.actor_id', sub, true)` 주입(db.py:103) → 감사 트리거가 actor를 정확히 기록. **앱은 감사 INSERT를 직접 하지 않는다**(트리거 소유).
- **에러 매핑:** 권한 403(ForbiddenError) · 중복 409(ConflictError) · 미존재 404(NotFoundError) · FK 위반 422(AppError `invalid_department`) · DB 일시장애 503(`_run_authed`). errors.py 예외 클래스 재사용.
