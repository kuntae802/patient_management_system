# Story 6.5: 환자 앱 예약

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a **환자**,
I want **앱에서 진료과·의사·시간을 골라 예약하기를**,
so that **전화 없이 스스로 예약한다.**

## Acceptance Criteria

**[AC1] 환자 앱 예약 흐름 — 가능 슬롯만 선택 (FR-010)**
- **Given** 환자 앱 예약 탭(UX-DR17)에서 (본인 진료기록이 연결된 환자가)
- **When** **진료과 → 의사 → 날짜 칩 레일 → 시간 슬롯 그리드** 흐름을 진행하면
- **Then** 휴진일·마감(이미 예약됨)·지난 시각 슬롯은 **비활성**으로 표시되고 **가능(available) 슬롯만 선택**된다. 슬롯은 직원 슬롯 계산(6.2 `compute_available_slots` = 근무−휴진−booked예약)을 **환자 스코프 읽기 경로**로 재사용한다.

**[AC2] 예약 확정 — 본인 예약 생성·12시간 표기·쉬운 말 확인**
- **Given** 진료과·의사·날짜·시간 선택을 마쳤을 때
- **When** **"예약 확정하기"(≥44px sticky CTA)** 를 누르면
- **Then** 예약이 생성되고(`status='booked'`), **12시간 표기(오후 2:30)** 와 쉬운 말로 확인이 표시된다. **예약 환자 = 세션 본인**(`patient_id` 는 서버가 JWT 주체→`patients.auth_uid` 로 도출 — 클라가 `patient_id` 미제공, 세션 uid 스코프). 더블부킹은 인라인 차단(409)된다.

**[AC3] 보안 경계 — 본인만·직원 차단·미연결 차단**
- **Given** 환자 예약 엔드포인트(`/scheduling/me/*`)에 대해
- **When** 직원(active 5역할) 토큰으로 호출하면 → **403**(`get_current_patient` 반전). 본인 진료기록 **미연결** 환자(자가가입했으나 self-link 미완)가 예약을 시도하면 → **연결 안내**(409 `no_self_patient` → 온보딩 유도).
- **Then** 환자는 **타인 예약을 생성할 수 없다**(patient_id 가 클라 입력이 아니라 세션에서만 도출되므로 구조적으로 불가). 슬롯·의사 조회 응답에 **환자 PII 누설 없음**(가용성만).

**[AC4] 가용 슬롯·캘린더 반영**
- **Given** 환자가 예약을 생성하면
- **When** 슬롯 계산(6.2)·원무 캘린더(6.3/6.4)를 다시 조회하면
- **Then** 해당 슬롯은 **마감(booked → 캘린더 confirmed)** 으로 반영된다(EXCLUDE·overlay 가 `status='booked'` 기준이므로 자동). 원무 대리 예약(6.4)과 동일 `appointments` 테이블·동일 더블부킹 불변식을 공유한다.

> **범위 노트(Story 6.5):** 본 스토리는 **환자 본인 예약 생성**(앱)만 다룬다. **예약 변경·취소(앱 "마이 메뉴")** = 이월(UX-DR17 CTA 도움말 "예약 후 변경·취소는 마이 메뉴에서" → 마이 탭 = Epic 8/후속). **SMS 실 발송** = 6.6(`sms_opt_in` 은 본 스토리가 저장만). **노쇼 임계 제한**(상습 노쇼 예약 차단) = 6.7(본 스토리는 차단 로직 미포함). **내 기록·마이 탭 내용** = Epic 8. **하단 3탭 셸**은 최소 스캐폴드만(예약 탭 활성·나머지 placeholder).
>
> **모델 결정(사용자 확정 — Option C):** `appointments.created_by`(현재 `not null references public.users(id)`)는 환자 auth sub 가 `users` 에 없어 FK 위반 → **0034 가 FK 만 제거**하고 `created_by` 를 **비정규화 생성자 auth uid**(직원 uid 또는 환자 auth_uid)로 의미 확장한다. `audit_logs.actor_id` 선례(직원·환자 uid 혼용·FK 미부착·문서화됨, [0004_audit.sql:5-10])와 일치. NOT NULL 유지·직원 예약 무영향.

### 확정 설계 요약 (환자 본인 예약 — 세션 uid 스코프)
**환자 = 본인 예약만.** 환자는 Supabase Auth 인증(비직원)이고 RBAC 권한이 **0** 이다. 권위는 "본인 JWT sub 로만"이며([security.py:147-158]), 예약의 `patient_id` 는 **서버가 `auth_uid = sub` 로 도출**한다(클라가 `patient_id` 미제공 — 3.4 self-link/get_self 선례와 동일). 직원 대리 예약(6.4)이 `appointment.create` 권한 + 임의 `patient_id` 인 것과 **대칭**: 환자 경로는 권한 게이트가 아니라 **자기 스코프**가 권위다. 따라서 교차환자 예약은 구조적으로 불가(IDOR 차단). 게이트 = `get_current_patient`(직원 5역할 → 403). 미연결 환자(자가가입했으나 self-link 미완)는 예약 불가 → 온보딩(3.4) 유도.

**Option C(created_by 비정규화):** `appointments.created_by` 는 0031 에서 `not null references public.users(id)` 였으나, 환자 auth sub 는 `users` 에 없다(분리 프로필). 0034 가 **FK 만 제거**해 `created_by` 를 비정규화 생성자 auth uid(직원 uid 또는 환자 auth_uid)로 의미 확장한다. `audit_logs.actor_id`([0004_audit.sql:5-10] — "직원·환자의 auth uid", "FK 미부착 이유" 명시)와 동일 패턴. NOT NULL·직원 예약 무영향.
