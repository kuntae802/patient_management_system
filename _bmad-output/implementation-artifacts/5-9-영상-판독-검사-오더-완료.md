# Story 5.9: 영상 판독 · 검사 오더 완료

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a **의사(판독의 겸임, doctor)**,
I want **판독 워크리스트에서 촬영 수행(performed)된 영상검사의 영상을 보고 판독 소견·결론을 기록하면 해당 검사 오더가 '완료(completed)'로 닫히기를**,
so that **검사 생명주기(`ordered → performed → completed`)가 닫히고 판독자·시각·소견이 추적성 있게 남는다.**

> **에픽 범위 노트:** 5.9는 영상검사 생명주기의 **마지막 구간 `performed → completed`(판독의 측 '판독·완료')**만 소유한다. 촬영 수행(`ordered → performed`, 영상 업로드·장비 배정·방사선사)은 **Story 5.8**이 이미 완료했다 — 워크리스트(`status='ordered'`)·업로드·`perform_examination`·장비 화면은 5.9에서 다시 만들지 말 것. **수가 자동발생(판독 완료 → 영상 수가 적재)은 Story 5.10**(트리거 가동) — 5.9는 영상 수가를 직접 적재하지 않는다. **진단검사(lab) 완료(외부 의뢰 결과 반영, FR-061)는 5.9 범위 밖**(imaging 전용, 5.8 posture 계승). **판독 정정/애드덤(완료 후 소견 수정)은 이월** — 5.9에서 `completed`는 종결 상태(재완료 = `PT409`).

## Acceptance Criteria

> 출처: `epics.md#Story-5.9`(FR-102) + 본 스토리 확정 결정 3건([사용자 확정] ① 판독지 = `findings`(소견·필수) + `reading_conclusion`(결론·선택) 2-필드, examinations 컬럼 추가 ② `completed` 종결·정정 이월 ③ `/doctor/radiology` 판독 워크리스트 신규 = 5.8 방사선사 워크리스트와 대칭).

**AC1 — 판독 워크리스트 (판독 대기 = performed)**
- **Given** 의사(`examination.complete` 보유)로 로그인한 상태에서
- **When** 판독 워크리스트(`GET /radiology/reading-worklist`)를 조회하면
- **Then** **오늘(KST) 활성 내원의 영상검사 오더 중 촬영 수행됐으나 미판독(`exam_type='imaging' AND status='performed'`)인 항목**이 수행 시각 오름차순(FIFO)으로 표시된다. 각 행은 `examination_id·encounter_id·chart_no·환자명·진료과명·검사 행위명(fee_name)·지시 의사명(ordered_by_name)·수행자명(performed_by_name)·수행 시각(performed_at)·지시 시각(ordered_at)·업로드된 영상 수(image_count)`를 포함하고, **raw 주민번호 등 PII는 포함하지 않는다**(비-PII 프로젝션, 5.8 워크리스트 미러).

**AC2 — 영상 조회(판독 근거) = 서명 URL 재사용 (5.8 자산)**
- **Given** 판독 워크리스트에서 검사를 선택했을 때
- **When** 그 검사의 영상 목록(`GET /examinations/{examination_id}/images`)을 조회하면
- **Then** 각 영상의 메타데이터 + **서버 발급 단기 서명 URL(`signed_url`, TTL ≤5분)**이 반환된다(5.8이 만든 엔드포인트를 **그대로 재사용** — `order.read` 게이트, 의사 보유). 5.9는 이 엔드포인트를 **신규 구현하지 않는다**. 서명 URL은 DB에 저장하지 않고 요청 시점마다 재생성한다.

**AC3 — 판독 소견·결론 기록 → 검사 완료(performed→completed) (FR-102)**
- **Given** 촬영 수행된 영상검사 오더(`status='performed'`, `exam_type='imaging'`)에 대해
- **When** 판독 소견을 기록하고 완료(`POST /examinations/{examination_id}/complete`, body `{findings, reading_conclusion?}`)를 호출하면
- **Then** `findings`(소견·**필수·non-blank**)·`reading_conclusion`(결론·선택)이 `examinations` 행에 저장되고, `complete_examination` RPC로 `status: performed → completed` 전이되며 `completed_by`(= 호출 의사 `auth.uid()`)·`completed_at`이 기록된다. 소견 쓰기와 완료 전이는 **단일 트랜잭션에서 원자적**으로 일어난다(전이 거부 시 소견 쓰기도 롤백).
- **And** 판독은 **중소병원 외래 기준 진료의 겸임이 허용**된다 — 별도 판독 전문의 직역 분리 없이 `examination.complete` 권한(doctor 보유)으로 게이트한다.

**AC4 — 전제조건·재완료 차단·검증 (FR-093 · 3중 검증)**
- **Given** 판독 완료를 호출할 때
- **When** 대상 오더의 상태/유형/소견이 전제에 어긋나면
- **Then** ▸ `findings`가 빈/공백-only → **422 `findings_required`**(서버 Pydantic 권위 + DB CHECK 최종선) ▸ 아직 미수행(`status='ordered'`) 또는 이미 판독(`status='completed'`)된 오더 재완료 → **409 `invalid_transition`**(DB 상태머신 최종선·소스상태 precondition, FR-093) ▸ lab 오더 판독 시도 → **422 `not_imaging`** ▸ 미존재 검사 → **404** ▸ `examination.complete` 미보유(방사선사·간호·원무) → **403**.

**AC5 — 판독 후 잠금(완료 = 종결) (UX-DR21 ⑤·⑧)**
- **Given** 이미 판독 완료(`completed`)된 검사에 대해
- **When** 판독 워크리스트·판독 패널을 보면
- **Then** 해당 검사는 워크리스트에서 사라지고(미판독 목록 = `performed`만), 판독 패널은 **잠금 상태**(소견·결론 read-only, 완료 버튼 비활성, `completed_by`/판독 시각 표시)로 렌더된다. 완료 mutation 중에는 버튼 disable(이중 제출 가드 1차선, UX-DR21 ⑤). 정정/재완료는 5.9 범위 밖이다.

**AC6 — 감사 마스킹 · 회귀 · 게이트 · 웹**
- 판독 소견(`findings`)·결론(`reading_conclusion`)은 **자유 임상 서사** → 감사 스냅샷 서버측 마스킹(`audit.py` `_SENSITIVE_KEY` + `audit.ts` `SENSITIVE_KEY` **양쪽 동기화**)에 등록되어 API 본문·구조적 로그로 평문이 새지 않는다(Story 3.6 / 5.7 `content` 선례).
- 기존 검사·영상 **오더 생성/조회(5.3)·촬영 수행/영상 업로드/장비(5.8)·다른 오더 도메인(처방 5.2·처치 5.4)·간호(5.6/5.7)** 회귀 0. `ExaminationResponse`에 `findings`/`reading_conclusion` 필드 추가는 **가산적**(기존 응답에 null 동반 — 기존 테스트 무파손).
- 신규 web 라우트 `/doctor/radiology`(nav 기선언)가 **의사 역할에만 노출**되고 서버 가드(`requirePermission("examination.complete")`)로 직접 진입을 차단한다.
- 비-의사(방사선사·원무 등) API 직접 호출 403, 검증 실패 422, 잘못된 전이 409, 미존재 404 — 표준 에러 봉투(`{error:{code,message,detail}}`).
