# Story 5.6: 간호 활력징후 기록

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 간호사,
I want 활력징후(혈압·맥박·체온·호흡수·SpO2)를 측정·기록하기를,
so that 의사가 진료 전 진료 허브에서 활력 데이터를 본다.

## Acceptance Criteria

1. **(FR-091) 활력징후 전용 기록 저장·내원 연결** — `0017_nursing.sql`의 `vital_signs`에 대해, 간호사가 혈압(수축기/이완기)·맥박·체온·호흡수·SpO2를 기록하면 활력징후 전용 레코드가 저장되고 해당 내원(`encounter_id`)에 연결된다. 모든 활력 수치는 **항목별 선택 입력**(부분 측정 허용)이되 **최소 1개 측정값**이 있어야 하며(전부 빈 값 → 422), 기록자(`recorded_by`)·기록시각(`recorded_at`)이 자동 부착되고 감사 로그가 남는다.

2. **(FR-032 연동) 의사 진료 허브 좌 컨텍스트 패널 표시** — 기록된 활력징후는 의사가 진료 허브(`/encounter/{encounterId}`)를 열면 **좌측 컨텍스트 패널의 "활력징후" 카드**에 표시된다(현재의 빈-상태 플레이스홀더를 실데이터로 교체). 최신 측정값 + 측정시각·측정자가 보이고, 정상 범위 밖 수치는 시각적으로 구분된다.

3. **(진입점) 간호 활력 워크리스트** — 간호사는 `/nurse/vitals`에서 **오늘 활성 내원**(`registered`·`in_progress`) 목록을 보고(환자명·차트번호·진료과·상태·최근 활력 시각), 한 건을 선택해 활력 입력 폼으로 진입한다. 기록 권한(`vital.record`) 없는 직역은 워크리스트·기록 모두 차단된다.

### ⚠️ 비자명 핵심 결정 (먼저 읽을 것)

1. **`vital.record`는 신규 권한이 아님** — 0002_identity_rbac.sql:97에 이미 시드(`('vital.record','활력징후 기록','vital','record')`). admin은 0002 cross-join으로 보유. **0017에서 permissions INSERT·admin 재grant 모두 불요**(5.2 `prescription.create` 선례와 동형 — "신규 마이그/신규 권한/admin 부트 grant = 전부 0"의 권한 부분). seed.sql에서 **nurse grant 1건만** 추가. `test_admin_role_has_all_permissions` 회귀 0.

2. **FastAPI = service_role(슈퍼유저) 풀 → RLS 우회. 조회 권위 = 라우터 `require_permission`** [db.py:3, 1631, 1869, 2033]. 즉 **간호사가 `encounter.read` 권한이 없어도** FastAPI 경유 워크리스트/활력 조회는 라우터 게이트(`vital.record`)만 통과하면 동작한다(RLS 행 필터 미적용). RLS 정책(`encounter.read OR vital.record` + self)은 **환자 포털 Supabase 직결 경로(Story 8.2 미래) 대비 방어심층** — FastAPI 경로엔 무영향이지만 일관성·미래 대비로 반드시 작성.
   - ❌ **절대 nurse에게 `encounter.read`를 grant하지 말 것** — nurse의 `encounter.*`/`patient.*` 권한 0이 Story 4.4/4.5의 403 baseline이며, 부여 시 `test_*_forbidden_nurse_403`(encounter 엔드포인트) 회귀.

3. **활력 입력 진입 = `/nurse/vitals` 워크리스트(사용자 확정)** — 간호사는 "오늘 활성 내원" 목록(`vital.record` 게이트)에서 환자를 선택해 기록한다. 이 워크리스트는 **Story 5.7 처치 워크리스트의 토대**(5.7이 동일 진입을 처치 수행으로 확장). encounter.read=0 유지하며 nurse가 작업 대상을 찾는 유일·실제적 경로.

4. **활력 필드 = 항목별 선택 + 최소 1개 강제(사용자 확정)** — 부분 측정(혈압+체온만 등)이 흔한 실제 임상 반영. 빈 활력 행 차단은 **3중 방어**: 클라 Zod(제출 disable) → 서버 Pydantic `model_validator`(422) → DB CHECK(최종선). 처방 `details min_length=1`(빈 처방전 무의미) 패턴과 동형.

```sql
create table if not exists public.vital_signs (
  id               uuid primary key default gen_random_uuid(),
  encounter_id     uuid not null references public.encounters (id),
  systolic         integer,        -- 수축기 혈압 mmHg
  diastolic        integer,        -- 이완기 혈압 mmHg
  pulse            integer,        -- 맥박 bpm
  body_temp        numeric(4,1),   -- 체온 °C
  respiratory_rate integer,        -- 호흡수 /min
  spo2             integer,        -- 산소포화도 %
  notes            text,           -- 임상 주석(선택)
  recorded_by      uuid not null references public.users (id),
  recorded_at      timestamptz not null default now(),
  is_active        boolean not null default true,
  created_at       timestamptz not null default now(),
  updated_at       timestamptz not null default now(),
  constraint vital_signs_systolic_range  check (systolic is null or (systolic between 50 and 300)),
  constraint vital_signs_diastolic_range check (diastolic is null or (diastolic between 20 and 200)),
  constraint vital_signs_pulse_range     check (pulse is null or (pulse between 20 and 300)),
  constraint vital_signs_temp_range      check (body_temp is null or (body_temp between 30.0 and 45.0)),
  constraint vital_signs_resp_range      check (respiratory_rate is null or (respiratory_rate between 4 and 80)),
  constraint vital_signs_spo2_range      check (spo2 is null or (spo2 between 50 and 100)),
  constraint vital_signs_at_least_one    check (
    systolic is not null or diastolic is not null or pulse is not null
    or body_temp is not null or respiratory_rate is not null or spo2 is not null
  )
);
create index if not exists idx_vital_signs_encounter_id on public.vital_signs (encounter_id);
create index if not exists idx_vital_signs_recorded_at on public.vital_signs (recorded_at);
```

### API 패턴 (orders → nursing 미러)

**`security.py` 신규 `require_any_permission`** (require_permission@125 미러):

```python
def require_any_permission(*codes: str) -> Callable[..., Awaitable[CurrentUser]]:
    """codes 중 하나라도 보유 시 통과. 전무 시 403. (활력 GET = doctor encounter.read ∨ nurse vital.record)"""
    async def _dependency(user: CurrentUser = Depends(get_current_user)) -> CurrentUser:
        for code in codes:
            if await db.fetch_has_permission(user.sub, code):
                return user
        raise ForbiddenError(detail={"required_any_permission": list(codes)})
    return _dependency
```

select e.id as encounter_id, p.chart_no, p.name as patient_name, d.name as department_name,
       e.status, e.created_at,
       (select max(v.recorded_at) from public.vital_signs v
        where v.encounter_id = e.id and v.is_active) as latest_vital_recorded_at
from public.encounters e
join public.patients p on p.id = e.patient_id
