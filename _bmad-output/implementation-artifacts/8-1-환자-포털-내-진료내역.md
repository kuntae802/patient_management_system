# Story 8.1: 환자 포털 — 내 진료내역

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a **환자**,
I want **앱에서 본인의 내원 이력(예약·진찰·진단)을 쉬운 말로 보기를**,
so that **내 진료가 어땠는지 스스로 이해한다**.

> **Epic 8 진입 — 첫 환자-대면 읽기 스토리.** Epic 1~7이 쌓은 누적 데이터(내원·진단) 위에 올라가는 **읽기 전용 뷰**다. 환자 포털은 항상 세션 uid로 스코프되고 본인 외 데이터는 0건이어야 한다(성공지표). 새 임상 데이터를 만들지 않는다 — 기존 데이터를 환자 언어로 비춘다.

## Acceptance Criteria

**AC1 — 내 기록 탭에 내원 이력 카드 (FR-120, UX-DR17)**
**Given** 환자 세션(uid 스코프)에서
**When** "내 기록" 탭을 열면
**Then** 본인 내원 이력 카드가 **최근순**으로 표시되고, 각 카드에 **날짜 · 상태 배지(A3·환자 톤) · 의사명/진료과 · 진단(쉬운 말 부연)** 이 표시된다.

**AC2 — RLS 본인 스코프 + 신뢰 노트 상시 (UX-DR22)**
**Given** RLS·세션 uid 경계가 강제되는 상태에서
**When** 데이터를 받으면
**Then** **본인 데이터만** 반환되고(타인 0건), 화면 상단에 **신뢰 노트**가 상시 표시된다 — "〇〇〇 님 본인의 정보만 안전하게 표시됩니다. 다른 사람은 볼 수 없어요."

**AC3 — 진단 쉬운 말 부연**
**Given** 카드에 부착된 주상병(primary diagnosis)이 있을 때
**When** 진단을 렌더하면
**Then** KCD 진단명 옆에 **쉬운 말 부연**이 괄호로 표시된다(예: "고혈압 (혈압이 높은 상태)"). 부연이 없는 진단은 진단명만 표시한다(우아한 폴백).

**AC4 — 상태별 카드 본문 분기**
**Given** 내원 상태가 6값(scheduled/registered/in_progress/completed/cancelled/no_show) 중 하나일 때
**When** 카드를 렌더하면
**Then** **취소** 내원은 진단 대신 **취소 사유**(쉬운 말)를 보여주고, **진단이 없는 내원**(예정·접수·진료중)은 진단 줄을 생략하거나 상태에 맞는 안내를 보여준다. 색에 의존하지 않는다(글리프+라벨, UX-DR20).

**AC5 — 시간 표기 · 빈/예외 상태**
**Given** 환자 화면에서
**When** 날짜·시간을 표기하면
**Then** **12시간(오후 2:30)**·KST·쉬운 말이 적용된다(직원 24시간과 별개). 내원 이력이 **0건**이면 빈 상태 안내("아직 진료 내역이 없어요")를, **미연결 환자**(self 레코드 없음)면 온보딩으로 유도한다.

### 이 스토리의 본질 (왜 이렇게 작은가)
데이터·RLS 기반이 **이미 다 깔려 있다**. 8.1은 "본인 내원을 환자 언어로 비추는 읽기 뷰" 한 겹을 올린다:
- **읽기 경계(self-scope)** = 기존 `encounters_select_self`(0010)·`encounter_diagnoses_select_self`(0014) RLS + `get_current_patient` 게이트 + `auth_uid=sub` WHERE. 새 보안 메커니즘 0.
- **데이터** = encounters(0010)·encounter_diagnoses(0014)·diagnoses(0007)·appointments(0031) — 전부 존재. 신규 = `patient_friendly_note` 컬럼 1개(쉬운 말 소스).
- **실작업** = ① 진단 쉬운 말 컬럼+시드 ② 환자 self 내원 조회 엔드포인트 ③ 환자 폰 셸+내 기록 화면.

### 🔒 보안 경계 — service_role 현실 (가장 중요)
**FastAPI는 service_role(로컬 postgres 슈퍼유저) 연결 → RLS를 우회한다**(`api/app/core/db.py:3` 주석 확인). `_run_authed(sub, ...)` 는 `request.jwt.claims={sub, role:authenticated}` 를 `set_config` 로 주입해 `auth.uid()` 만 해석되게 할 뿐, **RLS 필터는 적용되지 않는다**.
- ⇒ 본인 스코프는 **반드시 쿼리의 `where p.auth_uid = $1`(=sub) 로 명시**한다. RLS에 의존하지 말 것(RLS는 Supabase 직접조회 경로의 심층방어).
- ⇒ service_role이므로 `users`(의사명)·`departments`·`diagnoses` 조인이 자유롭다(환자 JWT 직접조회였다면 `users_select_self`(0003)에 막혀 의사명 NULL이 됐을 것 — 그래서 FastAPI 경유가 맞다).
- **안티패턴 금지**: 클라이언트 Supabase 직접조회로 내원/진단을 읽지 말 것(의사명 조인이 `users` RLS에 막히고, 컬럼 투영 통제가 안 돼 PII 경계가 약해진다). 환자 임상 데이터의 큐레이션 투영은 FastAPI가 소유. (참조 데이터인 `departments` 직접조회는 기존 예약 화면 선례지만 내원 이력은 다르다.)

### 데이터 모델 사실 (확인 완료)
- **encounters**(`0010_encounters.sql:28`): status 6값(scheduled/registered/in_progress/completed/cancelled/no_show), `patient_id`·`department_id`·`doctor_id`·`visit_type`(walk_in/reserved)·`registered_at`·`consult_started_at`·`completed_at`·`cancelled_at`·`cancel_reason`·`reservation_id`(0031, walk-in은 NULL)·`is_active`·`created_at`. 본인 RLS `encounters_select_self`(0010:278).
- **encounter_diagnoses**(`0014:18`): `is_primary`(주상병·활성 ≤1/내원), `diagnosis_id` FK. 본인 RLS `encounter_diagnoses_select_self`(0014:66).

- **appointments**(`0031:28`): `scheduled_start`/`scheduled_end`·`status`. `encounters.reservation_id` FK(0031:62).
