**AC1 — `payments` 테이블(수납 헤더 — 내원 1:1):**
`0045_payments.sql`에서 `payments` 테이블이 생성된다 — 한 외래 내원의 정산 헤더(내원 1:1). 핵심 컬럼:
- `encounter_id` uuid NOT NULL **UNIQUE** FK → `encounters`: 내원당 수납 1건(1:1 불변식).
- `status` text NOT NULL default `'draft'` CHECK in (`draft`, `finalized`, `cancelled`): `draft`=집계 전/중(7.2) · `finalized`=결제 완료(7.4) · `cancelled`=취소/노쇼 미발생(7.9).
- `billing_type` text NOT NULL default `'postpaid'` CHECK in (`postpaid`, `prepaid`): 후수납 기본 / 선수납(7.8 소비 — 아키텍처 §179 ④).
- **금액 집계 컬럼**(전부 integer NOT NULL default 0 CHECK `>= 0` — KRW 정수. **7.2 집계·7.3 산정·7.4 결제가 채움**, 7.1은 컬럼·불변식·기본값 0만): `total_amount_krw`(총 진료비=급여+비급여) · `covered_amount_krw`(급여 대상) · `non_covered_amount_krw`(비급여) · `copay_amount_krw`(본인부담금=환자 청구) · `insurer_amount_krw`(공단부담금) · `paid_amount_krw`(이미 납부 — 선수납 7.8).
- **결제 컬럼**(7.4 finalize가 채움): `payment_method` text CHECK in (`card`, `cash`, `transfer`)(nullable) · `payment_no` text UNIQUE(영수증/수납번호 — nullable, finalize 시 부여 — 7.4/7.5 문서) · `finalized_at` timestamptz · `finalized_by` uuid FK → `users` · `cancelled_at` timestamptz · `cancel_reason` text.
- `created_at`/`updated_at` timestamptz NOT NULL default now().

**AC2 — `payment_details` 테이블(수납상세 라인 — `fee_items` 집계 대상):**
`payment_details` 테이블이 생성되어 7.2가 `fee_items`를 집계해 적재할 라인 항목 구조를 제공한다. 핵심 컬럼:
- `payment_id` uuid NOT NULL FK → `payments` **ON DELETE CASCADE**(헤더 삭제 시 라인 동반 — draft 정리).
- `fee_item_id` uuid FK → `fee_items`(nullable — 출처 수가항목 역추적·집계원. nullable은 수기 가산 라인 여지 7.x·노쇼 수수료 7.9).
- **스냅샷 컬럼**(집계 시점 `fee_items`/`fee_schedules`에서 복사 — 청구 시점 고정): `fee_schedule_id` uuid FK → `fee_schedules`(nullable) · `code` text(EDI 코드) · `name` text(행위명) · `category` text(분류) · `quantity` integer NOT NULL default 1 CHECK `> 0` · `unit_amount_krw` integer NOT NULL CHECK `>= 0` · `amount_krw` integer NOT NULL CHECK `>= 0` · `coverage_type` text NOT NULL CHECK in (`covered`, `non_covered`).
- **본인부담 산정 컬럼**(7.3가 채움 — 7.1은 컬럼만): `copay_rate` numeric(4,3)(적용 본인부담률 스냅샷 예 `0.300`, nullable) · `copay_amount_krw` integer NOT NULL default 0 CHECK `>= 0` · `insurer_amount_krw` integer NOT NULL default 0 CHECK `>= 0`.
- 불변식: **`unique (payment_id, fee_item_id)`**(같은 수가항목이 한 수납에 2회 집계되지 않게 — fee_items 멱등의 집계측 거울. NULL fee_item_id 다중 허용 = 수기 라인). **`check (amount_krw = quantity * unit_amount_krw)`**(AC5).
- 인덱스: `idx_payment_details_payment_id` on (payment_id).

**AC3 — 진찰료 초진/재진 동적 판정(FR-116 진찰료=진찰 시 · 청구 단순화 선 = 동적):**
`fee_mappings`가 `source_event` CHECK를 `in ('encounter_start', 'encounter_start_initial', 'encounter_start_repeat')`로 확장하고, `fee_on_encounter_start()` 트리거가 **재정의**되어 초진/재진을 동적 판정한다:
- 판정 규칙(사용자 확정 — **이력 유무**): 동일 환자(`new.patient_id`)에 **과거 완료 내원**(`status='completed'`, `id <> new.id`)이 존재하면 **재진**(`encounter_start_repeat` → AA254 재진진찰료), 없으면(첫 방문) **초진**(`encounter_start_initial` → AA154 초진진찰료).
- 트리거는 판정 후 해당 `source_event`의 활성 매핑을 룩업해 `insert_fee_item`으로 적재. 해당 매핑이 없으면 **레거시 `encounter_start` 매핑으로 폴백**(5.10 단일 매핑 하위호환 — 폴백도 없으면 no-op·예외 아님).
- 트리거 부착(`AFTER UPDATE OF status` + `WHEN (old.status='registered' AND new.status='in_progress')`)·`source_type='encounter'`·`source_id=new.id`·금액 스냅샷은 5.10 메커니즘 불변. 30일 재진규칙·진료과 가산은 **미구현**(단순화 선 — Dev Notes §경계).

**AC4 — 만료·비활성 수가 적재 제외(5.10 이월 흡수 — 적재 시점 검증):**
`insert_fee_item()`이 **재정의**되어, `fee_schedules` 룩업 시 `id` 일치뿐 아니라 **현재 유효** 술어를 함께 적용한다: `is_active AND effective_from <= current_date AND (effective_to IS NULL OR effective_to >= current_date)`. 만료·미발효·비활성 수가는 **적재되지 않는다**(no-op return — 만료/폐지 EDI 코드의 stale 청구 차단, 0007 "현재 유효" 계약 충족). 유효 시점에 적재된 `fee_items`의 금액 스냅샷은 마스터가 후에 만료돼도 보존(적재 시점 고정 의도와 양립). [출처: deferred-work.md L344 — 5.10 코드리뷰 Edge Case Hunter 실증("AA254를 `effective_to=어제`/`is_active=false`로 만들어도 12590 적재")의 해소.]

**AC5 — `amount = quantity * unit_amount` 불변식 CHECK(5.10 이월 흡수):**
`payment_details`에 `check (amount_krw = quantity * unit_amount_krw)` 추가(집계가 quantity 가변·금액 정합 강제) + 기존 `fee_items`에도 동일 CHECK를 `ALTER TABLE ... ADD CONSTRAINT`로 추가(5.10 quantity 항상 1이라 기존 행 전부 충족 → 무회귀). [출처: deferred-work.md L346 — "Epic 7 quantity 가변 도입 시 add" 해소.]

### ⚠️ 초진/재진 동적 판정 — 트리거 변경의 핵심

5.10은 `fee_mappings` 단일 매핑(`encounter_start`→AA254 재진 고정)이었다(아키텍처 §445 "초진/재진=수납 에픽" 존중하여 보류). 7.1이 그 결정을 내려 **동적 판정**으로 전환한다:
- **판정 술어(이력 유무)**: `EXISTS (select 1 from encounters where patient_id=new.patient_id and id<>new.id and status='completed')` → 참=재진. 거짓(첫 방문)=초진. `encounters.patient_id`는 인덱스 존재(0010 `idx_encounters_patient_id`)·`status`도 인덱스 → EXISTS 저렴.
- **매핑 2분기**: source_event를 `encounter_start_initial`(→AA154)·`encounter_start_repeat`(→AA254)로 외부화 → 코드 수정 없이 시드로 진찰료 코드 관리(FR-116 외부화 정신 유지). 트리거는 판정 결과 event를 룩업.
- **하위호환 폴백**: 분기 매핑 부재 시 레거시 `encounter_start` 매핑으로 폴백 → 5.10 단일 매핑 환경에서도 동작(seed가 둘 다 시드하므로 정상 경로는 분기 사용).

- **AC4 적재 시점 검증**: `insert_fee_item`이 `fee_schedules`를 `id`만으로 룩업(5.10) → **현재 유효 술어 추가**. 만료/폐지 코드가 stale 금액으로 청구되는 갭(deferred-work L344·Edge 실증) 차단. "적재 시점 스냅샷"과 양립(유효 시점에 잡힌 금액은 후 만료돼도 보존 — 스냅샷 의도; 단 *신규* 적재는 유효한 것만).
- **청구 시점 검증**: 적재 시점에서 만료 코드를 거르면 `fee_items`엔 유효-적재분만 남음 → 7.2 집계는 자연히 청결. finalize(7.4)가 2차 방어(문서 출력 전 정합)는 7.4 점검 항목으로 위임.
- **AC5 amount CHECK**(deferred-work L346): 7.1이 `payment_details.quantity` 가변(집계)을 도입하므로 `amount=quantity*unit` CHECK 필수 + `fee_items`에도 소급(quantity=1이라 무회귀).
- **이월 지속**(만기 아님): 종결 내원 수가 적재 게이트(L345)·앱 낙관적 잠금·is_active soft-delete = DB 최종선/오더-내원상태 게이트 정책 스토리 소관(Epic 4 결정 계승).

### 적재 경로 · SECURITY DEFINER 보존

`payments`/`payment_details`는 쓰기 RLS 정책 없음(GRANT도 authenticated SELECT만) — 쓰기는 service_role(FastAPI 7.2/7.4). `fee_items` 적재는 SECURITY DEFINER 트리거(owner=postgres·RLS/GRANT 우회) — 재정의 함수도 `set search_path=public` 유지. ⚠️ `create or replace function insert_fee_item`은 서명 동일이라 기존 EXECUTE 회수(5.10 patch2)가 보존되지만, 마이그 자기완결성을 위해 `revoke all on function ... from public, anon, authenticated`를 **명시 재선언**(authenticated 직접 호출 수가 위조 회귀 차단 — 5.10 코드리뷰 핵심 결함).
