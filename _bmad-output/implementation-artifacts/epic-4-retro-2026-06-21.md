## 2. 잘된 점 (강점)

- **Epic 3 만성 🔴 청산 → Epic 4에서 성숙** — 3에픽째 미해결이던 "감사 PII/건강민감 평문 적재"를 전용 스토리(3-6)로 청산했고, Epic 4가 **확장·정밀화**: 4.6이 SOAP 4컬럼(`subjective/objective/assessment/plan`)을 `_SENSITIVE_KEY`(서버) + 웹 거울에 등록, 4.7은 **올바르게 무변경 판단**(진단은 `diagnosis_id` FK만 유입 → 마스킹 불요). 교차절단 부채 규율이 복리로 작동.
- **불변식 = DB 소유 일관 유지** — 상태머신 전이 트리거(0010 `PT409`)·호출 비-전이(0011)·완료 게이트(0014 `PT422`)·부분 unique(주상병 ≤1) 전부 Postgres 강제. FastAPI는 오케스트레이션·권한 1차선만. 7스토리 내내 흔들림 없음.
- **공유 인프라 레버리지** — 4.2가 만든 `_map_pg_sqlstate`(SQLSTATE→HTTP)를 4.4(start_consult)·4.7(complete + PT422 1줄)이 재사용. 4.4 세션 가드(`active-session`)를 4.6 autosave가 소비. 4.5 reveal RPC 패턴이 후속 소비. 핸드오프 청결.
- **현행 코드 우선 규율** — web이 아키텍처 문서(TanStack/shadcn/Zustand)와 달리 Base UI + 바닐라 훅(`useState/useEffect` + `apiFetch` + `sonner`)을 쓴다는 사실을 매 스토리가 명시·준수. 산출물 문구보다 실제 코드.
- **회귀 0 규율** — 매 스토리 done 시점에 API/Web 전수 green. 권한 변경(doctor grant)이 기존 테스트를 깨면 즉시 baseline 이관으로 청산.

## 9. 핵심 교훈 (Key Takeaways)

1. **교차절단 부채는 전용 주인 스토리로만 청산되고, 청산 후엔 후속 에픽에서 성숙한다** — 감사 마스킹(3-6 → 4.6 확장 → 4.7 정밀 무변경).
2. **불변식은 DB가 최종선** — 상태머신·게이트·unique를 Postgres에 두면 동시성 하드닝을 앱에서 재구현하지 않아도 일관 방어. 재수행 차단(Epic 5)도 같은 길.
3. **공유 계약(RPC·SQLSTATE·권한) 변경은 의존 테스트·boot grant·baseline을 쓸어담는다** — 4.7 `complete_encounter` 게이트 = 3 DB 테스트 청산.
4. **계획 문서 < 실제 코드/glossary** — 마이그 번호·웹 스택은 glossary/현행 코드가 단일 진실. 병렬 워크트리(Epic 6)로 번호 블록 조율이 새 제약.
