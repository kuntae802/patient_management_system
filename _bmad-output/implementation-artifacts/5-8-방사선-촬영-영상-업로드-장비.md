# Story 5.8: 방사선 촬영 · 영상 업로드 · 장비

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a **방사선사(radiologist)**,
I want **촬영 워크리스트에서 지시된 영상검사를 촬영 수행하고, 촬영한 영상을 Supabase Storage에 업로드하며, 촬영 배정에 쓸 장비 목록·상태를 보기를**,
so that **영상이 안전하게 저장(파일=Storage, DB=경로만)되고 검사 오더가 '수행'으로 넘어가 판독(5.9)으로 이어진다.**

> **에픽 범위 노트:** 5.8은 검사 생명주기 `지시(ordered) → 수행(performed)` 구간 중 **방사선사 측 '수행'(촬영+영상 업로드+장비 배정)**만 소유한다. **판독(`performed → completed`, 의사 겸임)은 Story 5.9**가 소유한다 — `complete_examination` RPC·판독 소견 컬럼·판독 화면은 5.8에서 만들지 말 것. **수가 자동발생(수행 완료 → 영상 수가 적재)은 Story 5.10**(트리거 가동). 5.8은 영상 수가를 직접 적재하지 않는다.

## Acceptance Criteria

> 출처: `epics.md#Story-5.8`(FR-100·FR-101·FR-103) + 본 스토리 확정 결정 3건(영상 1:N 테이블 · 장비 표시+배정만 · 수행 시 영상 ≥1 필수).

**AC1 — 촬영 워크리스트 (FR-100)**
- **Given** 방사선사로 로그인한 상태에서
- **When** 촬영 워크리스트(`GET /radiology/worklist`)를 조회하면
- **Then** **오늘(KST) 활성 내원의 영상검사 오더 중 아직 미수행(`exam_type='imaging' AND status='ordered'`)인 항목**이 지시 시각 오름차순(FIFO)으로 표시된다. 각 행은 `chart_no·환자명·진료과명·검사 행위명(fee_name)·지시 의사명·지시 시각·업로드된 영상 수(image_count)`를 포함하고, **raw 주민번호 등 PII는 포함하지 않는다**(비-PII 프로젝션).

**AC2 — 영상 업로드 → Storage(경로만 DB) (FR-101)**
- **Given** 영상검사 오더(`status='ordered'`, `exam_type='imaging'`)에 대해
- **When** 영상 파일(들)을 업로드하면(`POST /examinations/{examination_id}/images`, multipart)
- **Then** 파일은 **비공개 Supabase Storage 버킷 `examination-images`에 저장**되고, **DB(`examination_images`)에는 객체 경로(`storage_path`)만** 1:N으로 연결된다(파일 바이트·공개 URL·서명 URL은 DB에 저장하지 않음). 객체 경로/파일명에 **PII를 넣지 않는다**(`{examination_id}/{uuid4}.{ext}`). 허용 MIME(`image/png·jpeg·webp`)·용량(≤50MiB) 위반 → 422.
- **And** 업로드는 `examination.perform` 권한이 필요하다(미보유 → 403). lab 오더에 업로드 시도 → 422, 이미 수행/완료된 오더에 업로드 → 409.

**AC3 — 장비 목록·상태 표시 + 촬영 배정 (FR-103)**
- **Given** 촬영 배정 시
- **When** 장비 목록(`GET /equipment`)을 보면
- **Then** **활성 장비의 `code·name·modality·status(available/in_use/maintenance)`가 표시**된다(읽기 전용 — 5.8은 장비 상태를 변경하지 않는다).
- **And** 촬영 수행 시 선택한 장비를 해당 검사에 배정할 수 있다(`examinations.equipment_id` 세팅, 선택). 비활성/미존재 장비 배정 → 422.

**AC4 — 촬영 수행(ordered→performed) · 영상 ≥1 필수 · 재수행 차단 (FR-101·FR-093)**
- **Given** 영상이 1장 이상 업로드된 영상검사 오더에 대해
- **When** 촬영 수행(`POST /examinations/{examination_id}/perform`, body `{equipment_id?}`)을 호출하면
- **Then** `perform_examination` RPC로 `status: ordered → performed` 전이되고 `performed_by`(= 호출 방사선사 `auth.uid()`)·`performed_at`이 기록되며, 전달된 `equipment_id`가 배정된다. **활성 영상이 0장이면 전이를 거부(422 `image_required`)**한다.
- **And** **이미 수행(performed)/완료(completed)된 오더를 재수행하면 409 `invalid_transition`**(DB 상태머신 최종선, FR-093). 미존재 → 404, 권한 미보유 → 403.

**AC5 — 영상 조회 = 서명 URL(읽을 때 재생성)**
- **Given** 업로드된 영상이 있는 검사에 대해
- **When** 영상 목록(`GET /examinations/{examination_id}/images`)을 조회하면
- **Then** 각 영상의 메타데이터와 함께 **서버가 발급한 단기 서명 URL(`signed_url`, TTL ≤5분)**이 반환된다. 서명 URL은 **DB에 저장하지 않고 요청 시점마다 재생성**한다. 이 엔드포인트는 `order.read` 게이트(의사·간호·방사선) — **5.9 판독의(의사, `examination.perform` 미보유)도 영상을 볼 수 있어야** 하므로 `examination.perform`이 아닌 `order.read`로 게이트한다.

**AC6 — 회귀·게이트·웹**
- 기존 검사·영상 **오더 생성/조회(5.3)·다른 오더 도메인(처방 5.2·처치 5.4)·간호 워크리스트(5.7)** 회귀 0.
- 신규 web 라우트 `/radiology/worklist`·`/radiology/upload`·`/radiology/equipment`(nav 기선언)가 **방사선사 역할에만 노출**되고 서버 가드(`requirePermission`)로 직접 진입을 차단한다.
- 비-방사선사(원무 등) API 직접 호출 403, 검증 실패 422, 잘못된 전이 409, 미존재 404 — 표준 에러 봉투.
