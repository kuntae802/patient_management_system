# Story 4.4: 진료 대기열 · 진찰 시작

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a **의사(doctor)**,
I want **접수 완료(registered)된 환자의 진료 대기열에서 "진료 시작"을 눌러 진찰을 시작하고, 그 환자의 진료 허브로 진입하기를**,
so that **내원이 '진행중(in_progress)'으로 전이되고 담당의가 나로 기록되며, 한 화면(진료 허브)에서 진료를 이어갈 수 있다. 또한 한 세션에서 동시에 열린 진료 컨텍스트가 1개로 가드되어 잘못된 환자에 작업이 새는 것을 막는다.**

## Acceptance Criteria

1. **AC1 — 진찰 시작 전이 (FR-030, NFR-040, UX-DR8):** 의사가 진료 대기열(대기 현황판, `role="doctor"`)에서 **접수(`registered`) 상태 환자 행의 `진료 시작` 버튼**(button-ghost `.key` — 상태머신이 허용하는 다음-액션만)을 누르면, `POST /api/v1/encounters/{id}/start-consult` 가 **`start_consult` RPC**(0010)를 호출해 내원이 **`registered → in_progress` 로 전이**되고 **`consult_started_at`·`doctor_id = 호출자(auth.uid())`** 가 기록된다(FR-030). **mutation 진행 중 버튼 disable**(이중 제출 차단). 잘못된 상태(미접수/종결/이미 진행중) 진찰 시작은 **409** `invalid_transition`(RPC `status<>'registered'` precondition → 재호출·진료 탈취 차단, NFR-040), 권한 미보유는 **403**, 미존재 내원은 **404** 다. 전이는 실시간(`postgres_changes`)으로 다른 단말(원무 보드)에도 ≤5초 내 반영된다(4.3 재사용 — 추가 배선 불요).

2. **AC2 — 진료 허브 진입 + 세션당 활성 내원 1개 가드 (UX-DR21⑨):** 진찰 시작 성공 시 그 내원의 **진료 허브 셸**(신규 라우트 `(staff)/encounter/[encounter_id]`)로 네비게이트되어 진료 화면이 열린다. 진료 허브가 열리면 **세션당 활성 내원 컨텍스트가 1개로 가드**된다(UX-DR21⑨ 다중 진료/스테일 탭 가드): 다른 내원이 이미 활성(다른 탭)인 상태에서 또 다른 진료 허브를 열면 **경고 배너**가 뜨고("다른 진료가 이미 열려 있습니다"), 한 진료를 활성화하면 다른 탭은 **보류(superseded)** 로 표시되어 후속 쓰기(4.6 SOAP autosave)가 거부될 토대가 마련된다. ⚠️ **진료 허브 콘텐츠(환자 배너·과거 이력·활력=4.5, SOAP=4.6, 진단=4.7)는 이 스토리 밖** — 4.4 는 셸 진입·식별·세션 가드까지(후속 스토리가 패널을 채운다).

3. **AC3 — doctor 권한 활성화 + 회귀 0 (RBAC, NFR-051):** doctor 역할이 시드에서 **`encounter.read` + `encounter.start`** 를 보유해 `/doctor/waiting`(진료 대기) 접근·진찰 시작 골든 패스가 가동된다(4.3 까지 doctor=권한0 으로 강등됐던 라우트 활성화). 'doctor=권한0' baseline 에 의존하던 **기존 테스트는 새 무권한 baseline 계정(`nurse@pms.local`)으로 이관**되고, doctor 의 진찰 시작 성공·권한 거부(nurse)·잘못된 상태·미존재 경로가 신규 테스트로 커버되어 **회귀 0** 을 보장한다.
