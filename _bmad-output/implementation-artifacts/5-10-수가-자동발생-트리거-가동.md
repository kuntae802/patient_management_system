# Story 5.10: 수가 자동발생 트리거 가동 (DB)

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 백엔드 개발자,
I want 임상 이벤트(진찰 시작·검사/영상 수행·처치 수행)가 대응 수가 항목을 내원에 **원자적·멱등적으로 자동 적재**하도록 DB 트리거를 가동하고, 행위→수가코드 매핑을 시드로 외부화하기를,
so that 수납이 사람의 수가코드 암기가 아니라 시스템 규칙으로 채워지고, Epic 7 수납이 이 적재된 `fee_items`를 헤더·집계·문서로 소비할 수 있다.

## Acceptance Criteria

> ⚠️ **경계 결정(사용자 확정 2026-06-22)** — Dev Notes §경계 결정 참조. 본 스토리는 **순수 DB 스토리**(5.1 `0010`/`0015` 동형): 메커니즘·`fee_items` 적재·`fee_mappings` 진찰 시드까지. payments 헤더·집계·finalize·문서·급여/본인부담 산정 = Epic 7.

**AC1 — `fee_items` 테이블(내원별 수가항목 누적 + 금액 스냅샷):**
`0021_billing.sql`에서 `fee_items` 테이블이 생성된다 — 내원(`encounter_id` FK)에 발생한 수가 항목을 누적한다. 핵심 컬럼:
- `fee_schedule_id` FK(NOT NULL → `fee_schedules`): 적재된 EDI 수가.
- `source_type` text CHECK in (`encounter`, `examination`, `treatment`): 발생원 유형(약제비 `prescription`은 **본 스토리 미포함** — Dev Notes §경계 참조).
- `source_id` uuid(NOT NULL): 발생원 레코드 id(역추적 + 멱등 키).
- `quantity` integer(default 1, `> 0`), `unit_amount_krw` integer(NOT NULL, `>= 0` — **`fee_schedules.amount_krw` 시점 스냅샷**), `amount_krw` integer(NOT NULL, `>= 0` — `quantity * unit_amount_krw` 총액 스냅샷).
- `category` text·`coverage_type` text CHECK in (`covered`,`non_covered`): 적재 시점 분류 스냅샷.
- 멱등 불변식: **`unique (source_type, source_id)`** — 같은 임상 이벤트가 재발화해도 수가 항목이 1회만 적재된다.

**AC2 — `fee_mappings` 테이블(행위→수가코드 규칙 외부화):**
`fee_mappings` 테이블이 생성되어 **코드 수정 없이 시드로 매핑을 관리**한다(FR-116 외부화). `source_event` text CHECK in (`encounter_start`) + `fee_schedule_id` FK + `is_active`. 본 스토리 범위에서 매핑이 **비-항등(non-identity)인 유일한 이벤트는 진찰**이다(내원에는 자체 `fee_schedule_id`가 없음 → 매핑으로 진찰료 코드 결정). 검사·처치는 오더 행 자체가 `fee_schedule_id`를 보유 → **매핑 항등(직접 사용)**이므로 `fee_mappings` 미경유. CHECK는 `encounter_start` 1값만 — Epic 7이 가산·정액제 등 신규 `source_event`를 확장 추가한다.

**AC3 — 진찰료 자동 적재(`registered → in_progress`, FR-116 진찰료=진찰 시):**
내원이 진찰 시작(`start_consult` RPC, status `registered → in_progress`)으로 전이하면, `fee_mappings`의 활성 `encounter_start` 규칙이 가리키는 진찰료가 `fee_items`에 1행 적재된다(`source_type='encounter'`, `source_id=encounter.id`, `quantity=1`, 금액·category·coverage_type = 해당 `fee_schedules` 스냅샷). 트리거는 **`AFTER UPDATE OF status`** + `WHEN (old.status='registered' AND new.status='in_progress')`로 상태 전이만 정확히 1회 포착한다(walk-in INSERT·비-상태 컬럼 same-status UPDATE는 미발화). 활성 매핑이 없으면 적재 없이 통과(no-op, 예외 아님).

**AC4 — 검사·영상 수가 자동 적재(`ordered → performed`, FR-116 수행 시):**
검사·영상 오더가 수행 전이(`perform_examination` RPC, status `ordered → performed`)하면 `examinations.fee_schedule_id`가 직접 `fee_items`에 적재된다(`source_type='examination'`, `source_id=examination.id`). 트리거 = `AFTER UPDATE OF status` + `WHEN (old.status='ordered' AND new.status='performed')`. **`performed → completed`(판독 완료) 전이는 추가 적재하지 않는다**(수가는 수행 시점 1회 — Dev Notes §경계).

**AC5 — 처치 수가 자동 적재(`ordered → performed`, FR-116 수행 시):**
처치 오더가 수행 전이(`perform_treatment_order` RPC, status `ordered → performed`)하면 `treatment_orders.fee_schedule_id`가 직접 `fee_items`에 적재된다(`source_type='treatment'`, `source_id=treatment_order.id`). 트리거 = `AFTER UPDATE OF status` + `WHEN (old.status='ordered' AND new.status='performed')`.

**AC6 — 원자성·멱등성·금액 스냅샷 불변(NFR-040 트랜잭션 원자성):**
- **원자성:** 적재 트리거가 전이와 동일 트랜잭션에서 실행된다(`AFTER UPDATE` row 트리거) — 전이 성공 = 적재 성공.
- **멱등성:** 적재 INSERT는 `on conflict (source_type, source_id) do nothing` — 동일 이벤트 재발화 시 중복 적재 0(예: 트리거 재실행·동시성 재시도).
- **금액 스냅샷 불변:** 적재 후 `fee_schedules.amount_krw`(또는 coverage_type) 마스터가 변경되어도 기존 `fee_items` 행의 금액·분류는 보존된다(청구 시점 고정 — Epic 7 정산 정합).

**AC7 — 약제비 미적재(약가 데이터 부재 — 정직한 경계):**
처방 발행(`prescriptions` INSERT, status `issued`) 시점에는 `fee_items`에 **아무 행도 적재되지 않는다** — `drugs` 마스터에 약가(단가) 컬럼이 없어 금액을 산정할 수 없기 때문. 약제비 자동발생(FR-116 약제비=처방 발행 시)은 약가 모델링과 함께 Epic 7로 이월된다. 본 스토리는 이 한계를 테스트로 명문화한다(처방 발행 → fee_item 0).

**AC8 — 권한 카탈로그 + GRANT + RLS(0015 posture 계승):**
신규 권한 `fee_item.read`(수가항목 조회 — 원무·임상 조회용, Epic 7 수납이 소비)가 시드되고 **admin 부트 grant가 재실행**된다(⚠️ 0002 cross-join은 후행 마이그 권한 미포함 → 누락 시 `test_admin_role_has_all_permissions` 회귀 — 0010·0012·0013·0014·0015가 겪은 함정). RLS:
- `fee_items`: 직원 = `fee_item.read` 보유 시 전체 행 / 환자 = 본인 내원의 항목만(encounter→patient→auth_uid 경유, Epic 8 포털). **쓰기 정책 없음**(적재는 SECURITY DEFINER 트리거가 RLS 우회).
- `fee_mappings`: 전역 참조 마스터(authenticated SELECT 전체 — fee_schedules 미러). 쓰기는 service_role/시드.
- GRANT: 두 테이블 `revoke all from anon, authenticated` → `grant select to authenticated`(RLS 행 게이트) + `grant select,insert,update,delete to service_role`.

**AC9 — 감사 기록(0004 append-only):**
`fee_items`·`fee_mappings` 두 테이블에 `audit_trigger_fn`(0004) 트리거가 부착되어 적재(INSERT)·변경이 actor와 함께 `audit_logs`에 append-only 기록된다. 스냅샷 = FK·숫자·금액·짧은 구조화 텍스트(category·coverage_type·source_type)만 = **자유 임상 서사 없음** → `_SENSITIVE_KEY` 마스킹 집합 무변경(5.1 §AC6·encounter_diagnoses 동형 posture). ⚠️ 트리거 적재의 actor는 `start_consult`/`perform_*` RPC를 호출한 직원(`auth.uid()`) — SECURITY DEFINER 트리거가 동일 세션 컨텍스트를 계승하는지 확인.

**AC10 — `fee_mappings` 진찰료 시드(진찰 매핑만 — 청구 단순화 선은 Epic 7):**
`seed.sql`이 `fee_mappings`에 활성 `encounter_start` 규칙 1행(진료과 무관 단일 진찰료)을 시드한다. **초진/재진 동적 판정·진료과별 가산은 미구현**(아키텍처 §445 "초진/재진·가산 = 수납 에픽" 존중) — 기본 매핑은 단일 진찰료 코드(예 재진진찰료 `AA254`). seed.sql:6 "fee_mappings = Epic 7 범위 밖" 주석을 "메커니즘+진찰 매핑 = 5.10 / 청구 단순화 선(초진재진·가산)·약제비·집계 = Epic 7"로 정정한다.

### ⚠️ `fee_mappings`의 비대칭 — 진찰만 매핑이 필요한 이유

핵심 인사이트(dev가 반드시 이해해야 중복 설계 회피):
- **검사·처치**: `examinations.fee_schedule_id`·`treatment_orders.fee_schedule_id`가 **이미 오더 행에 직접 존재**(0015·NOT NULL, master picker `kind=fee_schedule`로 입력). 행위=수가코드가 항등이므로 트리거가 `NEW.fee_schedule_id`를 직접 `fee_items`에 복사 → `fee_mappings` 미경유.
- **진찰**: `encounters`에는 `fee_schedule_id`가 **없다**(내원은 진료과·의사·상태만 보유). "진찰 시작 → 어느 진찰료 코드?"는 규칙 결정이 필요 → `fee_mappings`의 `encounter_start` 규칙이 비-항등 매핑. 이것이 `fee_mappings`가 진짜로 필요한 유일한 경로.
- 따라서 `fee_mappings`는 본 스토리에서 `encounter_start` 1종만 담는다. 빈약해 보이지만 정직한 경계 — Epic 7이 가산·정액제·진료과 규칙으로 확장. CHECK를 `in ('encounter_start')` 단일값으로 둬 스코프를 코드로 잠근다.

### ⚠️ 트리거 정확성 — 중복 적재 방지(가장 중요)

진찰료/수가 **중복 적재가 본 스토리 최대 리스크**. 3중 방어:
1. **`AFTER UPDATE OF status` + `WHEN` 절**: 상태 컬럼이 실제 전이될 때만 발화. `AFTER INSERT`(walk-in 초기 registered)·same-status UPDATE(0010/0015의 비-상태 컬럼 갱신, 예 room_id·equipment_id)·비-status 컬럼 UPDATE는 미발화.
   - 검사 0015 주석: same-status UPDATE(equipment_id 5.8 배정 등) 통과 허용 → `OF status` 한정이 이를 정확히 배제.
2. **`unique (source_type, source_id)` + `on conflict do nothing`**: 트리거가 이론적으로 재발화해도(동시성·재시도) 멱등.
3. **RPC 소스상태 precondition**(0010/0015 기존): `start_consult`는 `status<>'registered'` 시 PT409 → 한 내원에 진찰 전이 1회만. `perform_*`도 `status<>'ordered'` 시 PT409 → 수행 전이 1회만. 즉 정상 경로에서 전이 자체가 1회 보장.

검증: AC6 테스트가 "재발화/재호출 시 1행 유지"를 명시 단언.

### 적재 경로 — SECURITY DEFINER 트리거가 RLS/GRANT 우회

`fee_items`는 쓰기 RLS 정책이 없다(GRANT도 authenticated SELECT만). 적재는 **SECURITY DEFINER 트리거 함수**(owner=postgres)가 수행 → RLS·GRANT 우회(0004 `audit_trigger_fn` 동일 패턴). 트리거가 `start_consult`/`perform_*`(SECURITY DEFINER RPC) 트랜잭션 내부에서 발화하므로 actor 컨텍스트(`auth.uid()`)는 RPC 호출자로 유지 → 감사 actor 정확. ⚠️ `insert_fee_item` 헬퍼·트리거 함수 전부 `set search_path = public`(0010/0015 SECURITY DEFINER 규칙).

### 회귀 리스크 — 기존 테스트에 적재 부작용 주입

본 스토리 트리거는 기존 RPC(`start_consult`·`perform_examination`·`perform_treatment_order`)에 **부작용을 추가**한다. 이 RPC를 호출하는 기존 테스트(`test_encounters_db`·`test_encounters_integration`·`test_orders_db`·`test_orders_integration`·`test_nursing_integration`·`test_radiology_integration`)가 이제 fee_items 행을 생성한다. begin/rollback 격리 테스트는 영향 없으나, **행 카운트·전역 단언이 깨지지 않는지** 확인(Task 5). 검사/처치는 0015에서 `fee_schedule_id` NOT NULL이라 적재가 항상 성공(NULL FK로 인한 트리거 실패 없음).
