# Story 8.5: 운영 대시보드 · 통계 (관리자)

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 관리자/원장,
I want 일별 내원·대기·매출(수납)·노쇼율을 한 화면(운영 대시보드)에서 보기를,
so that 직접 워크리스트를 뒤지지 않고 운영 현황을 한눈에 파악하고 의사결정한다.

## Acceptance Criteria

**AC1 — 운영 대시보드 진입 · 권한 게이트 (이중)**
**Given** `dashboard.read` 권한을 가진 관리자 세션에서
**When** 네비 "운영/대시보드"(`/admin/dashboard`)를 열면
**Then** 당일 운영 KPI(내원 수·대기·진료중·완료·순수납액·노쇼율)와 최근 N일 추세가 표시된다(FR-230).
**And** `dashboard.read` 미보유 직원은 메뉴가 노출되지 않고(`filterNav`), 직접 URL 접근 시 서버 가드(`requirePermission`)가 `/home`으로 강등한다(UI 노출 게이트 + 서버 가드 이중).

**AC2 — 복잡 집계는 FastAPI, 권한 범위 내 데이터 (RBAC)**
**Given** "복잡 집계는 FastAPI가 담당" 원칙에서
**When** 웹이 `GET /v1/dashboard/operations`를 호출하면
**Then** 일별 내원·순수납액·노쇼율이 서버에서 집계돼 단일 리소스로 반환되고, 라우터는 `require_permission("dashboard.read")`로 게이트된다.
**And** 미인증=401, 무권한=403(`{error:{code:"forbidden", detail:{required_permission:"dashboard.read"}}}`), 집계는 clinic-wide(관리자 운영 범위, service_role 경로라 RLS 우회 — 권위는 라우터 게이트)로 수행된다.

**AC3 — 메트릭 정의 정확성 (KST · refunded 차감 · divide-by-zero)**
**Given** 모든 `timestamptz`는 UTC 저장이라는 사실에서
**When** 일별 집계를 산정하면
**Then** 모든 일자 그룹핑은 `(col at time zone 'Asia/Seoul')::date` 기준이고(밤 23:30 UTC 건이 익일 KST로 정확히 귀속), 순수납액 = `Σ(paid_amount_krw − coalesce(refunded_amount_krw,0))`(환급 차감 — **Story 8.5가 `refunded_amount_krw`의 첫 리포팅 소비처**), 노쇼율은 분모가 0이면 `0`으로 반환(나눗셈 오류·`NaN` 없음).

**AC4 — 빈 데이터 · read-only · 디자인 언어 일관**
**Given** 데이터가 없는 날짜/시드 미적용 환경에서
**When** 대시보드를 열면
**Then** 모든 KPI가 `0`/`—`로 안전하게 렌더되고(빈 화면·크래시 없음), 대시보드는 **read-only**(쓰기/상태전이 액션 0)이며, 확정 디자인 언어(임상 틸 토큰·**색/음영 비의존 = 값·라벨 병기**·Pretendard·Lucide)·KRW 정수 포맷·KST 날짜·24h가 일관 적용된다(UX 대시보드=spine-only, 대기판/진료 허브 패턴 상속).

### 데이터 소스 & 집계 공식 (KST 필수 · 정확한 컬럼)

> 모든 일자 그룹은 `(col at time zone 'Asia/Seoul')::date`. `timestamptz`는 UTC 저장(KST=UTC+9). 기존 선례: `db.py` billing worklist `(e.created_at at time zone 'Asia/Seoul')::date`, `demo_seed.sql:43` `v_today := (now() at time zone 'Asia/Seoul')::date`.

**1) 내원(encounters) — `0010_encounters.sql`**
- status CHECK: `scheduled|registered|in_progress|completed|cancelled|no_show`. 시각 컬럼: `registered_at`(접수), `consult_started_at`(진찰시작), `completed_at`(완료), `no_show_at`, `cancelled_at`. `is_active boolean`.
- **당일 내원 수(도착)** = `count(*) where is_active and status in ('registered','in_progress','completed') and (registered_at at time zone 'Asia/Seoul')::date = :date`. (registered_at 존재 = 실제 내원. scheduled/취소/노쇼는 미내원 → 제외.)
- **대기(현재)** = `count(*) where is_active and status='registered'`. **진료중(현재)** = `status='in_progress'`. (순간 스냅샷 — `date=오늘`일 때만 의미; 기본 date=오늘.)
- **당일 완료** = `count(*) where status='completed' and (completed_at at time zone 'Asia/Seoul')::date = :date`.
- **일별 내원 시리즈** = `registered_at` KST 일자별 group by(같은 status 필터).

**2) 매출/수납(payments) — `0045_payments.sql` + 환급 `0052_payment_cancel.sql`**
- status CHECK: `draft|finalized|cancelled`. 금액(전부 KRW 정수): `total_amount_krw`, `covered_amount_krw`, `non_covered_amount_krw`, `copay_amount_krw`, `insurer_amount_krw`, `paid_amount_krw`, **`refunded_amount_krw`**(0052, 제약 `refunded ≤ paid`). `finalized_at timestamptz`.
- **순수납액(매출)** = `Σ(paid_amount_krw − coalesce(refunded_amount_krw,0)) where status='finalized' and (finalized_at at time zone 'Asia/Seoul')::date = :date`.
  - ⚠️ **Story 8.5가 `refunded_amount_krw`의 첫 리포팅 소비처**(메모리 "8.5=refunded 첫소비"). 환급은 원 finalize와 다른 날 발생할 수 있으나, 데모 단순화상 **원 payment 행의 finalize 일자에 순액 귀속**(refunded는 동일 행에 누적). 이 단순화를 dev note/코드 주석에 명시.
  - (보조 가능: `total_amount_krw` 청구총액, `insurer_amount_krw` 공단부담 분해 — 1차 KPI는 환자 순수납액. 필요 시만 추가.)
- **일별 수납 시리즈** = `finalized_at` KST 일자별 group by.

**3) 노쇼율(appointments) — `0031_appointments.sql` + `0033_appointment_lifecycle.sql`**
- status CHECK: `booked|cancelled|no_show|completed`(**`is_active` 컬럼 없음** — 취소=status). 시각: `scheduled_start`/`scheduled_end`(timestamptz UTC), `no_show_at`, `completed_at`, `cancelled_at`(0033).
- **노쇼는 appointments.status='no_show'가 단일 진실**(0036 `patient_no_show_count`도 동일 소스 — encounters.no_show 아님).
- **당일 노쇼율** = `no_show_count / nullif(no_show_count + completed_count, 0)`, 여기서 분자·분모 모두 `(scheduled_start at time zone 'Asia/Seoul')::date = :date`. cancelled(능동 취소)는 분모 제외, booked(미도래)는 과거일엔 없음·당일엔 아직 미발생 → 제외. 분모 0 → **0 반환(NaN 금지)**.
- **일별 노쇼율 시리즈** = `scheduled_start` KST 일자별 group by.

### API 설계 (단일 리소스 응답)

```
GET /v1/dashboard/operations?date=YYYY-MM-DD&days=14   # require_permission("dashboard.read")
→ 200 DashboardOperationsResponse {
    as_of_date: "2026-06-24",          # 서버가 결정한 KST 일자(클라 신뢰 X)
    today: {
      visits: 0, waiting: 0, in_progress: 0, completed: 0,
      revenue_net_krw: 0,              # KRW 정수, paid − refunded
      no_show_rate: 0.0,               # 0~1 (분모0 → 0)
      no_show_count: 0, appointment_total: 0
    },
    daily_series: [                    # 오래된→최신, 추세 차트용
      { date: "2026-06-11", visits: 0, revenue_net_krw: 0, no_show_rate: 0.0, no_show_count: 0 },
      ...
    ]
  }
```

- **봉투**: 대시보드 요약은 목록이 아니므로 단일 리소스 객체(페이지네이션 불요). `{data,meta}` 래퍼 쓰지 말 것.
- **에러 표준**: 401(미인증)/403(무권한, `detail.required_permission="dashboard.read"`)/422(date 파싱)/500. 봉투 `{error:{code,message,detail}}`.
- **집계 위치 = FastAPI 서비스 계층 SQL**(AC2). DB 함수(RPC)로 빼지 말 것 — 본 스토리는 "복잡 집계는 FastAPI가 담당" 명세를 실증하는 스토리다. 단순 조회였다면 Supabase 직접일 수 있으나, 다중 테이블·일자 그룹·파생 비율이라 FastAPI 집계가 정답.
- **service_role 경로 = RLS 우회**(db.py:3-7). clinic-wide 전체 읽기가 **의도된 관리자 범위**. 권위는 라우터 `require_permission("dashboard.read")`(billing worklist=`payment.read`, staff directory=`patient.read` 동일 방어심층 패턴). 읽기라 동일-txn 권한 재평가 불요.
