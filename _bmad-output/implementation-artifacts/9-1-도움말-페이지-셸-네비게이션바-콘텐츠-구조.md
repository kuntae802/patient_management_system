# Story 9.1: 도움말 페이지 셸 · 네비게이션바 · 콘텐츠 구조

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 로그인한 직원 사용자,
I want 내 화면에서 도움말 페이지에 들어가 내 계정에 보이는 메뉴 인덱스를 보고, 메뉴를 고르면 그 안내로 바로 이동하기를,
so that 선배 없이도 내가 쓰는 기능을 한 곳에서 찾아 익힐 수 있다.

## Acceptance Criteria

1. **(FR-250) 인증된 진입.** 직원 셸의 사이드바 푸터에 "도움말" 항목이 보이고, 클릭하면 인증된 도움말 페이지(`/help`)로 이동한다. 미인증 접근은 로그인으로 리다이렉트된다(공개 아님). 비직원(환자)·로그아웃 상태에서는 직원 도움말에 접근할 수 없다.

2. **(FR-251) 현재 계정 메뉴만 인덱스.** 도움말 페이지 상단에 sticky 네비게이션바가 렌더되고, 이 인덱스에는 **현재 로그인 계정에 실제로 표시되는 메뉴만**(= `filterNav(STAFF_NAV, role, has)` 결과) 나열된다. 역할에 없거나 권한으로 가려진 메뉴(예: 권한 없는 admin 메뉴)는 인덱스에도 본문에도 노출되지 않는다. 네비바 그룹핑은 사이드바와 동일한 섹션 순서를 따른다.

3. **앵커 이동.** 네비게이션바에서 특정 메뉴를 선택하면 본문의 해당 메뉴 안내 섹션으로 앵커 스크롤된다. 각 섹션은 안정적인 `id`를 가지며, sticky 네비바에 가리지 않도록 스크롤 오프셋이 보정된다. 현재 보고 있는 섹션이 네비바에서 활성 표시되면 가산점(필수 아님).

4. **(FR-253) 콘텐츠 데이터 구조 + 동적 렌더.** 도움말 콘텐츠는 "메뉴 키(href) → 화면 목록 → (이미지·번호 하이라이트·단계 설명)" 형태의 **데이터 구조**로 정의되고(`web/src/lib/help/`), 페이지는 그 데이터를 역할별 전체 하드코딩 없이 `filterNav` 결과로 필터해 현재 계정 메뉴 섹션만 렌더한다. 콘텐츠가 아직 채워지지 않은 메뉴(9.3~9.8에서 채움)는 "준비 중" 플레이스홀더로 섹션 자리를 유지한다. 페이지 골격(헤더·컨테이너·sticky 네비·섹션·이미지 슬롯·번호표)은 후속 스토리가 데이터만 채우면 되도록 완성한다.

5. **회귀 무파괴.** 기존 사이드바·라우팅·인증 흐름이 그대로 동작한다. `STAFF_FOOTER_NAV` 변경은 사이드바 푸터에 도움말 1개만 추가될 뿐 기존 메뉴 노출·active 판정에 영향이 없다. `npm run build`/lint/타입체크 통과, 생성 TS 타입 무수정.

> **범위 경계 (반드시 지킬 것):** 이 스토리는 **직원(staff) 도움말의 셸·네비바·콘텐츠 구조**까지다.
> - 실제 스크린샷 캡처 파이프라인·더미데이터 = **Story 9.2**.
> - 역할별 실제 콘텐츠(원무/의사/간호/방사선사/관리자) 채우기 = **Story 9.3~9.7**.
> - 환자 포털 도움말(별도 `(patient)` 셸·탭바 구조) = **Story 9.8**. 이 스토리에서 환자 셸은 건드리지 않는다.
> 9.1은 "골격 + 데이터 모델 + 1개 시범 콘텐츠(의사, 기존 9-2 샘플 이식)"까지만 한다. 나머지 메뉴는 플레이스홀더.

## Dev Notes

### 컨텍스트 요약 — 무엇을 만드나
9.1은 **직원 도움말의 골격**이다. (1) 사이드바 푸터에 도움말 진입점, (2) `/help` 페이지(현재 계정 메뉴만 sticky 인덱스 + 앵커 본문), (3) 후속 스토리가 "데이터만" 채우면 되는 콘텐츠 데이터 모델. 실제 캡처(9.2)·역할별 콘텐츠(9.3~9.7)·환자 포털(9.8)은 범위 밖. 의사 1개만 기존 9-2 샘플로 시범 이식한다.

### ⚠️ 비자명·실수 방지 (이 스토리에서 LLM이 틀리기 쉬운 것)
1. **Next.js 16 = 네가 아는 Next.js가 아니다.** `web/AGENTS.md`가 명시: breaking changes 있음 → `web/node_modules/next/dist/docs/`를 읽고 작성. 증거: 미들웨어가 이미 `web/src/proxy.ts`(`updateSession`)로 이관됨. `middleware.ts` 새로 만들지 말 것.
2. **인증 가드 재구현 금지.** `(staff)/help/page.tsx`는 `(staff)/layout.tsx`의 `requireStaff()`를 자동 상속한다. 미인증→`/login`은 `proxy.ts`가, 비직원→`/portal`은 layout이 이미 처리. page에서 또 가드를 넣으면 중복·충돌.
3. **메뉴 필터는 `filterNav` 그대로 재사용.** 역할별 메뉴를 도움말에서 다시 하드코딩하면 FR-251 위반이자 사이드바와의 이중 출처. `filterNav(STAFF_NAV, role, has)` 결과 = 단일 진실. 콘텐츠 가이드도 `href`를 키로 매핑해 같은 출처에 정렬.
4. **JSON/식별자 snake_case 규칙은 여기 대부분 무관**(이 스토리는 순수 프론트 정적 콘텐츠, DB·API 변경 0). 단 새 식별자(`HelpScreen` 등)는 영문, UI 라벨만 한국어.
5. **번호 하이라이트는 이미지에 구워져 온다.** 캡처 파이프라인(9.2)이 DOM 오버레이로 빨강 원+번호를 주입해 PNG로 굽는다(메모리·9-2 샘플 확인). 그러므로 페이지는 좌표 기반 오버레이를 그릴 필요 없이 **이미지 + 번호표(①·요소·설명)** 병기만 하면 된다. 데이터 모델을 거기에 맞춘다.
6. **sticky 기준점.** `(staff)` 셸은 `AppShell`의 `<main className="flex-1 overflow-auto">`가 스크롤 컨테이너다(Topbar는 그 밖, h-topbar=52px). 도움말 sticky 네비바는 이 main 기준 `top-0`이 자연스럽다 — 페이지를 직접 띄워 sticky가 떠 있고 앵커 점프가 제목을 가리지 않는지 눈으로 확인할 것(추정 금지).

페이지: `const menus = filterNav(STAFF_NAV, role, has)` → 사이드바처럼 섹션 그룹핑 → 각 메뉴 `HELP_GUIDES[menu.href]` 조회(있으면 렌더, 없으면 플레이스홀더). slug 헬퍼는 `href.replace(/^\//,"").replace(/\//g,"-")` 식으로 인덱스 앵커(`#help-<slug>`)와 섹션 `id`를 한 곳에서 생성(불일치 방지).

### 슬러그·중복 키 주의
- `/patients`(환자 검색)는 reception·doctor 둘 다의 `STAFF_NAV` 항목이지만, 한 사용자는 한 역할이라 `filterNav` 결과에는 메뉴가 역할에 맞게 한 번만 들어온다. 같은 href가 결과에 중복될 일은 없다. 그래도 렌더 시 `key`는 `href` 기준으로.
- 도움말 자신(`/help`)은 `STAFF_NAV`에 없고 `STAFF_FOOTER_NAV`에만 있으므로 인덱스(=`STAFF_NAV` 필터)에 자연히 빠진다. 의도된 동작.
