# Story 3.2: 환자 임상 프로필 입력 · 조회

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 원무·의사,
I want 환자의 임상 프로필(혈액형·알레르기·기저질환·복용약·특이사항)을 입력·갱신·조회하기를,
so that 처방·처치 시 안전하게 참조한다.

## Acceptance Criteria

1. **AC1 — 임상 프로필 입력·갱신 (FR-004, 이월 갭 ②):** 환자 레코드가 존재할 때, `patient.update` 권한 보유자가 임상 프로필(`blood_type`·`allergies`·`chronic_diseases`·`medications`·`notes`)을 입력·갱신하면 해당 값이 `patients` 행에 저장된다. **컬럼은 0009에 이미 존재**(전부 nullable text) — 3.2는 입력·갱신 경로(API + 웹 UI)만 추가하고 **신규 마이그레이션을 만들지 않는다**. 갱신은 0009 감사 트리거가 자동 기록한다(actor=GUC sub).
2. **AC2 — 임상 프로필 조회 / 안전 참조 (FR-005):** `patient.read` 보유 직원(원무·의사)이 환자 상세를 조회하면 임상 프로필이 표시되고, **알레르기는 can't-miss(누락 0)** 로 두드러지게 노출된다(`role="alert"`/`aria-live="assertive"`, danger 채움+테두리+굵은 라벨 — 음영 비의존, UX-DR22/DESIGN allergy-alert). **진료 허브 배너 연동은 Epic 4**, **알레르기↔오더 교차검증은 Epic 5** — 3.2는 상세 풀페이지에서의 입력·표시까지.
3. **AC3 — 권한·PII·쓰기 권위 경계:** 입력·갱신은 **FastAPI(service_role) 경로로만** 수행된다(authenticated 는 patients UPDATE 권한 없음 — 0009 GRANT·RLS 쓰기정책 부재). 게이트 = `require_permission("patient.update")` → 403, **실제 쓰기는 `update_patient_clinical_profile` 가 동일 트랜잭션에서 `has_permission('patient.update')` 재평가(TOCTOU 차단) 후 UPDATE**. 응답은 마스킹된 주민번호만(`resident_no_masked`) 포함하고 `resident_no_enc`/`_hash` 는 절대 직렬화하지 않는다(PII 경계 유지).

> **이월·교차절단 인수 조건(이 스토리에서 확인):** ① **신규 마이그레이션 0건** — 임상 컬럼·`grant select(...)`·감사 트리거가 모두 `0009_patients.sql` 에 존재함을 확인만 하고 재선언하지 말 것(중복 정책·이중 감사 위험). ② **A-2 TOCTOU 연속** — `patient.create` 쓰기와 동일하게 `update`도 in-txn 재평가(게이트+재평가 이중). ③ **A-3 감사 PII(긴급도 상승, 계속 defer)** — 임상 UPDATE 가 `audit_logs` 스냅샷에 **건강민감 데이터**(알레르기·기저질환·복용약)를 처음으로 유입시킨다. 현재 방어(`audit.read` 게이트 + 1.10 뷰어 렌더 마스킹)는 유지되나 서버측 감사 PII 마스킹 정책은 여전히 미해결 — **3.2 범위를 넓히지 말고**(교차절단, 전 PII 테이블 영향) Dev Notes §A-3 의 결정대로 deferred-work 에 긴급도 갱신만 한다. 상세는 Dev Notes §이월·교차절단 참조.

### 갱신 엔드포인트 설계 (`update_department` 미러)

- **경로:** `PUT /api/v1/patients/{patient_id}/clinical-profile`(sub-resource action — 상태 PATCH 아님, project-context §FastAPI "상태 전이는 액션 엔드포인트"의 데이터 갱신판). 임상 프로필을 하나의 논리 sub-resource 로 전체 교체(PUT) — 5필드 모두 전송(미전송=None=값없음). PATCH 병합 모호성 회피.
- **쓰기 권위:** authenticated 는 patients UPDATE 권한 없음(0009 GRANT·RLS 쓰기정책 부재) → service_role(FastAPI)만 쓰기. 게이트(`require_permission`)는 방어심층, **진짜 권위는 `_op` 안 `_require_patient_update(conn)` in-txn 재평가**(A-2 연속 — `insert_patient` L1074 와 동형). 평가↔쓰기 사이 권한 변동(TOCTOU) 차단.
- **404:** `update ... where id=$1 returning ...` 가 0행 → db 가 `None` 반환 → 서비스가 `NotFoundError`(`get_patient` 동형). RLS 우회(service_role)라 미존재와 권한밖을 동일 404 로(존재 누설 회피).
- **감사:** 0009 `trg_patients_audit`(after update) 가 before/after 스냅샷 자동 기록(actor=`app.actor_id` GUC=sub). 앱이 `audit_logs` 직접 INSERT 금지.
