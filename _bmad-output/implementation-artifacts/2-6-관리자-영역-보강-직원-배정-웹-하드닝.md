# Story 2.6: 관리자 영역 보강 — 직원 진료과 배정 · 웹 하드닝 · 테스트 격리

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 관리자/개발팀,
I want 직원에게 진료과를 실제 UI로 배정하고(생성·재배정), 마스터 관리 화면의 교차절단 웹 부채(다중행 pending 경합·단일 실패점·피커 무재시도)를 정리하고, 의존성 카운트 테스트가 시드 doctor에 의존하지 않게 격리하기를,
so that Epic 1·2가 쌓아온 관리자 영역의 미완 기능과 누적 부채를 Epic 3(환자·PII) 진입 전에 한 번에 청산하고, 골든 패스의 "진료과 소속 의사"를 시드 우회가 아닌 진짜 기능으로 뒷받침한다.

## Acceptance Criteria

> **출처:** 본 스토리는 epics.md 산하가 아니라 **Epic 2 회고(2026-06-21)의 액션 아이템**이다(epic-2-retro-2026-06-21.md §6). 회고 결정: "Epic 3 착수 전 전용 '관리자 영역 보강' 스토리 신설 — 웹 하드닝 3종 + 직원 진료과 배정 UI + 테스트 격리"(Project Lead 승인). 인수 대상은 전부 `deferred-work.md`에 추적된 이월 항목. **신규 마이그레이션·신규 스키마 컬럼 0**(department_id 컬럼·FK는 0002·0006에 이미 존재) — 순수 API 엔드포인트 + 웹 + 테스트 보강.

**AC1 (직원 진료과 배정 — 생성 시, 1.8 이월):**
**Given** 진료과 마스터가 존재하는 지금(Epic 2 완료)
**When** 관리자가 직원 생성 폼에서 소속 진료과를 선택하면
**Then** `StaffCreate.department_id`(이미 백엔드 수용 — schemas/users.py:47·insert_staff_profile db.py:228-288)에 실려 직원이 그 진료과 소속으로 생성된다. 피커는 **활성 진료과만** 노출하고(room-form 패턴 미러), 비활성 진료과 배정은 API가 422 `inactive_department`로 거부한다(기존 `_assert_department_assignable` 재사용). 1.8이 남긴 폼의 "진료과 마스터 구축 후 배정합니다" 보류 주석을 실제 피커로 대체한다.

**AC2 (직원 진료과 재배정 — 수정):**
**Given** 이미 생성된 직원에 대해
**When** 관리자가 소속 진료과를 변경(또는 해제)하면
**Then** 신규 `PATCH /v1/admin/users/{id}/department`(권한 `user.manage`)가 `update_user_department()`를 통해 `users.department_id`를 갱신한다. 비활성 진료과로의 신규 배정은 `_assert_department_assignable`로 422 `inactive_department`/`invalid_department`. 직원 디렉터리(staff-directory)에 "소속 진료과" 열을 추가하고(`departmentLabel` 재사용 — 비활성 "(비활성)"·미상 "(미상)"), 재배정 UI(인라인 select 또는 편집 흐름)를 제공한다. 변경은 0004 감사 트리거로 자동 기록된다.

**AC3 (웹 다중행 pending 경합 — 2.1~2.4 이월):**
**Given** 마스터 관리 화면에서 사용자가 여러 행/탭을 빠르게 토글할 때
**When** 한 행의 비동기 작업(active 토글·dependents 조회)이 진행 중에 다른 행을 토글하면
**Then** 단일 `pendingId: string|null`(masters-manager.tsx:115)을 **id 집합(`Set<string>`)** 으로 바꿔, 각 작업의 `finally`가 **자기 id만** 해제한다. 늦게 끝난 `finally`가 무관한 행의 pending을 조기 해제하거나 `setConfirm`이 다른 다이얼로그를 덮어쓰는 경합을 제거한다. staff-directory의 단일 `isPending`도 동형으로 점검(다중행 상태 변경 시).

**AC4 (마스터 조회 단일 실패점 → 부분 강등 — 2.2 이월):**
**Given** 관리 화면이 5종 마스터를 `Promise.all`로 조회할 때(masters.ts:101-136)
**When** 한 테이블 조회만 실패하면
**Then** **첫 에러로 전체 화면을 다운시키지 않고**, 정상 테이블은 표시하고 실패한 테이블만 에러 상태로 강등한다(per-table 결과 분리). 5종 중 1종 실패가 관리화면 전체를 깨뜨리던 표면을 제거한다. (현 fail-loud는 init RSC 전체 throw — 이를 부분 강등으로 전환하되, 실패 가시성은 유지.)

**AC5 (검색 피커 재시도 — 2.3 이월):**
**Given** 검색 피커가 코드 로드에 실패해 `loadError`가 설정된 상태에서(master-search-picker.tsx:100-121)
**When** 사용자가 복구를 시도하면
**Then** `role="alert"` 메시지 옆에 **재시도 버튼**(또는 자동 재조회)을 제공해 페이지 전체 remount 없이 재조회한다. fail-loud(입력 비활성·경고 표시)는 유지하되 복구 경로를 추가한다.

**AC6 (의존성 카운트 테스트 격리 — 2.4·2.5 이월):**
**Given** `test_department_dependents_count`(test_masters_integration.py:323-376)가 "첫 활성 비-admin 직원"으로 **시드 doctor(000000a2)를 차용**하던 상태에서
**When** 테스트를 리팩터하면
**Then** 시드 doctor를 빌리지 않고 **전용 임시 직원을 생성→검증→정리**(또는 시드 uid 제외 + 임시 직원)한다. 크래시/타임아웃 시에도 시드 doctor의 `department_id`가 오염되지 않는다(2.5가 doctor→IM 배정으로 키운 영향 제거). 2.5 seed.sql의 `doctor→IM`은 **데모 시드로 유지**하되(이제 AC1/AC2의 실제 기능이 뒷받침) 주석에서 "우회" 프레이밍을 데모 시드로 재분류한다.

**AC7 (회귀 가드):**
**Given** 본 스토리의 모든 변경에 대해
**When** 보강이 끝나면
**Then** (a) **신규 마이그레이션·DDL 0**(department_id 컬럼·FK 기존), (b) 기존 동작·테스트 무회귀(API 206 pass/web 122 pass 기준), (c) 쓰기=FastAPI(service_role+권한 게이트)·읽기=Supabase 직접조회·snake_case·에러 봉투·감사 append-only 규칙 준수, (d) 새 권한·새 nav 항목·새 라이브러리 도입 0(전부 기존 자산 재사용).
