## 2. 잘된 점 (강점)

- **아키텍처-우선 · DB 불변식 일관성** — "불변식은 DB가 소유"(RLS·트리거·제약·pgcrypto)가 10개 스토리 내내 흔들리지 않음. 상태/권한/감사 로직을 앱에서 재구현하지 않음.
- **패턴 재사용의 복리효과** — 1.3 감사 트리거 → 1.7·1.8·1.10 재사용; 1.5 `authenticated_conn` actor 주입 → 전 경로 토대; 관리자 화면 골격(RSC 가드+클라+`apiFetch`) → 1.7→1.8→1.10 복제; 쓰기 권위 1.7(단일 DML)→1.8(두 시스템 오케스트레이션)→1.10(읽기) 진화.
- **암묵지→명시지 (프로젝트 핵심 목표 달성)** — `deferred-work.md`가 매 리뷰의 "왜 미뤘는지 + 언제 처리할지"를 추적. 1.1 env fail-fast(→1.5 ✅), 1.5 TOCTOU(→1.8 확장 ✅)처럼 실제로 후속에서 해소됨.
- **보안 방어심층** — append-only 삼중 잠금(RLS deny+GRANT 회수+BEFORE 트리거), 3계층 RBAC(UI 노출/명령 권위/RLS), PII 암호화+HMAC+마스킹 심.
- **의존성 절제** — TanStack/zustand/server-only 미도입 유지로 표면 린하게 관리.
- **디자인 언어 일관성** — 음영 비의존·색+글리프+굵기 3중 인코딩이 1.2 디자인 시스템부터 1.10 감사 배지·diff까지 유지.

## 3. 개선점 (반복 마찰)

- **RLS 본인행 ↔ 관리자 조회 충돌** — `users` 본인행 RLS로 1.8·1.10이 직접조회 불가 → FastAPI 경유 강제 + `react-hooks/set-state-in-effect` 린트 예외 반복. (전역 참조 데이터인 마스터는 이 마찰 없음 → Epic 2 읽기는 직접조회 가능.)
- **TOCTOU(권한평가 ↔ 쓰기 트랜잭션 분리)** — 1.5 식별, 1.7·1.8이 "동일 트랜잭션 재평가"로 부분 대응. Epic 3+ 쓰기 엔드포인트에서 패턴 강제 필요.
- **Auth(GoTrue) ↔ DB 드리프트 클러스터** — ban 동기화 실패 재조정·고아 Auth 정리·임시비번 강제변경·last-admin 가드. 보안 하드닝으로 묶어 처리 권장.
- **CI·env 관대함** — `supabase db lint || true`·skip-not-fail·web env fail-fast 핑퐁(1.4→1.6). Post-MVP 의도지만 "녹색 ≠ 정확" 인식 유지.
- **PII 마스킹의 best-effort 한계** — 로그 마스킹 RRN 한정, 감사 마스킹 렌더 계층 한정. 1차 방어는 "raw PII 미로깅" 규율, Epic 3에서 서버측 정책 보강.
