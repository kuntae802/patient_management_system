# Story 3.3: 보호자 정보 기록

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 원무 직원,
I want 환자 보호자(성명·관계·연락처)를 기록·조회·수정·삭제하기를,
so that 고령·미성년 환자의 연락·동의 경로가 확보된다.

## Acceptance Criteria

1. **AC1 — 보호자 추가·조회·수정·삭제 (FR-006):** 환자 레코드가 존재할 때, `patient.update` 보유자가 보호자(`name`·`relationship`·`phone`)를 추가하면 `guardians` 행이 생성되어 환자(`patient_id`)와 연결되고, `patient.read` 보유 직원이 환자 상세에서 보호자 목록을 조회하면 표시된다. 한 환자에 보호자 **여러 명(1:N)** 등록·수정·삭제가 가능하다. **컬럼·GRANT·RLS·감사 트리거는 `0009_patients.sql`에 이미 존재** — 3.3은 입력·조회·수정·삭제 경로(API + 웹 UI)만 추가하고 **신규 마이그레이션을 만들지 않는다**. 모든 변경(insert/update/delete)은 0009 `trg_guardians_audit`가 자동 기록한다(actor=GUC sub).

2. **AC2 — 권한·쓰기 권위·연락처 PII 처리:** 쓰기(추가·수정·삭제)는 **FastAPI(service_role) 경로로만** 수행된다(authenticated 는 guardians 쓰기 권한 없음 — 0009 GRANT·RLS 쓰기정책 부재). 게이트 = `require_permission("patient.update")` → 403, **실제 쓰기는 동일 트랜잭션에서 `_require_patient_update` 재평가(TOCTOU 차단)** 후 DML. 조회 게이트 = `patient.read`. **보호자 연락처(`phone`)는 환자 연락처(phone/address)와 동일하게 평문·staff 직접조회**(마스킹·reveal 게이트 미적용) — 에픽 AC2의 "연락처 reveal 게이트" 요구는 환자 phone/address/email과 묶어 **'연락처 PII reveal 일관화' 교차절단으로 deferred-work에 이월**(아래 §이월·교차절단 ③). 미존재(환자 또는 보호자) → 404. 응답에는 민감 암호 컬럼이 없다(guardians 는 `_enc`/`_hash` 컬럼 자체가 없음).

3. **AC3 — 웹 보호자 관리 UI (환자 상세 통합):** 환자 상세 공유 풀페이지(`(staff)/patients/[patientId]`, 3.2 생성)에 **보호자 섹션**을 추가한다 — 보호자 목록 표시 + `<PermissionGate permission="patient.update">` 로 게이트된 **추가·수정·삭제**. `relationship`은 **자유텍스트**(공통값 프리셋 제안, enum 미강제 — 0009/glossary). 제출 중 버튼 disable(이중 제출 방지). 빈 목록은 "등록된 보호자 없음" 중립 표시(색 단독 의존 금지). 삭제는 확인 단계를 거친다.

> **이월·교차절단 인수 조건(이 스토리에서 확인):** ① **신규 마이그레이션 0건** — `guardians` 테이블·`grant select`·RLS(staff/self)·`trg_guardians_audit`가 모두 `0009_patients.sql`에 존재함을 **확인만** 하고 재선언하지 말 것(중복 정책·이중 감사·번호 드리프트 위험). ② **A-2 TOCTOU 연속** — guardian 쓰기(추가·수정·삭제)도 `patient.update` 쓰기와 동일하게 in-txn 재평가(게이트+재평가 이중). ③ **연락처 PII reveal 일관화(신규 교차절단 이월)** — 에픽 AC2/UX-DR22는 보호자 연락처 reveal을 주민번호 동일 게이트+감사로 요구하나, 현 구현 선례(3.1/3.2)는 **환자 phone/address/email을 평문·staff 직접조회**(reveal 게이트 미적용)로 두었고 guardians.phone도 평문 컬럼이며, read-감사 메커니즘(`decrypt_sensitive` 자가-감사)은 **암호화 필드 전용**이라 평문 phone에는 깨끗한 감사 훅이 없다(앱의 `audit_logs` 직접 INSERT 금지=트리거 전용). **결정(사용자 승인): 선례 미러 + reveal 이월** — 3.3은 보호자 CRUD까지, 연락처 reveal 게이트는 환자 연락처와 묶어 `deferred-work.md`에 신규 항목으로 이월(전용 감사·reveal 하드닝 스토리 또는 Epic 4 진료 허브 배너에서 일괄). **A-3 감사 PII(계속 defer)** — guardian `name`/`phone`이 audit 스냅샷에 적재되나, 이미 deferred-work L153이 patients/guardians 평문 PII를 통합 추적 중(중복 항목, 신설 금지). 상세는 Dev Notes §이월·교차절단 참조.

### 보호자 API 설계 (서브리소스 — `update_patient_clinical_profile`/`insert_patient` 미러)

- **경로:** `/api/v1/patients/{patient_id}/guardians`(목록·생성), `/{guardian_id}`(수정·삭제). 보호자는 환자의 **sub-resource** — URL 계층이 소유관계를 표현하고 기존 patients 라우터(prefix `/patients`)에 자연 편입. clinical-profile(3.2) 과 동일한 sub-path 관례.
- **쓰기 권위:** authenticated 는 guardians UPDATE/INSERT/DELETE 권한 없음(0009 GRANT·RLS 쓰기정책 부재) → service_role(FastAPI)만 쓰기. 게이트(`require_permission("patient.update")`)는 방어심층, **진짜 권위는 `_op` 안 `_require_patient_update(conn)` in-txn 재평가**(A-2 연속 — `insert_patient`/`update_patient_clinical_profile` 와 동형). 보호자 추가·수정·삭제는 모두 **환자 정보 수정** 행위 → `patient.update`(별도 guardian.* 권한 신설 금지, 0002 미시드).
- **404 매핑:** ① 생성 — patient 미존재 시 FK 위반(`ForeignKeyViolationError`) → db 가 `NotFoundError`(환자 404). ② 수정/삭제 — `where id=$1 and patient_id=$2` 0행 → db `None`/None → 서비스 `NotFoundError`(보호자 404). RLS 우회(service_role)라 미존재와 권한밖을 동일 404 로(존재 누설 회피, fetch_patient 동형).
- **patient_id 스코프(보안 필수):** 수정·삭제는 `where id=guardian_id and patient_id=patient_id` 로 **반드시 환자 스코프** — URL 의 patient_id 와 무관한 guardian_id 로 타 환자 보호자를 수정/삭제하는 IDOR 차단.
- **hard delete:** guardians 는 `is_active`(soft delete) 컬럼이 없다(patients 와 대비 — 보호자는 진료·법적 보존 레코드가 아님). 삭제 = 물리 DELETE, 0009 `trg_guardians_audit`(after delete)가 before_data 스냅샷으로 추적성 보장. (cascade 는 환자 삭제용 — 현재 환자 hard delete 없음.)
- **목록 응답 형태:** 작은 바운디드 sub-collection → `list[GuardianResponse]` 직접 배열(아키텍처 §Format "성공=리소스/배열 직접"; `{data,meta}` 페이지 봉투는 top-level 페이지네이션 목록용). 정렬 = `created_at`(등록순).
- **감사:** 0009 `trg_guardians_audit`(after insert/update/delete) 가 자동 기록(actor=`app.actor_id` GUC=sub). **앱이 `audit_logs` 직접 INSERT 금지.**

### PII 경계 + 연락처 reveal 일관화(이월) — 교차절단 결정 근거

- **보호자 연락처 = 평문·staff 직접조회(환자 phone 선례 미러):** guardians.phone 은 0009 에서 평문 컬럼(`_enc`/`_hash`/`_masked` 없음). 환자 phone/address/email 도 동일하게 평문이고 `PatientSummary`(patient-detail.tsx:177-188)가 phone 을 **그대로 표시**한다 — 현 구현은 **주민번호만** reveal 게이트(암호화+마스킹+`decrypt_sensitive` 자가-감사), 연락처류는 `patient.read` staff 경계로 직접 노출한다. 3.3 은 이 선례를 따른다(보호자가 환자보다 더 보호되는 모순 회피).
