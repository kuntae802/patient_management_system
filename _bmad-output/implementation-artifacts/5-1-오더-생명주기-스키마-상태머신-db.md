## Story

As a 백엔드 개발자,
I want 오더 유형별(처방·검사·처치) 생명주기와 지시자/수행자/시각 추적, 재수행 차단을 DB로 강제하기를,
so that 모든 오더가 일관된 상태머신과 추적성·감사성을 갖고, 이후 발행·수행·판독·수가 스토리(5.2~5.10)가 안전하게 이 토대에 매달린다.

## Acceptance Criteria

> 출처: `epics.md` Epic 5 / Story 5.1. FR-080(지시자·수행자·시각 분리 추적), FR-093(재수행 차단 DB 최종선),
> NFR-040(상태 전이 무결성 — 역행·건너뛰기·재수행 차단). 아키텍처 §35·§175·§428(오더 유형별 상태머신 + 상태 어휘 통일 ⑤).

**AC1 — 오더 테이블 + 유형별 상태 어휘(state machine vocabulary):**
`0015_orders.sql`에서 `prescriptions`(+`prescription_details`)·`examinations`·`treatment_orders`·`equipment` 테이블이 생성되고,
유형별 상태 어휘가 `text + CHECK`로 강제된다:
- 처방(`prescriptions.status`): `issued`(발행) → `dispensed`(발급, 원외 약국). 초기 = `issued`.
- 검사·영상(`examinations.status`): `ordered`(지시) → `performed`(수행) → `completed`(판독/완료). 초기 = `ordered`.
- 처치(`treatment_orders.status`): `ordered`(지시) → `performed`(수행) → `completed`(완료). 초기 = `ordered`.

**AC2 — 지시자·수행자·완료자/시각 FK 분리 기록(FR-080):**
각 오더가 지시자(`ordered_by` + `ordered_at`)·수행자(`performed_by` + `performed_at`)·완료자(`completed_by` + `completed_at`)를
유형에 맞게 분리 기록한다(전부 `users` FK). 전이 RPC가 해당 액터/시각을 원자적으로 세팅한다.

**AC3 — 상태 전이 강제 + 비정상 초기상태 차단(NFR-040):**
전이 트리거가 INSERT 초기상태(유형별 초기값만 허용, 그 외 `PT409`)와 UPDATE 전이 매트릭스(정의된 전진만 — 역행·건너뛰기 차단,
위반 시 `PT409`)를 강제한다. 직접 `update`·`service_role` 쓰기까지 봉쇄(최종선).

**AC4 — 재수행 차단(FR-093 DB 최종선):**
이미 수행(`performed`)된 검사·처치 오더에 대해 수행 전이를 재시도하면 DB가 차단한다(전이 RPC의 소스상태 선검사 = `ordered`이어야 함 →
재호출 시 `PT409`; 트리거 매트릭스가 `performed→performed`를 거부하는 최종 백스톱).

**AC5 — 전이 RPC 3종(SECURITY DEFINER · 권한 자가 게이트):**
`perform_examination`·`complete_examination`·`perform_treatment_order` RPC가 `has_permission` 동일-txn 재평가 게이트 +
`for update` 행 잠금 + 소스상태 precondition을 갖고 생성된다(0010 전이 RPC 패턴 동형). 권한 미보유 = `42501`(→403),
대상 없음 = `PT404`(→404), 잘못된 전이/재수행 = `PT409`(→409).

**AC6 — 모든 오더 전이가 감사 기록:**
5개 테이블 전부 `audit_trigger_fn`(0004) 트리거 부착 → INSERT(create)·UPDATE(update)·DELETE(delete)가 actor와 함께
`audit_logs`에 append-only 기록된다. 오더 스냅샷은 구조적 FK·플래그·숫자만(자유텍스트 임상 서사 없음) → `_SENSITIVE_KEY` 마스킹 집합 무변경.

**AC7 — 권한 카탈로그 + RLS:**
신규 권한 `order.read`·`examination.perform`·`examination.complete`가 시드되고 admin 부트 grant가 재실행된다.
RLS: 직원 = `order.read` 권한 보유 시 전체 오더 행 / 환자 = 본인 내원의 오더만(encounter→patient→auth_uid 경유) / `equipment`는
전역 참조(authenticated SELECT 전체). 쓰기 정책 없음(쓰기는 service_role/RPC).

### 상태머신 모델 — 0010 패턴 재사용(전이 트리거 + 전이 RPC + 커스텀 SQLSTATE)

오더는 **유형별 per-table 상태머신**(아키텍처 §35 "오더 유형별 상태머신"). 단일 통합 orders 테이블 없음 — `order`(총칭, glossary §41)는
개념적 추상(우 오더 패널 5.5가 query/UI에서 union). 각 유형 매트릭스(gap ⑤ 확정 — 본 스토리가 최종):

```
prescriptions   : (INSERT)→ issued    | issued → dispensed                         [dispense RPC = Epic 7]
examinations    : (INSERT)→ ordered   | ordered → performed → completed            [perform/complete RPC = 본 스토리]
treatment_orders: (INSERT)→ ordered   | ordered → performed → completed            [perform RPC = 본 스토리; complete 예약]
종결·역행 없음(forward-only). 잘못된 전이/비정상 초기상태/재수행 = SQLSTATE 'PT409'(→409).
```

**`enforce_act_order_transition()` 공용**: examinations·treatment_orders 매트릭스가 동일(ordered→performed→completed)하므로 트리거
함수 1개를 두 트리거가 공유(0010 단일-테이블-단일-함수에서 한 단계 확장 — 매트릭스 동일성이 근거). `prescriptions`는 별도
`enforce_prescription_transition()`(매트릭스 상이). **트리거 = INSERT 초기상태 가드 + UPDATE 전이 매트릭스 강제**(direct update·
service_role 쓰기까지 봉쇄 = NFR-040 최종 백스톱, 0010:62-98 동형). same-status UPDATE(비-상태 컬럼 갱신, 예 5.8 equipment_id/
result path 세팅) 통과 허용.

**전이 RPC = 전진의 단일 진실**(0010:100-246 `start_consult` 동형):
- `SECURITY DEFINER` + `set search_path = public` + `has_permission(...)` 동일-txn 재평가(TOCTOU) + `select … for update` 행 잠금 +
  **소스상태 precondition**(예 perform_examination는 status='ordered'이어야 — 아니면 PT409).
- 이 precondition이 **FR-093 재수행 차단의 핵심**: 트리거의 same-status 통과(비-상태 컬럼 갱신 허용)가 만드는 사각(이미 performed인
  오더에 perform 재호출 → 액터/시각 덮어쓰기)을 차단. 0010:152 `start_consult` 진료 탈취 차단과 동일 메커니즘.
- 액터 = `(select auth.uid())`(0010:157 `start_consult` doctor_id 세팅 동형). performed_by/completed_by = 호출자.

**오더 생성(issue/order)은 RPC 아님 — service_role 직접 INSERT(다운스트림)**: 0010에서 walk-in INSERT가 service_role 직접(4.2
`insert_walk_in_encounter`)이고 전이만 RPC였던 선례. 처방 발행(5.2)·검사/처치 지시(5.3/5.4)는 service_role 직접 INSERT(`_require_*` API
TOCTOU 권한 재평가 + 트리거가 초기상태 검증). 본 스토리는 스키마 + 초기상태 트리거 가드 + 전진 RPC만 제공.

### ⚠️ 커스텀 SQLSTATE — 신규 매핑 0 (PT409/PT404 재사용)

`api/app/core/db.py:_map_pg_sqlstate`가 이미 **PT409**(→409 `invalid_transition`)·**PT404**(→404)·**PT422**(→422, 4.7)를 매핑한다.
오더 전이의 잘못된 전이·재수행·종결 재전이 = **PT409 재사용**(어휘 동일 — "잘못된 상태 전이"), not-found = **PT404 재사용**. **신규
SQLSTATE·`_map_pg_sqlstate` 변경 불필요**(본 스토리는 DB-only이므로 db.py 무변경; API 스토리 5.2~5.9도 기존 매핑 재사용 — 4.2 이후
첫 신규 매핑이 없음). 단, 5.5 알레르기 하드블록 등 신규 도메인 오류가 생기면 그 스토리가 신규 PTxxx + 매핑 소유.

### FR-051 — 처방 ↔ 진단 연결(encounter_diagnoses FK)

`prescriptions.encounter_diagnosis_id` → `encounter_diagnoses.id`(nullable, 4.7 테이블). 처방 근거 = 그 내원에 부착된 진단(주/부상병).
**값 세팅은 5.2**(처방 발행 시 진단 A와 연결). 본 스토리는 nullable FK 컬럼만 생성. (마스터 `diagnoses`가 아니라 내원진단
`encounter_diagnoses`를 참조 — 근거는 "이 내원의 부착 진단".)

### ⚠️ 감사 마스킹 — 변경 0 (4.7 동형, 4.6과 핵심 차이)

오더 테이블 스냅샷 = **구조적 FK(encounter_id·drug_id·fee_schedule_id·equipment_id·*_by)·플래그·숫자(dose/duration_days)·짧은 구조화
텍스트(frequency·usage_instruction)**. **자유텍스트 임상 서사 없음**(0010:52-53 encounters가 자유텍스트 회피한 자세 계승):
- 약품은 `drug_id` FK(불투명 UUID — diagnosis_id 동일 posture, 4.7 §결정4 미마스킹). dose/frequency는 drug_id 조인 없이는 무의미 →
  진단 FK보다 더 드러내지 않음 → `_SENSITIVE_KEY` **무변경**.
- **판독 소견(examination finding)·처치 수행 내용 같은 자유 서사는 본 스토리에 컬럼 없음** — 0010이 SOAP/진단 자유텍스트를 4.6/4.7로
  미룬 선례 그대로, 판독 소견 = **5.9 소유**(컬럼 추가 + `_SENSITIVE_KEY` 마스킹 확장 양쪽[audit.py + audit.ts]), 처치 수행 내용 =
  **5.7 소유**(`nursing_record` 별도 테이블). 본 스토리는 자유 서사 컬럼을 두지 않아 감사 마스킹 교차절단을 회피한다.

주석으로 명시(0014:87-90 미러): "스냅샷 = FK·플래그·숫자·구조화 텍스트 → 3.6 마스킹 집합 변경 불요(자유 임상 서사는 5.7/5.9 소유)".

### 권한 · 시드 grant · 403 baseline 이동(중요 — Epic 4 교훈)

**신규 권한 3종**(0015 마이그): `order.read`(RLS 직원 게이트 — 의사·간호·방사선만, 원무 제외 = 최소권한, diagnosis.read 자세
계승)·`examination.perform`(촬영/검체 수행 — 방사선사·간호)·`examination.complete`(판독 — 의사). **admin 부트 grant 재실행 필수**.
`treatment.perform`은 0002 기존(admin-only였음 → 본 스토리가 nurse에 grant).

**seed.sql 역할 grant**(데모/통합테스트용 — 프로덕션 grant는 1.7 매트릭스 소유):
| 역할 | grant | 용도 |
|---|---|---|
| doctor | order.read, examination.complete | 오더 패널 조회 + 판독(5.9) |
| nurse | order.read, examination.perform, treatment.perform | 검체/처치 수행 워크리스트(5.3/5.7) |
| radiologist | order.read, examination.perform | 촬영 워크리스트(5.8) |
| reception | (없음) | **오더-403 baseline** |

**⚠️ 403 baseline 이동(Epic 4 "공유 계약 변경=baseline 스윕" 교훈)**: seed.sql 19행은 nurse = "무권한 baseline(403 통합 검증용)"이라
적었고, 주석은 "간호 권한은 Epic 5"라 **예고**한다. 본 스토리가 nurse에 오더 권한을 주므로 nurse는 **오더 권한의 zero-baseline이
더 이상 아님**. 그러나 nurse는 여전히 encounter.*/patient.* 권한 0 → **기존 4.4/4.5 nurse-baseline 회귀(test_*_forbidden_without_read·
rpc_permission_denied_for_nurse·rls_patient_sees_only_own)는 무영향**(그 테스트는 encounter/patient 권한을 검사, 오더 grant는 비중첩).
**오더 RPC 403 테스트는 reception(EMP0003, 오더 권한 0)을 baseline으로 사용**. seed.sql 주석을 이 분리(nurse=encounter/patient
baseline, reception=order baseline)로 정정한다.

**radiologist 데모 계정**: 현재 데모 계정은 admin/doctor/reception/nurse(EMP0001~0004)뿐. radiologist 계정은 없음 →
`perform_examination` 성공 테스트는 **nurse**(examination.perform 보유)로 검증(radiologist 계정 신설은 5.8 소유 — 촬영 UI 착수 시).
radiologist 역할 grant는 forward-looking(5.8 대비)으로 두되 본 스토리 테스트는 nurse/doctor/reception 계정만 가동.
