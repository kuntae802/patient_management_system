## Story

As a 원무 직원,
I want 결제 수단(카드/현금/계좌이체)을 기록해 draft 수납 건을 finalize 하고, 같은 트랜잭션에서 내원을 '완료'로 전이하기를,
so that 한 번의 내원이 결제·정산으로 원자적으로 닫히고(영수증 번호 부여·결제 기록 동결), 7.5~7.6(계산서·영수증·세부산정내역서)이 이 finalized 수납 건 위에서 출력된다.

> **Epic 7 네 번째 스토리 — finalize·결제·내원 완료 레이어(Epic 7 "가장 어려운 도메인"의 닫힘 지점).** 7.1(`0045`)이 결제 6컬럼(`payment_method`·`payment_no`·`finalized_at`·`finalized_by`·…)을 **선언만**, 7.2(`0046` `build_payment`)가 라인 적재·총액 롤업, 7.3(`0047` `price_payment`)이 본인부담 산정까지 채웠다. **7.4는 그 draft 수납 건을 finalized 로 전이**한다: `0048` `finalize_payment` DB 함수가 결제수단·영수증번호(`payment_no`)·`finalized_at/by`·`paid_amount_krw` 를 기록하고 **`complete_encounter`(0010/0014·기존)를 호출해 내원을 in_progress→completed 로 전이**한다. finalize = `build_payment`→`price_payment`→`finalize_payment`→`complete_encounter` 를 **한 FastAPI 트랜잭션**으로 호출(원자성·NFR-041). 웹은 결제수단 토글 + **환자 신원(이름+차트번호) 재진술 confirm**(고위험 비가역·UX-DR21) 후 finalize.

**AC1 — `payment_no` 영수증번호 시퀀스(0048·전역 시퀀스 + 날짜 프리픽스, 설계 결정 ②):**
`0048`에서 영수증번호 생성 토대가 만들어진다:
- `create sequence if not exists public.payment_no_seq;`(전역·일별 리셋 없음 — 카운터 테이블/락 불요·충돌 불가).
- `payment_no` 포맷 = `'R-' || to_char(now() at time zone 'Asia/Seoul', 'YYYYMMDD') || '-' || lpad(nextval('public.payment_no_seq')::text, 6, '0')`(예 `R-20260623-000042`). 날짜는 **finalize 시점 KST** 가독용(`now()` timestamptz UTC→KST 변환·`fetch_encounters` 일자 계약 동형), 시퀀스가 유일성 보장. `payments.payment_no UNIQUE`(0045)가 최종선.
- 포맷 생성은 `finalize_payment` 내부에 **인라인**(별도 함수 없이 — price_payment 의 인라인 산식 선례). `payment_no_seq` 는 신규 식별자 → glossary 등재(AC9).

**AC2 — finalize 함수 `finalize_payment(encounter_id, payment_method)`(0048·SECURITY DEFINER·결제+완료 원자, 설계 결정 ①③):**
`0048`에서 `finalize_payment(p_encounter_id uuid, p_payment_method text) returns uuid`(`create or replace`·`language plpgsql`·`security definer`·`set search_path=public`)가 생성된다. draft 수납 건을 finalized 로 전이하고 내원을 완료한다:
- **헤더 룩업 + 행 잠금(1회)**: `select id, status, total_amount_krw, copay_amount_krw into ... from public.payments where encounter_id = p_encounter_id for update`. 헤더 없음(빌드 전) → **`raise exception ... using errcode='PT404'`**(정상 경로는 build 선행이라 미발생·방어).
- **상태 가드(비가역·이중결제 차단)**: `v_status <> 'draft'` 면 **`raise exception ... using errcode='PT409'`**(이미 finalized/cancelled — 재finalize·이중결제 차단·멱등 아님: 고위험 비가역이라 엄격). `_map_pg_sqlstate` → 409 `invalid_transition`.
- **정산 대상 가드**: `v_total <= 0` 면 **`raise exception ... using errcode='PT409'`**(정산할 수가 항목 0 — 빈 내원 finalize 차단). 정상 경로(in_progress=진찰료 ≥1·5.10)에서 미발생·방어.
- **결제수단 가드**: `p_payment_method` 는 컬럼 CHECK(`in ('card','cash','transfer')`·0045)가 최종선. API 레이어 Pydantic `Literal` 이 1차선(422). 함수는 컬럼 CHECK 에 위임(잘못된 값=23514).
- **결제 컬럼 기록(UPDATE payments)**: `status='finalized'` · `payment_method=p_payment_method` · `payment_no=`(AC1 인라인 생성) · `finalized_at=now()` · `finalized_by = nullif(current_setting('app.actor_id', true), '')::uuid`(authenticated_conn GUC — build_payment 감사 actor 동형) · `paid_amount_krw = v_copay`(**전액 정산 — 본인부담금 전액 결제·설계 결정 ③**) · `updated_at=now()`.
- **내원 완료 전이(complete_encounter 호출·설계 결정 ①)**: `perform public.complete_encounter(p_encounter_id);`. 기존 RPC(0010·0014 재정의)가 **in_progress→completed 전이 + 주상병 게이트(PT422) + 상태 가드(PT409) + not-found(PT404)** 를 그대로 수행(상태머신 재구현 0). 주상병 미지정 시 PT422 → finalize 전체 롤백(원자) → 422 `primary_diagnosis_required`(원무는 의사 진단 완료 전 결제 불가 = 안전 게이트). RPC 내부 `has_permission('encounter.complete')` 는 **reception 가 보유**(AC4 seed grant)해야 통과.
- 반환: payment_id(FastAPI 가 헤더+라인 재조회).
- 함수 EXECUTE posture(build_payment·price_payment 동형): `revoke all on function public.finalize_payment(uuid, text) from public, anon, authenticated` + `grant execute on function public.finalize_payment(uuid, text) to service_role`(service_role/FastAPI만 — finalize 위조·이중결제 위조 차단).

**AC3 — `payments` finalize 컬럼 일관성 partial CHECK(0048·deferred-work L365 해소):**
`0048`에서 `alter table public.payments add constraint payments_finalized_consistency check (status <> 'finalized' or (payment_no is not null and finalized_at is not null and finalized_by is not null and payment_method is not null))` 추가. finalized 행은 영수증번호·결제시각·담당자·결제수단을 **모두** 가져야 함(부분 finalize·일관성 깨짐 차단). draft 행은 `status <> 'finalized'` true → 통과(기존 행 무영향·finalized 행 0이라 ALTER 성공). **cancelled 컬럼 일관성(`cancelled_at`/`cancel_reason`) CHECK = 7.9 소관**(취소 로직 미구현 — 스코프 누수 방지). deferred-work L365 의 finalized 부분 해소·cancelled 잔여 표기.

### ⚠️ 내원 완료 모델 (확정·비자명 — 반드시 이해)

이 PMS 는 진료(SOAP·오더·진단)가 끝나도 내원이 **`in_progress` 로 유지**된다. 근거: billing 워크리스트(`fetch_billing_worklist`·db.py L4163)가 `where e.status = 'in_progress'` 필터 — 정산 대상 = in_progress 내원. **결제 finalize 가 완료 전이의 트리거**(in_progress→completed). 따라서:
- 7.4 `finalize_payment` 가 `complete_encounter` 를 호출해 완료시킨다 → reception 이 `encounter.complete` 보유해야 함(AC4 grant).
- Story 4.7 의 `complete_encounter` + `POST /encounters/{id}/complete`(doctor 게이트)는 **의사가 직접 완료할 수 있는 경로로 존재하나, 표준 외래 흐름은 billing-completes**. 의사가 선완료하면 내원이 completed → 워크리스트 미노출(billing 진입 불가) → 정상 경로 미발생.
- **엣지(이월)**: 의사 선완료 후 billing finalize 시도 → complete_encounter PT409(비-in_progress) → finalize 422/409. 워크리스트가 in_progress 만 노출하므로 정상 경로 회피. 완료 주체 단일화(누가 완료하는가)는 운영 정책 — 현 구현은 billing-completes 기본 + 의사 경로 잔존(이월).

### ⚠️ finalize 원자성 = build→price→finalize→complete 한 트랜잭션 (NFR-041)

project-context §"불변식은 DB가 소유 · 다단계 명령 오케스트레이션=FastAPI". finalize 엔드포인트의 `_op` 는 한 `authenticated_conn` 트랜잭션에서:
```
_require_payment_manage(conn)              -- 쓰기 TOCTOU 재평가(payment.manage)
exists check (encounter)                   -- 404
select public.build_payment($1)            -- 집계(멱등·신선화)
select public.price_payment($1)            -- 산정(L385 해소: price 가 finalize 선행)
select public.finalize_payment($1, $2)     -- 결제 컬럼 + complete_encounter(완료 전이)
_PAYMENT_HEADER_SELECT + details 재조회      -- 결과 반환
```
중간 RPC 실패(예: complete_encounter PT422 주상병 미지정) → **전체 롤백**(payment 여전히 draft·encounter 여전히 in_progress). build_payment·price_payment 는 status≠draft early-return 가드 → finalize 전엔 항상 draft 라 신선 집계·산정. finalize 후엔 status='finalized' 라 재호출 시 build/price no-op + finalize PT409(이중결제 차단). **L385(finalize-before-price) 해소**: price 가 항상 finalize 직전 실행.

### ⚠️ finalize_payment 로직 = DB 함수 (Python/TS 재구현 금지)

결제 컬럼 기록·영수증번호 생성·완료 전이 호출은 **`finalize_payment` DB 함수**가 소유. FastAPI 는 RPC 호출 + 응답 매핑만(build_payment·price_payment 선례). `payment_no` 생성(시퀀스+포맷)·`paid_amount_krw=copay` 산식·`complete_encounter` 호출 = SQL(DB 소유). **`complete_encounter` 를 재호출**(상태머신·주상병 게이트 재구현 금지) — 7.4 가 전이 로직을 복제하지 않는다.

### ⚠️ finalized_by = app.actor_id GUC

`finalize_payment`(SECURITY DEFINER)는 `current_setting('app.actor_id', true)` 로 actor 를 읽는다. `authenticated_conn`(db.py L114-130)이 트랜잭션 진입 시 `set_config('app.actor_id', sub, true)` → finalize 함수에서 `nullif(current_setting('app.actor_id', true), '')::uuid` 로 `finalized_by` 채움(build_payment 감사 actor 동일 GUC). `finalized_by uuid references users(id)`(0045) — sub 는 users.id(= auth uid).

### ⚠️ payment_no 포맷·시퀀스 (정확히)

```
payment_no := 'R-' || to_char(now() at time zone 'Asia/Seoul', 'YYYYMMDD')
                   || '-' || lpad(nextval('public.payment_no_seq')::text, 6, '0')
```
- `now()` = timestamptz UTC → `at time zone 'Asia/Seoul'` = KST wall time(자정 경계 KST 기준 — 영수증 날짜 직관). `fetch_encounters`/워크리스트 일자 계약(KST) 일관.
- `payment_no_seq` = **전역 단조증가**(일별 리셋 없음 → 락/카운터 테이블 불요·동시 finalize 충돌 0). 날짜는 가독용·유일성은 시퀀스. `payments.payment_no UNIQUE`(0045) 최종선.
- 6자리 패딩(`000042`)·일 100만건 전 무충돌(외래 규모 충분). 초과 시 lpad 가 자릿수 확장(유일성 유지).
