# Story 6.7: 노쇼 카운트 · 임계치 제한

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 시스템(원무·환자 예약 흐름),
I want 환자별 노쇼(미방문) 횟수를 세고 임계치(기본 2회) 초과 시 신규 예약을 제한하기를,
so that 상습 노쇼로 인한 슬롯 낭비를 줄이고 진짜 필요한 환자가 슬롯을 쓰게 한다.

> FR-015: 시스템은 환자별 노쇼 횟수를 기록하고, 임계치(기본 2회) 초과 시 예약을 제한한다.
> UX-DR15: "노쇼 임계(기본 2회) 초과 제한"(예약 캘린더 정책의 일부).

## Acceptance Criteria

**AC1 — 노쇼 카운트 기록·집계(FR-015)**
- **Given** 예약이 노쇼로 종결될 때(`appointments.status = 'no_show'` — 6.4 `mark_appointment_no_show` 가 이미 기록)
- **When** 환자별 노쇼 횟수를 조회하면
- **Then** 해당 환자의 `status='no_show'` 예약 수가 정확히 집계된다(단일 진실 함수 `public.patient_no_show_count`).

**AC2 — 임계치 초과 시 신규 예약 제한 + 사유 안내**
- **Given** 노쇼 횟수가 임계치(기본 2회)를 **초과**한(= `count > 2`, 즉 3회 이상) 환자에 대해
- **When** 신규 예약을 시도하면(원무 대리 `POST /scheduling/appointments` **또는** 환자 본인 `POST /scheduling/me/appointments`)
- **Then** 예약이 생성되지 않고 **409 `no_show_threshold_exceeded`** 와 사유(노쇼 횟수·임계치)가 안내된다.

**AC3 — 경계·비대상 보존(회귀 가드)**
- **Given** 노쇼 횟수가 정확히 임계치와 같은(2회) 환자
- **When** 신규 예약을 시도하면
- **Then** 예약이 **정상 생성**된다(`초과` = 엄격히 `>`, 임계치 도달은 허용).
- **And** 기존 예약의 **변경(reschedule)·도착접수(check-in)** 는 임계치 초과여도 차단되지 않는다(신규 예약 아님).
- **And** 기존 예약 흐름(더블부킹 409·과거 422·슬롯-윈도우 422·미존재 404·비활성 422·403 권한)은 **무변경**으로 보존된다.

**AC4 — UX: 사유 표면화(직원·환자)**
- **Given** 임계 초과 환자를 대상으로 예약을 시도하거나(직원 booking-peek·환자 앱), 직원이 환자를 선택했을 때
- **When** 제한이 적용되면
- **Then** 음영 비의존 인라인 경고(직원=경고 칩·노쇼 색 `status-received`, 환자=쉬운 말 안내)로 사유가 표시되고 예약 버튼은 막힌다.

## Dev Notes

### 핵심 설계 결정 (반드시 따를 것)

1. **노쇼 카운트 = 파생(derived), 비정규화 컬럼 없음.** `patients` 에 `no_show_count` 컬럼을 추가하지 **말 것**. `appointments.status='no_show'` 가 단일 진실(Option A — 예약 생명주기는 appointment.status 가 소유, 6.4 확정). 카운트는 SQL 함수 `public.patient_no_show_count` 로 파생 → 드리프트(카운터 동기화 버그) 구조적 차단. **AC1 의 "기록"은 6.4 의 no_show 전이가 이미 충족** — 6.7 은 그것을 **집계·강제**한다.
   - ⚠️ **encounters.no_show 는 세지 않는다.** 노쇼 슬롯 낭비는 *예약(reservation)* 미방문 문제다. walk-in 내원 no_show 는 예약 슬롯을 점유하지 않았으므로 카운트 대상이 아니다(이중 집계·비-예약 사건 혼입 방지). `appointments` 만 집계.

2. **임계치 = 앱 상수 `NO_SHOW_THRESHOLD = 2`(db.py).** 클리닉 설정 테이블이 아직 없어 단일 값을 위해 테이블을 만드는 건 과설계. 상수로 두되 명명·주석으로 튜너블 의도 표기. DB 는 임계치를 모른다(카운트만 소유) — 임계 판단은 앱(가드 + read 엔드포인트의 `blocked`). **web 은 임계치를 하드코딩하지 말 것**(read 엔드포인트의 `threshold`·`blocked` 사용).

3. **"초과" = 엄격히 `count > threshold`.** 임계치 2 → 0·1·2회 허용, **3회 이상 차단**. 에픽·FR-015 의 "초과"(exceed)를 문자 그대로 해석. (대안 `>=`[2회부터 차단]은 Clarifications 참조 — 의도적으로 `>` 채택.)

4. **강제 위치 = `db.py` 가드(트랜잭션 내부).** 카운트 조회 → 판정 → INSERT 가 **동일 트랜잭션**이어야 TOCTOU 안전(`_require_appointment_create` 와 동일 사상). 따라서 service 가 아니라 `insert_appointment`/`insert_self_appointment` 의 `_op` 안. 더블부킹(DB EXCLUDE)·슬롯-윈도우(service `_assert_slot_bookable`)와 **다른 층**임에 주의: 노쇼 제한은 *소프트 정책*(5.2 중복경고·5.5 알레르기 교차검증처럼 오케스트레이션 층)이라 DB 트리거가 아니라 쓰기 경로 가드로 강제. (0033 이 INSERT-가드 트리거를 의도적으로 배제한 결정과도 정합.)

5. **HTTP 409 `no_show_threshold_exceeded`.** 요청 자체는 유효하나 환자의 노쇼 이력(상태)과 **충돌**하여 생성 불가 → 409(`double_booking`·`no_self_patient` 와 동류·에러봉투 규약). detail = `{patient_id, no_show_count, threshold}`. (422 도 방어 가능 — Clarifications 참조.)

6. **차단 대상 = 신규 예약 2경로만**(`POST /scheduling/appointments` 원무 대리 + `POST /scheduling/me/appointments` 환자 본인). **reschedule·check-in 은 비대상**(기존 예약 이동/도착 — "신규 예약" 아님). **오버라이드(원무 강제 예약) 없음**(AC 미요구 — 의도적 스코프 제외, Clarifications 참조).

## Clarifications (구현 후 확인 — 차단 아님, 합리적 기본값 채택)

1. **임계치 비교 `>` vs `>=`**: 본 스토리는 "초과"(exceed)를 문자 그대로 `count > 2`(3회째 차단) 채택. 만약 "2회 노쇼 시 즉시 제한"(`>= 2`, 2회째 차단) 의도라면 `_assert_no_show_under_threshold` 의 비교와 `blocked` 만 `>=` 로 바꾸면 됨(테스트 경계도 함께). — **기본값: `>`**.
2. **HTTP 409 vs 422**: 노쇼 제한을 "상태 충돌"(409·`double_booking` 동류)로 봤으나, "이 요청은 충족 불가"(422·`slot_unavailable` 동류) 해석도 가능. — **기본값: 409**.
3. **원무 오버라이드**: 실제 병원은 원무가 사유 입력 후 강제 예약하는 경우가 있으나 AC 미요구라 **제외**(차단만). 필요 시 별도 권한(`appointment.override_no_show`)·force 플래그로 후속 가능.
4. **롤링 윈도우**: 현재 **누적(lifetime)** 카운트. 현실 시스템은 "최근 6개월" 등 윈도우를 두기도 함. — **기본값: 누적**(에픽 "환자별 노쇼 횟수" 그대로). 윈도우는 함수 시그니처 확장으로 후속 가능.
5. **환자 앱 프로액티브 차단**: 환자 앱은 **확정 시점 차단**(409→안내)만 구현. 슬롯 선택 전 사전 안내(`/me/no-show-status` 같은 self 엔드포인트)는 스코프 밖(직원 booking-peek 배지만 프로액티브). 필요 시 후속.
