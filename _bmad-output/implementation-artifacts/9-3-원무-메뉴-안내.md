# Story 9.3: 원무(reception) 메뉴 안내

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 원무 직원,
I want 내가 쓰는 메뉴(대기 현황·접수·예약 관리·리마인더·환자 등록·환자 검색·수납·수납 내역)의 사용법을 실제 화면·번호 하이라이트·단계 설명으로 보기를,
so that 선배 없이도 접수→대기→호출, 수납→영수증·재출력 같은 핵심 업무를 정확히 처리한다.

## Acceptance Criteria

1. **(FR-253) 원무 8개 메뉴 전수 수록.** 원무 계정으로 `/help`에 들어가면 사이드바 "운영/환자/정산" 섹션에 보이는 **8개 메뉴 전부**가 안내 섹션으로 제공된다: **대기 현황 `/reception/waiting`·접수 `/reception/intake`·예약 관리 `/reception/schedule`·리마인더 `/reception/reminders`·환자 등록 `/reception/register`·환자 검색 `/patients`·수납 `/reception/billing`·수납 내역 `/reception/billing/history`**. 빠진 메뉴 없이 스크롤만으로 원무 전 기능을 파악할 수 있다. "준비 중" 플레이스홀더로 남는 원무 메뉴가 없어야 한다.

2. **(FR-252) 각 메뉴 = 실제 화면 스크린샷 + 번호 하이라이트 + 단계 설명.** 각 메뉴 안내는 (1) 실제 캡처된 화면 PNG, (2) 그 화면 핵심 동작요소에 입혀진 번호 원(①②③…), (3) 번호별 "요소·무엇을·어떻게" 설명표(`hotspots`), (4) 작업 흐름(`flow`)로 구성된다. 번호는 캡처 파이프라인(9.2)이 이미지에 구워 넣고, 페이지는 이미지 + 번호표를 병기한다. **캡처 스펙의 번호(n) = help-content `hotspots[].num`** 1:1 일치.

3. **핵심 흐름 설명.** 다음 두 흐름이 관련 메뉴의 `flow`/`intro`에 명시된다: **(a) 접수 → 대기 → 호출**(접수 화면 walk-in 등록 → 대기 현황에 registered 표시 → 호출), **(b) 수납 → 영수증·재출력**(수납 워크리스트 → 집계·결제(정산/선수납) → 완료 후 영수증·세부내역서·처방전 출력 → 수납 내역에서 "영수증 보기"로 재출력).

4. **(9.2 파이프라인 소비) 캡처 자동 생성·정적 임베드.** 모든 원무 화면 이미지는 **`tools/screenshots/specs.mjs`에 reception 스펙을 추가하고 `node capture.mjs reception`을 실행해** 생성하며, `web/public/help/reception/<screen>.png`에 배치되어 도움말이 정적 임베드한다(런타임 캡처 0, 외부 편집 도구 0). 캡처는 더미 데이터가 적용된 환경 대상(데이터 0 화면 없음). 의사(9.1) 콘텐츠·이미지는 무변경.

5. **회귀 무파괴·범위 한정.** 변경 = `web/src/lib/help/help-content.ts`(reception 가이드 데이터 추가)·`tools/screenshots/specs.mjs`(reception 캡처 스펙 추가)·`web/public/help/reception/*.png`(생성 이미지) + (선택) help-content 테스트. **앱 화면(`web/src/app`·`web/src/components`의 원무 기능 코드)·API·마이그레이션·권한·생성 TS 타입은 변경하지 않는다.** `npm run build`/lint/타입체크 통과. 기존 도움말 셸(9.1)·의사 콘텐츠·사이드바·라우팅 회귀 0.

> **범위 경계 (반드시 지킬 것):** 이 스토리는 **원무(reception) 역할의 콘텐츠 채우기**까지다.
> - 9.3 = reception help-content 가이드(8개 menu href) + reception 캡처 스펙(specs.mjs) + reception PNG 생성.
> - **9.3 아님:** 의사(9.4)·간호(9.5)·방사선사(9.6)·관리자(9.7)·환자(9.8) 콘텐츠. 캡처 파이프라인/lib/러너 자체 수정(9.2 완료 — 그대로 사용). 앱 기능 코드 수정(원무 화면을 캡처가 잘 되게 바꾸고 싶어도 금지 — 캡처 스펙으로 해결).

### 컨텍스트 요약 — 무엇을 만드나
9.3은 **9.2 파이프라인의 첫 역할 콘텐츠 소비처**다. 세 가지를 만든다: (1) `help-content.ts`에 원무 8개 메뉴 가이드 데이터(번호·설명·흐름), (2) `specs.mjs`에 원무 캡처 스펙(화면 도달 + 번호 locator), (3) `node capture.mjs reception`으로 생성한 `web/public/help/reception/*.png`. 페이지·파이프라인·lib은 **이미 완성**(9.1·9.2) — 데이터/스펙/이미지만 채우면 도움말이 자동 렌더한다. 이 스토리가 9.4~9.8의 작업 패턴(역할별 동일 3종 세트)을 확립한다.

### ⚠️ 비자명·실수 방지 (이 스토리에서 LLM이 틀리기 쉬운 것)
1. **앱 코드를 고치지 마라.** 원무 화면이 캡처하기 까다로워도(검색 입력·진료과 선택 필요 등) **앱을 바꾸지 말고 스펙 `goto`로 상태를 만든다.** 변경 표면 = help-content + specs + public 이미지 뿐(+선택 테스트). web/src/app·components의 원무 기능 코드 0.
2. **번호 정합(n ↔ hotspots[].num).** 캡처 스펙의 `n`과 help-content의 `hotspots[].num`이 어긋나면 설명이 엉뚱한 번호를 가리킨다. 한 화면에서 둘을 같은 번호·순서로.
3. **상태 의존 화면 — 캡처 전 상태를 만들어야 안 빈다.** ⚠️ 예약 관리(진료과 선택 전 그리드 미표시)·리마인더(실행 전 로그 빈칸)·환자 검색(입력 전 결과 0)·접수(검색·선택 전 빈 폼)·수납 상세(행 클릭으로 진입·상태별 화면 상이). 맵의 "도달/상태 준비"를 goto에 구현.
4. **viewport 좌표 함정(9.2와 동일).** `annotate()`는 viewport(1440×900) 기준 `position:fixed`. 폴드 아래 요소는 어긋나거나 누락. **수납 상세만 세로로 김** — 결제·완료 버튼/문서 미리보기가 폴드 아래 가능 → goto에서 스크롤 후 좌표화하거나 in-viewport 요소만 번호화. 나머지 화면은 `max-w-*` 컨테이너로 짧아 폴드 위.
5. **클라우드 캡처는 클라우드 Supabase에 시드 선행.** 러너 기본 BASE=클라우드. demo_seed를 클라우드 DB에 적용해야 화면에 데이터가 보인다(로컬 DB 아님). 오늘 KST 상대 데이터라 캡처 직전 재적용 권장.
6. **PII — 원무 화면엔 reveal 없음, 마스킹만.** 환자 등록/검색/수납 어디에도 평문 주민번호·reveal 버튼이 없다(마스킹 `resident_no_masked`만). 하이라이트는 "마스킹 표시" 자체를 정책 시연으로 지목 가능. **캡처에 평문 PII가 찍힐 일은 없으나**, 검색어(이름)는 PII이므로 흔한 성("김") 정도만 — 설명에 실제 환자 식별정보를 박지 말 것.
7. **약한 locator는 라이브 검증.** 맵에서 getByRole(name)으로 안정 추출되는 건 그대로. "확인 필요"·data속성·텍스트 폴백 표기된 건 dev가 캡처 돌려보며 확정(러너가 못 찾으면 번호 누락 + 경고 로그). 깨지면 실제 DOM 기준으로 locator 보정.
