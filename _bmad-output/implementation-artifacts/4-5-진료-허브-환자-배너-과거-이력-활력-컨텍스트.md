# Story 4.5: 진료 허브 — 환자 배너 · 과거 이력 · 활력 컨텍스트

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a **의사(doctor)**,
I want **진료 허브(진행중 내원)의 상시 환자 배너와 좌측 컨텍스트 패널(읽기전용)에서 환자 신원·임상 프로필·과거 내원 이력·활력 컨텍스트를 한 화면에서 보고, 필요 시 주민번호·연락처를 권한 게이트+감사로 reveal 하며, 알레르기 등 안전 경고를 can't-miss 로 항상 보기를**,
so that **탭 이동·과거 차트 뒤지기 없이 안전하게 진료를 시작하고(임상 안전), 민감정보 접근은 누가·언제·무엇을 봤는지 추적 가능하게(감사) 통제된다.**

## Acceptance Criteria

1. **AC1 — 환자 배너 + 좌 컨텍스트 패널(읽기전용) (FR-005·FR-031·FR-032, UX-DR 3-pane):** 진료 허브(진행중 `in_progress` 내원)가 열리면 **상시 환자 배너**(이름·차트번호·나이/성별[birth_date·sex 파생]·혈액형·보험구분·진행중 상태 pill + 진료 경과[`consult_started_at`])가 3-pane 위에 노출되고, **좌 컨텍스트 패널(~280px, 읽기전용)** 에 ① **임상 프로필**(혈액형·알레르기·기저질환·복용약·특이사항 — 0009/3.2 데이터) ② **과거 내원 이력 타임라인**(이 환자의 과거 내원: 일자·진료과·담당의·상태, 최근순) ③ **활력징후** 섹션이 표시된다. 배너 데이터는 `encounter.patient_id` → 기존 `GET /patients/{id}`(마스킹 컬럼 + 임상 5필드), 타임라인은 **신규 `GET /patients/{id}/encounters`** 로 로드한다. ⚠️ **데이터 현실(반드시 준수)**: 간호 **활력 테이블은 미구축**(Epic 5 / 5.6 소유, `vital.record` 권한만 시드됨)·**과거 진단/처방/검사도 미구축**(4.7·Epic 5) → 해당 섹션은 **소유 스토리를 명시한 빈-상태(empty state)** 로 렌더한다(은폐 아닌 명시 placeholder — 4.4 셸 선례). 중앙(SOAP=4.6)·우(오더=Epic 5) pane 은 placeholder 유지.

2. **AC2 — 주민번호·연락처 reveal (권한 게이트 + 감사) (FR-241·FR-242, UX-DR9·UX-DR22):** 배너에서 **주민번호는 기본 마스킹**(`710314-2******` — `resident_no_masked`), **연락처는 기본 마스킹**(클라이언트 마스킹 `010-****-3082`). **"표시"(눈 아이콘 + "감사기록" 라벨)** 버튼을 누르면 ① 주민번호 → `POST /api/v1/patients/{id}/reveal-rrn`(신규 `reveal_rrn` RPC — `has_permission('patient.reveal_rrn')` 동일-txn 재평가 → `decrypt_sensitive` 가 복호 + `read` 이벤트 자가-감사 → full RRN 반환), ② 연락처 → `POST /api/v1/patients/{id}/reveal-contact`(신규 `reveal_contact` RPC + **신규 권한 `patient.reveal_contact`** — `has_permission` 재평가 → `audit_logs` `read` 자가-기록 → full phone/address/email 반환). 권한 미보유 = **403**, 미존재 환자 = **404**. reveal 로 값이 출현하면 polite 라이브 리전으로 낭독되고, "표시" 컨트롤 접근가능명에 "조회 시 감사 기록됨"이 포함된다(UX-DR20). **raw 주민번호/연락처는 로그·토스트·에러봉투·URL·실시간에 절대 미노출.**

3. **AC3 — 알레르기 can't-miss 안전 경고 (FR-005, UX-DR10·UX-DR21):** 환자에게 알레르기(`patients.allergies` 가 비어있지 않음)가 있으면 배너 상단에 **상시** `role="alert"` / `aria-live="assertive"` 경고가 노출된다 — **음영 비의존**(색 채움 + danger 테두리 + danger 채움 글리프 + 굵은 "환자 안전 경고" 라벨), 전체 알레르기 텍스트가 **truncation/"더보기" 없이 전부 노출**(critical 은닉 금지). 색 단독 금지(색 + 글리프 + 굵기 중복 인코딩). 알레르기 미기재면 경고 미노출(빈 경고 금지). ⚠️ 약물 상호작용(와파린 병용 등) 교차검증 경고는 **Epic 5(5.5 오더 패널 교차검증)** — 4.5 범위 밖(`allergies` 자유텍스트만 노출).

4. **AC4 — doctor 권한 활성화 + 회귀 0 (RBAC, NFR-051):** doctor 역할이 시드에서 **`patient.read` + `patient.reveal_rrn` + `patient.reveal_contact`** 를 보유해 진료 허브 배너·임상 프로필·과거 이력·reveal 골든 패스가 가동된다(4.4 까지 doctor 는 patient 권한 0 이었음). 'doctor = patient 권한 0' baseline 에 의존하던 **기존 테스트는 무권한 baseline 계정(`nurse@pms.local`)으로 이관**되고(회귀 3건), 신규 권한 `patient.reveal_contact` 는 **0012 admin boot grant** + 웹 `SENSITIVE_PERMISSIONS` 등록으로 카탈로그/매트릭스 정합을 유지해 **회귀 0** 을 보장한다. reveal 성공·권한거부(nurse)·미존재 경로가 신규 테스트로 커버된다.

1. **데이터 현실 우선 — 빈-상태는 거짓말이 아니다(스코프 무결성).** 진료 허브 mockup(`key-encounter-hub.html`)은 활력 스파크라인·과거 진단/처방 타임라인이 **꽉 찬** 화면을 보여주지만, **4.5 시점 DB 에는 그 데이터 소스가 없다**: 간호 활력 테이블=Epic 5(5.6, `vital.record` 권한만 시드)·encounter_diagnoses=4.7·SOAP medical_records=4.6·처방/검사 오더=Epic 5 전부 미구축(마이그레이션 0001~0011 에 부재 — 0012 가 본 스토리 첫 신규). 따라서 좌패널은 **실재 데이터(임상 프로필 0009/3.2·과거 내원 0010)는 채우고, 미구축 소스(활력·진단·처방)는 소유 스토리를 명시한 빈-상태로 렌더**한다. 이는 4.4 셸의 "명시 placeholder(은폐 아님)" 선례와 동형이며, **가짜 활력/진단 데이터를 임의 생성하는 것(완료 거짓)·5.6 활력 테이블을 미리 만드는 것(스코프 침범·마이그레이션 소유권 충돌)을 둘 다 금지**한다. FR-032("사전 입력 데이터 확인")는 데이터 소스가 생기면 같은 섹션이 채워지도록 **구조를 먼저** 세우는 것으로 충족(NFR-060 컨텍스트 확장성 정신).

2. **reveal = 신규 RPC 2종(`reveal_rrn`/`reveal_contact`) — DB 가 권한·감사를 동시 강제.** `patient.reveal_rrn` 권한·`decrypt_sensitive`(복호 자가-감사) 프리미티브는 0002/0005 에 이미 있으나 **소비처가 없었다**(patients.py:7 "reveal 엔드포인트는 첫 노출처 3.3/Epic 4 — 범위 밖"). 4.5 가 **최초 소비처**다. 메커니즘은 **SECURITY DEFINER RPC**: ① `resident_no_enc` 는 컬럼 GRANT 에서 authenticated 제외(0009:82) → definer(owner) 만 읽음 ② RPC 안에서 `has_permission` 재평가(동일-txn TOCTOU + 방어심층, FastAPI `require_permission` 게이트는 1차선) ③ 감사는 DB 가 기록(RRN=`decrypt_sensitive` 자동·연락처=수동 'read' insert) → **우회 불가**("복호=감사"를 평문 연락처로 확장). **대안(기각)**: FastAPI 가 enc 를 직접 fetch+`decrypt_sensitive` 호출(권한 재평가·감사 책임이 앱↔DB 로 분산 — RPC 가 단일 권위로 더 정석). `_run_authed` 는 `set role` 없이 GUC 만 주입(연결=service_role/owner, RLS 우회·enc 접근 가능, db.py:106~122) → RPC 가 definer 로 작동.

3. **연락처 reveal = 평문 + 자가-감사 RPC(암호화 마이그레이션 아님).** 연락처(phone/address/email)는 **평문 저장·평문 반환**이 현 posture(0009:35~38·patients.py 응답 — "reveal 게이트 대상 아님, patient.read 로 조회"). UX-DR22("모든 PII reveal 일관 게이트")를 만족시키는 두 길 중 **자가-감사 RPC**(평문 유지 + reveal 시 'read' 감사 + 신규 `patient.reveal_contact` 게이트)를 택한다. **대안(기각): 연락처 컬럼 암호화 마이그레이션** — patient-detail(원무 평문 표시)·전역 검색(3.5 연락처 검색)·환자 포털 본인 조회를 전부 깨뜨리는 교차절단 변경(고위험·범위 폭발). ⚠️ **알려진 한계(명시 이월)**: `GET /patients/{id}` 는 **여전히 평문 연락처를 페이로드로 반환**한다(앱 전역 기존 posture) → 4.5 의 허브 배너는 연락처를 **클라이언트 마스킹**(`maskPhone`)으로 가리고 reveal 시 감사된 full 값을 받지만, "평문이 reveal 전에도 와이어를 건넘"은 남는다. **서버측 연락처 마스킹(GET /patients/{id} 응답에서 연락처를 마스킹·reveal 로만 full)** 은 patient-detail·검색을 함께 손대는 교차절단 하드닝 → deferred(연락처 PII reveal 일관화, deferred-work). 4.5 는 **게이트·권한·감사 RPC·배너 마스킹**까지 — 의사 허브 표면에 UX-DR22 를 적용하는 첫 소비처.

4. **doctor patient 권한 활성화 + 무권한 baseline 이관(nurse, 4.4 패턴 재현).** 허브 배너는 `GET /patients/{id}`(=`patient.read`)를 호출하는데 **doctor 는 patient 권한이 0**이었다(시드는 encounter.read/start 만, seed.sql:113~118). 4.5 가 doctor 에 `patient.read`+reveal 2종을 grant 한다. **부작용**: admin·doctor 둘 다 patient.read 보유 → "patient.read 미보유 403" 검증 계정 소실 → **무권한 baseline 을 `nurse@pms.local`(4.4 신설)로 이관**(회귀 3건: list/search forbidden·rls_self). 4.4 가 encounter 에서 한 것과 **정확히 동형**. **새 권한 `patient.reveal_contact` 는 카탈로그 +1** → 0012 가 **admin boot grant 를 반드시 재실행**(0002 cross-join 은 후행 권한 미포함 — 누락 시 `test_admin_role_has_all_permissions` 회귀)하고, 웹 `SENSITIVE_PERMISSIONS` 에 등록(매트릭스 정합).

5. **과거 내원 이력 = 신규 `GET /patients/{id}/encounters`(대기판 조인 재사용·다른 접근 패턴).** 기존 `GET /encounters` 는 **department_id 필수**(대기판 — 진료과×일자, db.py:1602)라 "이 환자의 전 진료과 과거 내원"에 안 맞는다. 신규 sub-resource 엔드포인트가 `_ENCOUNTER_LIST_COLUMNS` 조인(denormalized 진료과·담당의명, db.py:1472)을 **재사용**하되 `where patient_id` + **최근순** 정렬. `encounter.read` 게이트(데이터=encounters, doctor 4.4 보유). 진단/처방 per-visit 부착은 4.7/Epic5(이력 항목 빈-상태 주석).
