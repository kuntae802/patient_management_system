# Story 6.8: 휴진 시 영향 예약 재배정

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 관리자·원무,
I want 의사 휴진을 등록할 때 그 기간에 걸린 기존(확정) 예약을 한눈에 보고 다른 슬롯으로 재배정하거나 환자에게 안내(취소·통지)를 기록하기를,
so that 휴진으로 진료를 못 받는 환자가 빈손으로 내원하지 않고, 휴진 슬롯에 남은 예약이 캘린더에서 고아로 떠도는 것을 막는다.

> FR-016: 의사 휴진 등록 시, 영향받는 예약을 표시하고 재배정·안내를 지원한다.
> FR-221: 관리자는 휴진·예외(휴가·학회 등)를 등록하며, 이는 예약 가능 슬롯(FR-012)·휴진 재배정(FR-016)의 근거가 된다.
> UX-DR15: "휴진 등록 시 영향 예약 표시·재배정 prompt."(예약 캘린더 정책의 일부)

**Epic 6 캡스톤(마지막 스토리).** 6.1 휴진(`doctor_time_offs`)·6.3 예약(`appointments`)·6.4 변경/취소(`reschedule`/`cancel`)·6.6 알림 이음매(`notification_logs`)를 한 흐름으로 묶는다. **신규 핵심 능력 = 영향 예약 조회 1건뿐** — 재배정은 6.4 `reschedule` 재사용, 안내 기록은 6.6 `notification_logs` 이음매 확장(사용자 확정: "6.6 알림 이음매 확장").

## Acceptance Criteria

**AC1 — 영향 예약 조회(FR-016·핵심 신규 능력)**
- **Given** 의사 휴진 기간(`doctor_id` + `[start_at, end_at)`)이 주어질 때
- **When** 영향 예약을 조회하면(`GET /scheduling/affected-appointments?doctor_id=&start_at=&end_at=`)
- **Then** 그 의사의 **`status='booked'`** 예약 중 휴진 기간과 **겹치는 것만**(반열림 `[)` 오버랩) 환자명·시각과 함께 반환된다. `cancelled`·`no_show`·`completed` 은 제외(슬롯 미점유·이미 종결).
- **And** 게이트 `appointment.read` → 미보유 시 **403**. 겹치는 예약 0건이면 **빈 배열**(404 아님). 날짜 파싱 실패 → 422.

**AC2 — 휴진 등록 후 영향 예약 표면화(UX-DR15)**
- **Given** 관리자가 근무 스케줄 화면에서 휴진·예외를 **등록(저장)**했을 때
- **When** 그 기간에 확정(booked) 예약이 1건 이상 있으면
- **Then** 영향 예약 목록(환자명·예약 시각·`[재배정]`·`[취소·안내]` 액션)이 **패널(슬라이드오버)**로 표시된다(UX-DR15 "휴진 등록 시 영향 예약 표시·재배정 prompt"). 0건이면 패널을 띄우지 않고 기존 토스트만.
- **And** 비활성된 휴진 행에서도 **"영향 예약"** 액션으로 패널을 다시 열 수 있다(재방문).
- **And** 음영 비의존(라벨+글리프+색)·키보드 조작·PII 경계(환자명만·주민번호/연락처 미노출) 준수.

**AC3 — 재배정: 다른 슬롯으로 이동 + 안내 기록(AC2의 "다른 슬롯으로 이동")**
- **Given** 영향 예약에 대해
- **When** 같은 진료과의 가용 슬롯(같은 의사 다른 날짜 **또는** 같은 진료과 다른 의사)으로 재배정하면(6.4 `POST …/appointments/{id}/reschedule` 재사용)
- **Then** 예약이 새 슬롯으로 이동하고(`booked` 유지), 기존 보장(더블부킹 409 `double_booking`·슬롯-윈도우 422 `slot_unavailable`·과거 422·잘못된 전이 409·미존재 404)이 **그대로 보존**된다.
- **And** **다른 의사로 재배정 시** `appointment.department_id` 가 새 의사의 진료과로 **동기화**된다(부서-스코프 캘린더에서 고아 방지 — deferred 항목 "reschedule 가 다른 진료과 의사로 변경 시 department_id 미동기화" 청산). 같은 의사 재배정은 `department_id` **불변**(다중 진료과 의사 회귀 방지).
- **And** 재배정 성공 후 환자 **재배정 안내**(`reschedule_notice`)를 1건 기록한다(AC4 메커니즘 공유).

**AC4 — 취소·안내: 환자 안내 기록(AC2의 "환자 안내가 기록된다" — 6.6 이음매 확장)**
- **Given** 가용 슬롯이 없는 등 재배정이 불가한 영향 예약에 대해
- **When** "취소·안내"를 수행하면
- **Then** 예약이 취소되고(6.4 `cancel` 재사용·`booked→cancelled`·사유 "의사 휴진") **환자 취소 안내**(`cancellation_notice`)가 `notification_logs` 에 **시뮬 기록**된다(`POST …/appointments/{id}/notify-change`).
- **And** 안내 기록은 6.6 규약을 그대로 따른다: **마스킹 수신처**(`recipient_masked`)·**비-식별 body**(환자명·주민번호·연락처 금지)·연락처 없으면 `status='skipped'`(`no_recipient`)·**멱등**(`UNIQUE(appointment_id, reminder_kind)` → 재실행 새 행 0)·게이트 `notification.send` → 403.

## Dev Notes

### 핵심 설계 결정 (반드시 따를 것)

1. **신규 핵심 능력 = "영향 예약 조회" 1건뿐.** 6.8 의 새 도메인 로직은 `fetch_affected_appointments`(그 의사·booked·휴진 윈도우 겹침) 하나다. **재배정 = 6.4 `reschedule_appointment` 그대로**(다른 의사 허용 위해 department 동기화만 보강)·**안내 = 6.6 `insert_notification_log` 그대로**(reminder_kind 값만 확장). 바퀴를 다시 만들지 말 것 — 6.8 은 기존 조각을 "휴진" 맥락으로 **묶고 표면화**하는 캡스톤이다.

2. **"안내 기록" = 6.6 `notification_logs` 이음매 확장(사용자 확정).** `reminder_kind` CHECK 에 `reschedule_notice`(재배정 통지)·`cancellation_notice`(취소 통지) 2값 추가(0037). 6.6 규약 전부 계승: 마스킹 수신처·비-식별 body·연락처 없으면 skipped·멱등 `UNIQUE(appointment_id, reminder_kind)`·게이트 `notification.send`. **실 SMS 발송은 여전히 시뮬**(이음매)·환자에게 진짜 문자가 가는 건 운영 전환 시. 컬럼명이 `reminder_kind`인데 비-리마인더 통지를 담는 **네이밍 부채**는 인지하되 정식 리네임(`notification_kind`)은 범위 밖(주석 1줄로 표기).

3. **재배정 = 다른 의사 허용 → `department_id` 동기화로 deferred 청산.** 휴진의 본질은 "그 의사가 그 시간 진료 못 함"이므로 **같은 진료과 다른 의사**로의 재배정이 1급 시나리오다. 6.4 까지 web 은 같은 의사만 변경(`doctor_id` 고정)이라 deferred 항목(line 294)이 잠복했다. 6.8 이 cross-doctor 재배정을 실제 UI 경로로 열므로 **이 스토리가 청산을 소유**한다:
   - `reschedule_appointment` 에서 **새 doctor_id 가 기존과 다를 때만** `appointment.department_id` 를 새 의사의 home 진료과(`users.department_id`)로 동기화. **같은 의사면 미변경**(다중 진료과 의사가 시각만 옮길 때 home-dept 로 덮어쓰는 회귀 차단 — 미묘하지만 중요).
   - UI 의사 피커는 `fetch_bookable_doctors(department_id=영향예약의 진료과)` 로 **같은 진료과로 제한** → 정상 흐름에선 동기화가 사실상 no-op(새 의사 home dept == 그 진료과). cross-department 직접 API 만 동기화가 실효(고아 방지).
   - ⚠️ **잔여 이월**: "새 의사가 그 진료과를 실제 진료(doctor_schedules 멤버십)하는지" 서버 검증(line 302 `_assert_doctor_in_department`)은 **여전히 별개 이월** — 6.8 은 department_id 정합(고아 방지)만 보장하고, UI 가 같은 진료과 피커로 제한해 멤버십 위반은 미도달. 혼동 말 것.

4. **영향 = `status='booked'` ∩ 윈도우 겹침만.** `cancelled`·`no_show`·`completed` 은 슬롯을 점유하지 않거나 이미 종결 → 영향 아님(6.2 `fetch_booked_appointments_in_range`·6.4 캘린더 overlay 의 booked-only posture 정합). 겹침 = 반열림 `[)`(`scheduled_start < end_at and scheduled_end > start_at` — `fetch_doctor_time_offs_in_range` 미러).

5. **휴진 등록과 영향 조회는 분리(시간 결합 아님).** 0030 휴진 등록은 영향 예약을 **자동 취소·이동하지 않는다**(부수효과 없는 등록 = 6.1 설계). 6.8 은 등록 **후** 영향을 **조회·표면화**하고, 재배정/취소는 관리자가 **명시 액션**으로 수행. → 휴진 INSERT 와 영향 예약 사이 TOCTOU 무관(휴진은 이미 저장됨·영향 조회는 읽기). 캘린더는 휴진 슬롯 위에 booked 예약을 confirmed 로 overlay(6.4 `_build_doctor_column`)하므로 미처리 영향 예약은 **휴진 빗금 위 확정 칩**으로도 보인다(6.8 패널이 그걸 능동 해소).

6. **안내 기록은 best-effort·재배정/취소가 권위 액션.** web 흐름 = `reschedule`(또는 `cancel`) **성공 후** `notify-change` 호출. 안내 기록 실패(시뮬·네트워크)는 예약 변경을 되돌리지 않는다(이음매 철학). 멱등이라 재시도 안전. **reschedule_notice 는 재배정 후 호출**(예약 행의 새 scheduled_start 를 body 가 반영) — 순서 중요.

7. **게이트 정합.** 영향 조회=`appointment.read`(원무·관리자·캘린더 동일)·재배정/취소=`appointment.update`(6.4)·안내=`notification.send`(6.6). **신규 권한 0** → admin grant 재실행 불요·`test_admin_role_has_all_permissions` 무영향(6.7 posture). 6.8 의 주 surface 는 **관리자 근무 스케줄 화면**(휴진 등록=`master.manage` admin 전용) — admin 은 위 3권한 전부 보유(0031·0033·0035 admin grant). 원무의 ad-hoc 재배정은 6.4 캘린더 `BookingDetail` 이 이미 커버(6.8 은 휴진-트리거 일괄 흐름 추가).

1. **"안내 기록" 메커니즘**: 사용자 확정 = **6.6 알림 이음매 확장**(0037 — `reminder_kind` +2값·`notify-change` 엔드포인트·시뮬 SMS notice). 대안(린: cancel_reason 만·마이그 없음)은 채택 안 함.
2. **재배정 doctor 범위**: 기본 = **같은 진료과**(UI 의사 피커가 `bookable-doctors(department_id)` 로 제한). cross-department 재배정은 미요구(환자가 예약한 진료과를 임의 변경하는 건 비현실) → UI 미노출. 단 `reschedule` department_id 동기화는 직접-API cross-department 에도 정합 보장(고아 방지·line 294 청산). 새 의사의 진료과 멤버십 서버 검증(line 302)은 **잔여 이월**.
3. **안내 트리거 = 명시 액션 + best-effort**: 재배정/취소 성공 **후** web 이 `notify-change` 호출. 자동(서버 트랜잭션 결합)이 아니라 분리 호출 — 안내 실패가 예약 변경을 되돌리지 않음(이음매·멱등 재시도). 자동 통지·환자 수신함 UI 는 범위 밖(Epic 8/운영 전환).
4. **휴진 등록이 영향 예약을 자동 처리하지 않음**: 0030 휴진 INSERT 는 부수효과 없음(6.1 설계) — 6.8 은 등록 후 **표면화·수동 해소**만. 자동 일괄 취소/이동(정책)은 미요구·범위 밖.
5. **주 surface = 관리자 근무 스케줄**: 휴진 등록(`master.manage`)이 admin 전용이라 6.8 휴진-트리거 일괄 흐름도 admin. 원무의 개별 재배정은 6.4 캘린더 `BookingDetail` 이 이미 커버(원무용 affected 패널 surface 는 후속 여지).
6. **reminder_kind 네이밍 부채**: 컬럼이 리마인더+변경통지를 함께 담음. 정식 리네임(`notification_kind`)은 범위 밖(이월·주석 표기). 의미 구분은 값 어휘로.
