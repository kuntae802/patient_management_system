# Story 5.2: 처방 오더 발행 · 중복 경고

Status: ready-for-dev

## Story

As a 의사,
I want 약품 마스터에서 약을 선택해 처방전을 발행하고 진단과 연결하기를,
so that 표준 약품·용법으로 처방하고 근거가 남는다.

## Acceptance Criteria

(epics.md Epic 5 / Story 5.2 — FR-050·FR-051·FR-052)

1. **(FR-050) 처방전 발행** — 진료 허브 오더 패널에서 약품 마스터 검색 피커로 처방전을 발행하면, **처방전 헤더 + 처방상세 라인(약품·용량·횟수·일수·용법)**이 생성되고 free-text 약품 입력이 차단된다(약품 = 마스터 FK `drug_id`로만).
2. **(FR-051) 처방↔진단 연결** — 처방을 발행할 때 내원에 부착된 진단(주상병 A 등)과 연결하면 처방 근거가 `encounter_diagnosis_id`로 기록된다(연결은 선택).
3. **(FR-052) 동일 성분 중복 경고** — 동일 성분(`ingredient_code`)이 이미 처방된 상태에서 같은 성분을 추가하면 **인라인 중복 경고**가 표시된다(비차단 — 의사가 사유 판단 후 진행 가능; 하드 블록은 알레르기 교차검증 = 5.5).
4. **조회·추적성** — 한 내원의 발행된 처방전 목록을 조회하면 헤더(상태 `issued`·발행 의사·발행 시각)와 상세 라인(약품 코드·명칭·용량·용법)이 표시된다. 모든 처방 생성은 감사 트리거로 기록된다(FR-080 추적성).
5. **권한·경계** — 처방 발행 = `prescription.create`(의사), 조회 = `order.read`(의사·간호·방사선). 권한 미보유(원무 = 오더 baseline) → 403. 발행은 **service_role 직접 INSERT**(register/RPC 아님 — 자유 CRUD), 권한은 쓰기 직전 동일 트랜잭션 재평가(TOCTOU).

### ⚠️ 비자명 gotcha — 중첩 응답은 순수 dict 트리로 [Source: db.py:37 `_to_encounter` = model_validate(dict(row))]

`PrescriptionResponse.details: list[PrescriptionDetailResponse]` 는 중첩 모델. 기존 코드는 `Model.model_validate(dict(row))` 로 **asyncpg.Record → dict** 변환 후 검증한다(Record 는 `record['k']` 매핑은 되나 `record.k` 속성 접근 불가 → Pydantic from_attributes 의 getattr 가 중첩 Record 에서 실패). 따라서 db(또는 서비스)가 **순수 dict 트리**를 조립: `{**dict(header_row), "details": [dict(d) for d in detail_rows]}` → `PrescriptionResponse.model_validate(tree)`. 헤더·각 상세 모두 `dict()` 변환(중첩 Record 를 그대로 넘기지 말 것).

### ⚠️ 비자명 gotcha — dose `numeric` 은 Decimal 필수 [Source: 0015:63]

`prescription_details.dose` 는 `numeric`. **asyncpg 는 numeric 컬럼에 Python `float` 바인딩 시 DataError 를 던진다 — `Decimal` 을 요구**. Pydantic `PrescriptionDetailWrite.dose: float | None`(JSON number 수용)으로 받되, **db.py INSERT 직전 `Decimal(str(dose))` 변환**(`from decimal import Decimal`). 읽기는 asyncpg 가 Decimal 반환 → 응답 Pydantic `dose: float | None` 이 Decimal→float coerce → JSON number 로 깔끔 출력(amount_krw 는 int 라 무관 — dose 만 이 처리 필요). 금액(KRW 정수)과 달리 dose 는 비-화폐 → float 표현으로 충분(데모 용량 0.5/1/1.5/2 는 float-exact, DB numeric 이 store of record).

### 처방전 생성 모델 — 헤더 + N상세 원자적 1 POST (설계 결정)

- **처방전 = 한 번에 발행되는 단위**(처방전 발행 의미·1:N 스키마). 의사가 클라에서 드래프트(약품 라인들 + 라인별 용량/횟수/일수/용법 + 선택 진단 근거)를 구성 → "처방 발행" 1 POST 로 헤더 + 전 상세를 **단일 트랜잭션**에 생성. 라인별 개별 POST(4.7 진단의 select→즉시부착 "어더") 가 **아님** — 처방전은 라인 그룹을 함께 발행하는 게 현실적이고 헤더/상세 분리 스키마의 의도.
- `details` 최소 1 라인(Pydantic `min_length=1`) — 빈 처방전 무의미.
- 라인 파라미터(dose/frequency/duration_days/usage_instruction)는 **전부 nullable**(0015 CHECK 가 null 허용) — 최소 유효 라인 = `drug_id` 만. 의사가 선택적으로 채움.
- `encounter_diagnosis_id`(FR-051) = 헤더 레벨 1개(처방전 전체의 근거 진단). 라인별 진단 연결 아님.

### 중복 경고 FR-052 — 클라 측 `ingredient_code` 비교 (비차단) [Source: epics.md:925-927, 0007:52 drugs.ingredient_code, masters.ts:249-262 MasterPickerItem]

- **클라이언트 측 소프트 경고** — 서버는 차단하지 않는다(하드 블록 = 알레르기 교차검증 5.5; FR-052 는 정보성 인라인). `MasterPickerItem`(drug)·`PrescriptionDetail` 둘 다 `ingredient_code` 노출 → 라운드트립 없이 즉시 비교 가능.
- 후보 약품의 `ingredient_code` 를 **(기존 발행 처방의 활성 상세 ingredient_code) ∪ (현 드래프트 라인 ingredient_code)** 와 비교. 비-null·일치 → 인라인 경고. null ingredient_code(예: 시드의 코푸시럽·생리식염수) → 동일성 판정 불가 → 경고 없음.
- ⚠️ **데모 약품 17종은 ingredient_code 가 전부 고유**(seed.sql:283-302) → 서로 다른 약 2개로는 중복 데모 불가. **같은 약을 두 번 추가**하면 동일 ingredient_code → 경고 발동(실무에서 가장 흔한 중복 케이스 = 같은 약 재처방). 테스트·데모는 이 경로.
- 경고 인코딩 = 색+글리프+라벨 3중(음영 단독 금지, UX-DR20) + `role="alert"`/aria-live(AT 낭독, UX-DR18). 발행 버튼은 경고 중에도 활성(비차단).
