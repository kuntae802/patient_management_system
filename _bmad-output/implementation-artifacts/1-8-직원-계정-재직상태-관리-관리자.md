# Story 1.8: 직원 계정 · 재직상태 관리 (관리자)

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a **관리자**,
I want **직원 계정을 생성하고 역할·소속·면허·재직상태를 관리하기를**,
so that **인사 변동(입사·휴직·퇴사)이 시스템 접근에 정확히 반영된다.**

> **에픽 맥락:** Epic 1 = 플랫폼 기반·신원·접근 통제. 이 스토리는 **신원 라이프사이클의 "쓰기 표면"** — 1.3이 `users` 스키마와 감사 트리거를 놓고, 1.4가 로그인 분기를, 1.5/1.6/1.7이 인증·UI 게이트·권한 매트릭스를 세웠다면, **1.8은 그 신원 자체를 만들고(입사) 무효화/복원(휴직·퇴사·복귀)** 한다. 핵심: ① **이 시스템 최초로 Supabase Auth Admin API(supabase-py)를 호출**해 auth 사용자를 프로비저닝하고(seed.sql의 DEV-ONLY raw SQL 경로를 대체), ② 1.7이 확립한 "동일 트랜잭션 권한평가+쓰기·0004 자동감사" 패턴을 **두 시스템(Supabase Auth ↔ Postgres)에 걸친 다단계 오케스트레이션**으로 확장한다(보상 트랜잭션). **재직상태 차단은 이미 1.3의 DB 헬퍼(`has_permission`/`auth_user_role`이 `employment_status='active'` 필터 내장)가 절반을 처리** — 이 스토리는 그 상태를 *쓰는* 표면 + 로그인 차단(auth ban)을 더한다. seed.sql:11-12가 이 스토리를 명시적으로 예고한다("실제 직원 계정 생성은 Story 1.8").

## Acceptance Criteria

> 출처: `_bmad-output/planning-artifacts/epics.md:491-509`(Story 1.8 BDD) · `epics.md:112-113,271-272`(FR-214·FR-215) · `0002_identity_rbac.sql:39-56`(users 스키마) · `0003_rls_helpers.sql:9-42`(active 필터 헬퍼).

**AC1 — 직원 계정 생성(Auth 사용자 + 프로필)**
- **Given** `user.manage` 권한을 가진 관리자가 `/admin/users`에서
- **When** 신규 직원(사번·이름·이메일·임시 비밀번호·역할·선택: 면허·전화·입사일)을 입력해 생성하면
- **Then** **Supabase Auth 사용자가 Admin API로 생성**(`email_confirm=true`)되고, 그 uid로 **`public.users` 프로필 행이 INSERT**된다(`employee_no`·`name`·`role_id`·`license_no`/`license_type`·`phone`·`hire_date`, `employment_status` 기본 `active`). 역할은 **직원 5역할(reception·doctor·nurse·radiologist·admin)** 만 허용하고 `patient`은 거부(422). `employee_no`/`email` 중복은 409로 거부된다. users INSERT는 **0004 `trg_users_audit`가 자동 감사**한다(actor=관리자). (FR-214)

**AC2 — 재직상태 변경 = 접근·로그인 차단/복원**
- **Given** 기존 직원에 대해
- **When** 재직상태를 **휴직(`on_leave`)/퇴사(`terminated`)** 로 변경하면
- **Then** ① `users.employment_status`가 갱신되어(자동 감사) **DB 헬퍼(`has_permission`/`auth_user_role`)가 그 직원의 역할·권한을 즉시 무효화**(모든 명령 403, 직원 셸 진입 차단=`requireStaff` 강등) → **접근 차단**, ② **Supabase Auth 사용자가 ban**되어(세션 revoke + 신규 로그인 차단) **로그인 차단**된다. **재직(`active`) 복귀 시** 상태 복원 + ban 해제로 접근·로그인이 모두 복원된다. (FR-215)

**AC3 — 모든 계정·재직 변경의 자동 감사**
- **Given** 계정 생성·재직상태 변경이 커밋될 때
- **When** `public.users` 행이 INSERT/UPDATE되면
- **Then** **0004 트리거**가 `audit_logs`에 **행위자(=관리자 uid, `app.actor_id` GUC)·대상(`users`/대상 uid)·동작(create/update)·변경 전/후 스냅샷**을 기록한다. **앱은 감사 INSERT를 직접 하지 않는다**(트리거가 단일 진실). raw 비밀번호는 어디에도(로그·감사·토스트·응답) 노출되지 않는다(`public.users`에 비밀번호 컬럼 없음).

**AC4 — 쓰기 권위 · 동일 트랜잭션 · 다단계 오케스트레이션(횡단)**
- **Given** 계정/재직 쓰기가 일어날 때
- **When** 권한·대상이 검증되면
- **Then** 모든 쓰기는 **반드시 FastAPI(service_role/postgres 풀) 경유**로만 수행되고(0002가 `authenticated`에 `users` SELECT만 grant — 클라/authenticated 직접 쓰기 불가), 엔드포인트는 `require_permission('user.manage')`로 403 게이트하며, DB 쓰기 직전 **같은 `authenticated_conn` 트랜잭션에서 `has_permission('user.manage')`를 재평가**(TOCTOU 차단, 1.7 패턴). **Auth 생성↔DB INSERT는 두 시스템에 걸쳐 비원자적**이므로 **보상(compensation)** — users INSERT 실패 시 방금 만든 Auth 사용자를 `delete_user`로 롤백해 고아 계정을 막는다. **자기 자신을 비활성(휴직/퇴사)으로 바꾸는 자가-락아웃은 서버가 거부**(409, 1.7 admin-lock과 동형). `service_role`/`secret` 키는 클라 번들에 절대 없다.

**AC5 — 관리 UI(목록·생성·상태변경) · 접근성**
- **Given** 관리자가 `/admin/users`에서(사이드바 "직원 계정" 항목은 1.2/1.6에서 이미 `user.manage` 게이트로 배선됨)
- **When** 화면을 열면
- **Then** **직원 목록**(사번·이름·역할·면허·재직상태 배지)이 표시되고(목록 데이터는 **FastAPI 경유** — `users` RLS가 본인 행만 허용하므로 관리자 직접조회 불가), **계정 추가** 폼(React Hook Form + Zod, Pydantic 거울)으로 생성하며, 각 직원의 **재직상태 변경**은 휴직/퇴사 시 **확인 다이얼로그**(접근 차단 경고, 1.7 `confirm-dialog` 재사용)를 거친다. 실패 시 한국어 봉투 메시지 토스트, 성공 시 목록 갱신. 시맨틱 `<table>`·네이티브 컨트롤·`:focus-visible`·라이브 리전을 지킨다(UX-DR20).

### 🔌 Auth 프로비저닝 이음매 — supabase-py 최초 사용 + 보상 트랜잭션 (가장 높은 위험)

⚠️ **왜 raw SQL이 아니라 Admin API인가:** `seed.sql:22-76`이 `auth.users`/`auth.identities`에 raw INSERT하는 DEV-ONLY 경로를 보여주지만, **GoTrue 토큰 text 컬럼을 `''`로 채우지 않으면 로그인 시 "Database error querying schema"(500)** 같은 함정이 있다(seed.sql:46-47). 프로덕션 계정 생성은 **Supabase Auth Admin API**(`auth.admin.create_user`)가 auth.users + auth.identities + 모든 GoTrue 내부를 정확히 처리한다. seed.sql:11-12가 명시: "실제 직원 계정 생성은 Story 1.8(관리자 UI)". **raw SQL INSERT 금지 — Admin API만.**

⚠️ **두 시스템에 걸친 비원자성:** Auth 사용자 생성(Supabase HTTP)과 `public.users` INSERT(Postgres 트랜잭션)는 **하나의 트랜잭션이 아니다**. 생성 순서는 강제됨 — `public.users.id`가 `auth.users.id`를 FK 참조하므로 **Auth 먼저 → uid 획득 → users INSERT**. INSERT가 실패하면(중복 employee_no 등) **고아 Auth 사용자**가 남는다. → **보상(compensation):**

```
create_staff(sub, payload):
  1) 앱 사전검증: role_code → role_id (staff role, patient 거부 422); license_type 정합
  2) uid = admin_create_user(email, password, email_confirm=True)   # Supabase HTTP
  3) try:
       insert_staff_profile(sub, uid=uid, ...)                      # authenticated_conn 트랜잭션, 자동감사
     except <모든 예외>:
       admin_delete_user(uid)                                       # 보상 — 고아 Auth 제거(users 부분행도 FK CASCADE)
       raise
  4) return StaffResponse  # email/비밀번호 미포함
```

- `employee_no` 중복은 DB unique 제약이 최종선(`UniqueViolationError` → 409 `employee_no_taken`). 보상이 Auth 고아를 정리.
- `email` 중복은 GoTrue가 2단계에서 거부 → 409 `email_taken`(보상 불요, users INSERT 미도달).

### 🔒 재직상태 = 이중 차단 (접근 + 로그인)

AC2 "로그인·접근이 차단" 두 단어를 모두 만족시킨다:

| 차단 면 | 메커니즘 | 권위 |
|---|---|---|
| **접근**(역할·권한·셸) | `users.employment_status` UPDATE → `has_permission`/`auth_user_role`이 active 아니면 역할 NULL·권한 0 → 모든 명령 403, `requireStaff`가 직원 셸에서 강등 | **DB(단일 진실)** |
| **로그인·세션** | Auth ban(`update_user_by_id(uid, {ban_duration:"876000h"})`) → refresh 토큰 revoke + 신규 로그인 차단. 복귀 시 `{ban_duration:"none"}` 해제 | **Supabase Auth(enforcement)** |

- **순서 = DB 먼저, ban 나중**(차단 방향 fail-safe): DB UPDATE가 권위·감사이므로 먼저 커밋. ban은 로그인 표면 enforcement. 앱이 ban 전에 죽어도 **DB 헬퍼가 이미 접근을 무효화**(기존 토큰도 명령 403·셸 강등) → 안전. ban 실패는 소프트 경고+로깅, 멱등 재시도. 복귀(→active)도 DB(접근 복원) → unban(로그인 복원) 순; unban 실패 시 직원은 여전히 로그인 불가(=안전한 방향).
- **설계 결정 — Option B(이중 차단, 권장) vs Option A(DB만):** Option A는 ban 없이 DB UPDATE만 — 접근은 완전히 무효화되나(명령 403·셸 강등) 휴직자가 *인증 자체*는 통과하고 기존 토큰이 exp까지 살아있다(무력하지만). AC가 "로그인 차단"을 명시하고 과제 목표가 "최대한 실제처럼"이므로 **Option B 권장**. ban 인프라가 마찰이면 Option A를 MVP 수용 + deferred(ban 추가)로 기록 가능 — **단 그 경우 AC2의 '로그인 차단'을 '접근 차단으로 실질 무력화'로 해석함을 명문화**. (착수 시 결정 — §Open Questions)
- **자가-락아웃 가드:** 관리자가 자신을 휴직/퇴사로 바꾸면 마지막 관리 경로가 끊긴다 → `user_id == sub AND status != 'active'` → 409 `self_lockout`(1.7 admin role-lock과 동형). last-admin(유일 active admin 퇴사) 가드는 카운트 필요 → **deferred**(자가-락아웃이 흔한 케이스 커버).

### 🌐 읽기 권위 — 목록은 왜 FastAPI인가

1.7은 매트릭스 읽기를 Supabase 직접 조회했다(roles/permissions/role_permissions가 `authenticated` SELECT `using(true)`, 0003:55-65). **그러나 `users`의 RLS는 본인 행만**:

```sql

create policy users_select_self on public.users
  for select to authenticated using (id = (select auth.uid()));
```

→ 관리자가 Supabase로 직접 조회해도 **자기 행 1개만** 보인다. 전 직원 목록은 **FastAPI(postgres 풀=RLS 우회) 경유**가 유일 경로. 이는 아키텍처의 "쓰기·복잡/민감 조회=FastAPI, 단순 조회=Supabase"(architecture.md:380-381)와 정합 — 관리자의 전-직원 조회는 권한 게이트된 관리 조회다. `GET /v1/admin/users`는 `require_permission('user.manage')`로 게이트. **users RLS 정책을 완화하는 마이그레이션을 추가하지 말 것**(본인행 제한은 환자/타직원 경계의 방어심층 — 유지). 목록 fetch는 클라 컴포넌트가 `apiFetch`로(RSC 서버 fetch 인프라는 YAGNI — 1.1 deferred `API_INTERNAL_URL`은 미도입 유지).
