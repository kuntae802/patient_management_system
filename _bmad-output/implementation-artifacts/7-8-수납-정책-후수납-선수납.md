# Story 7.8: 수납 정책 — 후수납 / 선수납

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

> ⚠️ **설계 결정 4건(사용자 확정 2026-06-24, AskUserQuestion)** — Dev Notes §설계 결정 참조.
> ① **선수납 진입점 = 수납 워크리스트 확장** — 워크리스트에 `registered`(접수 후·진찰 전) 내원도 포함(상태 칩 '접수=선수납 가능'/'진찰중=정산 대상'), 동일 `billing-detail` 화면이 상태별로 선수납 패널/정산 패널 노출. 결제 로직을 수납 영역에 일원화.
> ② **정책 범위 = 건별 운용** — 후수납이 기본(`payments.billing_type` default `'postpaid'`, 7.1 선언). 선결제를 받으면 그 건만 `billing_type='prepaid'`로 전환. **신규 설정 테이블 0**(기존 `billing_type` 컬럼이 곧 이월 갭 ④ 정책 플래그). 기관 차원 기본값 플래그 미채택.
> ③ **차액 정산 = 단일 누계 `paid_amount_krw`** — 선결제가 `paid_amount_krw`에 누적, finalize가 `paid = greatest(copay, paid)`로 정산. 차액(납부할 금액) = `copay − paid`는 영수증 `due_amount_krw`(7.5 인프라)가 이미 표시. **신규 컬럼 0**(7.1 "단일 헤더 + paid 누적·별도 행 아님" 의도 준수). 선납액 스냅샷 컬럼 미채택.
> ④ **과납(선납 > 최종 본인부담) = 표시·플래그 + 환급은 7.9 이월** — 차액 음수면 '환급 대상' 표시, finalize는 허용(`paid` 초과분 보존). 실제 환급 처리·취소/노쇼 환급은 7.9(취소·노쇼 정산). finalize 차단/즉시 환급 미채택.

## Story

As a 원무 직원,
I want 후수납(기본)과 선수납 옵션을 운용하기를,
so that 병원 정책에 맞게 정산 시점을 조정한다.

## Acceptance Criteria

**AC1 (FR-117 후수납 기본 — 무회귀):**
**Given** 수납 정책 플래그(`payments.billing_type`, default `'postpaid'`)가 있을 때
**When** 선결제 없이 진료 후 수납 화면에서 결제·완료하면
**Then** 7.4의 후수납 전액 정산 동작이 **변경 없이** 유지된다(`paid_amount_krw = copay_amount_krw`, draft→finalized, `complete_encounter`). billing_type은 `'postpaid'` 그대로.

**AC2 (선결제 RPC — `prepay_payment`):**
**Given** `registered`(접수 후·진찰 전) 또는 `in_progress`의 draft 수납 건에 대해
**When** 원무가 선결제 금액(>0)·결제수단을 기록하면
**Then** `prepay_payment(encounter_id, amount, method)` RPC가 한 트랜잭션에서 ① 헤더 upsert(없으면 생성·`registered`에서 수가 0이어도 동작) ② `billing_type = 'prepaid'` 전환 ③ `paid_amount_krw += amount` 누적 ④ `payment_method` 기록 ⑤ status는 `'draft'` 유지(내원 미완료) 한다. `payments` 감사 트리거가 actor와 함께 UPDATE를 자동 기록한다.

**AC3 (선결제 상태 가드):**
**Given** 이미 `finalized` 또는 `cancelled`인 수납 건에 대해
**When** 선결제를 시도하면
**Then** `PT409`→409(`invalid_transition`)로 거부된다(선결제는 draft에서만). 금액 ≤ 0은 Pydantic 422(1차)·RPC 방어(최종선)로 거부된다.

**AC4 (차액 정산 finalize — 단일 누계):**
**Given** 선결제된(`paid_amount_krw > 0`) draft 수납 건이 진료를 거쳐 본인부담금(`copay_amount_krw`)이 산정됐을 때
**When** finalize하면
**Then** `paid_amount_krw = greatest(copay_amount_krw, paid_amount_krw)`로 정산되어(차액 = `copay − 선납` 수금) 완납 상태가 되고, draft→finalized·`complete_encounter` 완료 전이가 원자적으로 일어난다(7.4 흐름 재사용). 후수납(선납 0)은 `greatest(copay, 0) = copay`로 **무회귀**.

**AC5 (결제상태 A3 — 부분 상태 신설, UX-DR14):**
**Given** 수납 건의 결제 상태에 대해
**When** 결제상태 배지를 렌더하면
**Then** ① `finalized`=완료(그린 `status-done`·✓) ② `draft` & `paid_amount_krw > 0`=**부분(앰버 `status-received`/`status-received-ink`·◐)** ③ `draft` & `paid = 0`=미수납(로즈 `status-cancelled`·○) ④ `cancelled`=취소(✕·취소선)로 표시된다. 색비의존(글리프+라벨 병기).

**AC6 (선수납 패널 — billing-detail):**
**Given** draft 수납 건 상세 화면에서
**When** 화면을 열면
**Then** 결제수단 토글 + 선결제 금액 입력 + "선결제" 버튼이 노출되고, 현재 `이미 납부`(paid)·`납부할 금액`(차액 = copay − paid, `registered`에서 수가 미발생 시 안내 문구)이 표시된다. 선결제 확정은 **신원 재진술 confirm**(이름·차트번호·선결제액)을 거친다(UX-DR21). 금액=KRW 정수·`tabular-nums`·"원".

**AC7 (선수납 진입점 — 워크리스트 확장):**
**Given** 수납 워크리스트(정산 대상 내원)에서
**When** 목록을 조회하면
**Then** `registered`와 `in_progress` 내원이 모두 노출되고(오늘·KST), 각 행에 상태 칩(`접수`=선수납 가능 / `진찰중`=정산 대상)이 표시된다. 행 선택 시 동일 `billing-detail`로 진입한다.

**AC8 (과납 표시 — 환급 이월):**
**Given** finalize 시 `paid_amount_krw > copay_amount_krw`(과납)인 경우
**When** 영수증/완료 패널을 보면
**Then** `due_amount_krw`(=copay − paid) 음수가 '환급 대상' 으로 표시되고 finalize는 허용된다(`paid` 초과분 보존). 실제 환급 처리는 7.9 이월임을 명시.

**AC9 (권한·경계):**
**Given** 선결제 쓰기에 대해
**When** 권한을 평가하면
**Then** `payment.manage`(기존·7.2 reception grant)가 게이트한다(신규 권한 0). nurse·doctor는 403(baseline). **신규 마이그 1건(0051)·신규 컬럼 0·신규 권한 0·신규 테이블 0.**

**AC10 (테스트):**
**Given** 변경 표면에 대해
**When** 테스트를 실행하면
**Then** API(prepay 누적·billing_type 전환·registered 선결제·draft 가드·차액 finalize·과납·권한 403·워크리스트 registered 포함)와 web(부분 배지·선수납 패널·차액 표시·워크리스트 상태 칩) 테스트가 통과하고, 7.4 후수납 회귀 테스트가 깨지지 않는다.

### 내원 상태머신 — 선수납 시점 (0010_encounters.sql)

```
(INSERT) → scheduled | registered
scheduled  → registered(register_encounter) | cancelled | no_show
registered → in_progress(start_consult)     | cancelled        ← 선수납은 여기(registered, 진찰 전)
in_progress→ completed(complete_encounter)   ← finalize가 트리거(7.4 billing-completes)
종결(completed·cancelled·no_show) = 이탈 없음(역행 금지)
```
- 진찰료 fee_item은 `registered→in_progress`(start_consult) 트리거(`fee_on_encounter_start`)로 적재 → **registered에선 수가 0**(선결제=예치). 검사/처치료는 perform 시 적재.
- 선결제 후 in_progress 전이 시 동일 payment 헤더 유지(encounter_id UNIQUE)·`paid` 보존. in_progress에서 build_payment가 수가 집계(헤더는 on conflict do nothing).
- finalize는 in_progress에서만(complete_encounter 게이트). 선납만 받고 진료 안 한 registered 내원은 finalize 불가(total≤0=PT409) — 취소 시 환급=7.9.
