# Story 9.2: 스크린샷 캡처 파이프라인 · 더미 데이터

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 도움말 작성자(개발자),
I want 역할별로 로그인해 각 화면을 자동 캡처하고 번호 하이라이트를 DOM 오버레이로 입히는 **재현 가능한 파이프라인**과, 빈 화면을 막는 **충분한 더미 데이터**를,
so that 9.3~9.8 이 메뉴별 안내 이미지를 별도 편집 도구 없이 일관되게 생성·갱신하고, 도움말 페이지가 그 정적 이미지를 임베드할 수 있다.

## Acceptance Criteria

1. **(파이프라인) 역할 로그인 → 화면 캡처 → PNG.** `tools/screenshots/`에 커밋된 재현 가능한 파이프라인이 있다. playwright + headless chromium 으로 **역할 계정으로 로그인**한 뒤 지정 화면(URL 또는 내비게이션 단계)으로 이동해 **클라우드 실데이터 화면을 PNG 로 저장**한다. 단일 인라인 스크립트의 반복이 아니라, "캡처 대상(역할·화면·이동·번호 하이라이트 스펙)"을 **데이터로 선언**하면 러너가 처리하는 구조다(현재 `capture-doctor.mjs` 의 인라인 패턴을 일반화). 의사(doctor) 2화면이 이 일반화된 러너로 **9.1 과 동일한 결과**를 재생산함을 증명한다.

2. **(FR-252) DOM 오버레이 번호 하이라이트 — 외부 편집 도구 의존 0.** 캡처 시 핵심 동작요소에 **번호 원(①②③…) + 빨강 테두리**를 DOM 으로 주입해 굽는다(`lib.mjs` 의 `annotate()`). Pillow·포토샵 등 별도 이미지 편집 단계가 없다. 다음 화면 캡처 전 오버레이는 제거된다(`clearAnnotations()`). 캡처 스펙의 번호는 해당 화면의 `help-content.ts` `hotspots[].num` 과 1:1로 일치한다.

3. **(더미 데이터) 빈 화면(데이터 0) 방지 — 오늘 기준.** `supabase/demo_seed.sql` 을 보강해, **직원 5역할의 각 메뉴 화면이 현실적인 데이터로 채워진 상태**로 캡처되도록 한다(대기 현황·처치/촬영 워크리스트·판독 대기·수납·예약·리마인더 등이 0행이 아님). 시드는 **오늘(KST) 기준 상대 날짜**로 생성되어 언제 재실행해도 "오늘" 데이터가 있고, **멱등 재실행**(결정적 UUID 프리픽스로 SEED 행만 정리)과 **불변식**(주민번호=크립토 함수, 수가=상태전이 트리거 발화, DDL 미생성)을 지킨다.

4. **(AC4) 정적 임베드 — 런타임 캡처 0.** 생성된 PNG 가 `web/public/help/<role>/<screen>.png` 경로(이미 커밋된 의사 이미지의 규칙)에 배치되고, 도움말 페이지는 `help-content.ts` 의 `image` 값으로 **정적 임베드**한다. 페이지/서버는 런타임에 캡처를 호출하지 않는다. 파이프라인 산출물이 이 경로에 바로 떨어지거나(권장), 산출물 → public 복사 단계가 README 에 문서화된다.

5. **재현 가능성·문서화·회귀 무파괴.** `tools/screenshots/`에 **README**(전제조건·설치·계정·env BASE·실행법·산출 위치·이미지 커밋 정책)가 있어 새 사람이 그대로 따라 재현할 수 있다. `node_modules/`·`out/` 는 `.gitignore` 로 제외되고 `web/public/help/**` 는 커밋된다. 웹 앱 코드(`web/src/**`)·DB 스키마(마이그레이션)·API 는 **변경하지 않는다**(seed 데이터와 devtool 만). 기존 9.1 도움말 페이지·의사 이미지 표시가 그대로 동작한다.

> **범위 경계 (반드시 지킬 것):** 이 스토리는 **파이프라인(기계) + 더미 데이터 + 출력 규칙 + 의사 워크드 예제**까지다.
> - **9.2 는** 재사용 가능한 캡처 러너·스펙 포맷·README 를 만들고, demo_seed 를 전 직원 화면이 비지 않게 보강하며, 의사 2화면을 일반화된 러너로 재생산해 증명한다.
> - **9.2 는 하지 않음:** 원무/의사판독/간호/방사선사/관리자 **각 역할의 help-content.ts 콘텐츠 작성 + 전체 화면 캡처·임베드** — 그건 **9.3~9.7**. 환자 포털 도움말·환자 시드·환자 캡처 = **9.8**(데모 환자 계정 미시드·모바일 셸 특성상 후순위).
> - 즉 9.2 의 러너는 **의사 스펙만** 들고 있어도 된다(machinery 증명). 9.3~9.8 이 각자 "캡처 스펙 1줄 + help-content 1줄 + 러너 실행"으로 자기 이미지를 만든다.
> - **더미 데이터 보강은 전 직원 화면을 대상으로 한다**(seed 는 공용 자산이므로 9.2 에서 한 번에 충분히 깐다). 단, 콘텐츠(번호·설명)·이미지 임베드는 각 역할 스토리 소관.

### ⚠️ 비자명·실수 방지 (이 스토리에서 LLM이 틀리기 쉬운 것)
1. **바퀴 재발명 금지 — WIP 가 이미 있다.** `tools/screenshots/`(미추적이지만 9.1 의 의사 이미지를 실제로 생산한 검증된 코드)를 정독하고 **일반화만** 한다. `lib.mjs` 의 `login()`/`annotate()`/`clearAnnotations()` 가 단일 출처 — 인라인으로 다시 짜지 말 것.
2. **번호는 이미지에 굽는다(좌표 오버레이 아님).** `annotate()` 가 캡처 시점에 빨강 원+테두리를 DOM 에 주입해 PNG 에 박는다. 그러므로 페이지(9.1)는 좌표 오버레이를 그리지 않고 이미지 + 번호표만 병기. **캡처 스펙의 `n` 은 `help-content.ts` 의 `hotspots[].num` 과 1:1** — 어긋나면 설명이 엉뚱한 번호를 가리킴.
3. **클라우드 실데이터가 기본 캡처 대상.** `lib.mjs` `BASE = https://kuntae802.mooo.com/patient_management_system`. AC1 이 "클라우드 실데이터"를 명시. 따라서 **더미 데이터(demo_seed)는 그 클라우드 Supabase 에 적용**되어야 캡처에 보인다(로컬 DB 가 아님). `BASE` 는 env 로 오버라이드 가능하게 둬 로컬(`supabase start`+`npm run dev`) 캡처도 선택지로(다만 정식은 클라우드).
4. **`annotate()` 는 viewport 좌표(`position:fixed`)다 — 폴드 아래 함정.** `boundingBox()`(viewport 기준) + `fixed` 오버레이라, 1440×900 뷰포트 **밖(아래)** 요소는 박스가 어긋나거나 안 잡힌다. `fullPage` 스크린샷은 fixed 오버레이를 깨므로 사용 금지(`capture-doctor` 도 viewport 캡처). 폴드 아래 요소를 표시하려면 스펙에서 `page.evaluate(()=>scrollTo…)` 후 재좌표화하거나, **in-viewport 핵심 요소만** 번호 매김(9.3~9.8 작성자에게 README 로 안내).
5. **DB 불변식·멱등성 깨지 말 것(demo_seed).** 주민번호=`encrypt_sensitive`/`blind_index`(평문 INSERT 금지·reveal 동작 보존), **수가(fee_items)=상태전이 UPDATE 로 트리거 발화**(직접 INSERT 금지), 결정적 UUID 프리픽스로 멱등 정리(신규 자식행은 상단 delete 블록에도 반영해 FK 위반 없게), 오늘(KST) 상대 날짜. **DDL/마이그레이션 생성 금지** — demo_seed 는 데이터만.
6. **오더-by-내원상태 게이트(마이그 0053).** 처방·검사·처치 오더 INSERT 는 내원이 종결되기 *전*(registered/in_progress)에 해야 BEFORE INSERT 트리거를 통과. 시드가 "발행 → 진찰시작 → 완료" 순서를 지키는 이유(L198·208·291 주석). 갭 채울 때 동일 순서 유지.
7. **앱 코드·스키마 무변경.** 9.2 작업면은 `tools/screenshots/**` + `supabase/demo_seed.sql` + `web/public/help/**`(이미지) 뿐. **`web/src/**` 변경 0 · 마이그레이션 0 · API 0 · 권한 0 · 생성 TS 타입 무수정.** help-content 콘텐츠 데이터 행 추가는 9.3~9.8(의사는 9.1 에 이미 있음).

### 캡처 스펙 포맷 (제안 — capture-doctor.mjs 일반화)
```js
// tools/screenshots/specs.mjs (제안)
export const SPECS = [
  {
    role: "doctor", account: "doctor@pms.local",
    screenKey: "waiting", output: "../../web/public/help/doctor/waiting.png",
    goto: "/doctor/waiting",                       // 직접 URL 도달 화면
    annotate: (page) => [
      { n: 1, locator: page.getByRole("link", { name: "진료 대기" }) },
      { n: 2, locator: page.locator("select").first() },
      { n: 3, locator: page.getByRole("button", { name: "진료 시작" }).first() },
      { n: 4, locator: page.getByRole("button", { name: /진료 계속/ }).first() },
    ],
  },
  {
    role: "doctor", account: "doctor@pms.local",
    screenKey: "hub", output: "../../web/public/help/doctor/hub.png",
    steps: async (page) => {                        // 상호작용으로 도달하는 화면
      await page.getByRole("button", { name: /진료 계속/ }).first().click();
      await page.waitForURL("**/encounter/**", { timeout: 30000 });
      await page.waitForTimeout(3500);
    },
    annotate: (page) => [ /* n:1~6, help-content hub hotspots 와 일치 */ ],
  },
];
```
러너: 스펙 순회 → (역할 바뀌면) 로그인 → `goto`(URL) 또는 `steps`(상호작용) → 안정화 대기 → `annotate` → screenshot → `clearAnnotations`. 인자로 `role`/`screenKey` 필터. **번호는 help-content hotspots 와 동기**(작성자 책임 — README 에 명문화).
