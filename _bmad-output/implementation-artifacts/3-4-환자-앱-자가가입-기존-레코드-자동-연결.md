# Story 3.4: 환자 앱 자가가입 · 기존 레코드 자동 연결

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 환자,
I want 앱에서 회원가입(본인인증 시뮬)하면 병원이 만들어 둔 기존 레코드와 자동으로 연결되기를,
so that 중복 환자 없이 본인 데이터를 쓸 수 있다.

## Acceptance Criteria

1. **AC1 — Supabase Auth 자가가입 + 계정 생성 (FR-001):** `supabase/config.toml`의 공개 가입 차단을 **재활성**(`[auth] enable_signup = false → true`)하고, 환자가 앱 `(auth)/signup` 화면에서 이메일·비밀번호로 회원가입하면 **Supabase Auth 계정(`auth.users`)이 생성**된다(클라 `supabase.auth.signUp` — `enable_confirmations=false`라 즉시 세션 발급). 가입만으로는 `patients` 행이 생기지 않는다(연결은 AC2). 비밀번호는 `minimum_password_length=8`·`lower_upper_letters_digits` 정책을 따른다(Story 1.4 하드닝 — config 이미 설정됨, 본 스토리는 master 스위치만 켠다). **직원 계정은 영향 없음**(직원=Story 1.8 관리자 생성 유지, 가입 화면은 환자 전용).

2. **AC2 — HMAC blind index 자동 연결 + 본인인증 시뮬 (FR-003):** 가입(세션 보유) 후 본인인증 시뮬 화면에서 환자가 **주민번호 + 성명**을 입력하면, **인증된** `POST /api/v1/patients/self-link`(Bearer)가 `blind_index(normalize_rrn(주민번호))`로 기존 환자를 매칭한다. **연결 대상 `auth_uid` 는 JWT 주체(`sub`)에서만 도출**(클라 제공 uid/patient_id 미수용 — 세션 uid 스코프). 결과 분기:
   - **정확히 1건 매칭 · `auth_uid` IS NULL · 성명 일치** → `auth_uid = sub`로 **연결**(service_role UPDATE, 0009 감사 트리거가 자동 기록). 응답 = 마스킹 요약(`chart_no`·`name`·`birth_date`·`sex`·`resident_no_masked`). **이후 RLS `patients_select_self`(`auth.uid()=auth_uid`)로 본인 포털(Epic 8) 조회 가능**.
   - **이미 본인(`auth_uid == sub`)에 연결됨** → **멱등 성공**(200, 같은 요약 반환 — 재시도·중복 제출 안전).
   - 매칭 행이 정의된 전이(연결 가능)가 아니면 AC3의 안전 폴백.

   **신규 마이그레이션 0건** — `auth_uid` 컬럼·`idx_patients_auth_uid`·RLS 본인 정책·감사 트리거는 전부 `0009_patients.sql`에 이미 존재. 쓰기는 service_role(RLS 우회)이라 새 RLS 쓰기 정책 불요. `auth_uid` 설정·hash 조회는 0005 프리미티브(`blind_index`)·0009 컬럼을 **확인·소비만** 한다.

3. **AC3 — 매칭 실패/모호 시 안전 폴백 (안내, 중복·탈취 방지):** 자동 연결이 불가하면 신규 레코드를 함부로 만들지 않고 **안내(guided fallback)**로 안전하게 처리한다(에픽 AC "신규 레코드 생성 **또는** 안내" 중 **안내** 채택 — §결정 D-2):
   - **0건 매칭**(등록된 진료기록 없음) → `404 no_patient_record`("등록된 진료 기록이 없습니다. 병원 방문/문의 후 다시 연결하세요").
   - **주민번호 일치 · 성명 불일치**(본인인증 시뮬 시대의 사칭 방지 1차선) → `422 identity_mismatch`("입력하신 정보가 기록과 일치하지 않습니다. 병원에 문의하세요") — **연결하지 않는다**.
   - **이미 다른 계정에 연결된 주민번호**(`auth_uid` 가 sub 아닌 값) → `409 already_linked_other`("이미 가입·연결된 주민번호입니다") — 계정 탈취 차단.
   - **이 계정이 이미 다른 환자에 연결됨**(sub 가 다른 patient 의 `auth_uid`) → `409 account_already_linked`("이 계정은 이미 다른 환자에 연결되어 있습니다") — 1 계정 = 1 환자 불변식.

   raw 주민번호·성명은 응답·로그·에러봉투·URL 어디에도 echo하지 않는다(코드·마스킹만). 직원(active 5역할) 계정은 self-link 호출 불가(`403` — 직원 uid 가 환자 행에 묻는 것 방지).

4. **AC4 — 웹 가입·연결 UI + 접근성:** ① `(auth)/signup` 가입 폼(이메일·비밀번호·비밀번호 확인, RHF+Zod, Supabase `signUp`) — 로그인 화면에 "회원가입" 링크 추가, **미들웨어 공개 경로에 `/signup` 추가**(미인증 접근 허용). ② 가입 성공(세션) → `(patient)/onboarding` 본인인증 시뮬 화면(주민번호·성명 입력 → `apiFetch POST /v1/patients/self-link`). 성공(`linked`/`already_linked`) → 토스트 + `/portal` 이동. 폴백(404/422/409) → 화면 내 안내 메시지(색 단독 의존 금지, `role="status"`/`role="alert"`). ③ `/portal` placeholder 에 "본인 진료기록 연결" 진입점(미연결 환자 재진입). 주민번호 입력은 마스킹 불요(본인이 입력), `tabular-nums`, "(필수)" 표시 + `aria-required`, 제출 중 버튼 disable(이중 제출 방지).

> **이월·교차절단 인수 조건(이 스토리에서 확인):** ① **신규 마이그레이션 0건** — `auth_uid`/인덱스/RLS/감사가 `0009_patients.sql`에 존재함을 **확인만**(재선언·새 0010 금지 — 다음 번호는 Epic 4 몫, glossary L184). ② **세션 uid 스코프 강제** — `auth_uid`는 항상 검증된 JWT `sub`에서만(클라 입력 uid 절대 미수용, project-context PII 경계). ③ **본인인증 = 시뮬 seam** — 실 PASS 미연동(아키텍처 §시뮬 범위), 시뮬 시대의 사칭 방지는 **성명 일치 가드**가 1차선(주민번호+성명만 독립 신호; birth_date·sex는 RRN 파생이라 종속). 실 본인인증 도입 시 이 seam이 교체점. ④ **A-3 감사 PII(계속 defer)** — self-link UPDATE가 `audit_logs` 스냅샷에 name/phone 평문 적재(기존 deferred-work L166 통합 추적, 신설 금지). ⑤ **공개 가입 재활성 = 새 공격면** — 이메일 미확인 throwaway 계정·RRN 추측 가능. 게이트(성명·이미연결 거부)는 1차선, 레이트리밋/이메일확인/잠금은 보안 하드닝으로 이월(§위협 모델).

### self-link 설계 (핵심 — 보안·정합)

- **경로·권위:** `POST /api/v1/patients/self-link`(인증, **비직원**). 게이트 = `get_current_patient`(직원 403). **진짜 권위 = "본인 sub로만 연결"** — `auth_uid`는 `db.link_self_patient`가 인자로 받는 `sub`(JWT 주체)에서만 도출, **클라가 patient_id/uid를 제공하지 않는다**(스키마에 그 필드가 없음 — project-context "환자 포털=세션 uid 스코프, 클라 제공 patient_id 미수용"). 직원 권한 게이트(`require_patient_*`) 사용 금지(환자는 RBAC 권한 0).
- **RLS 우회 필수:** 자가연결은 **타 환자 행**(원무가 만든 미연결 레코드)을 hash로 찾아야 한다. `authenticated_conn`은 소유자 role로 연결돼 RLS를 우회하므로 cross-patient 조회 가능(`patients_select_self`로 막히지 않음). 이것이 self-link를 **반드시 FastAPI(service_role) 경유**로 두는 이유(클라 Supabase 직접조회는 RLS로 본인행만 → 매칭 불가).
- **조건부 UPDATE(동시성):** `UPDATE ... SET auth_uid=$sub WHERE id=$id AND auth_uid IS NULL`. 두 요청이 같은 레코드를 동시에 연결하려 하면 `auth_uid IS NULL` 술어가 1명만 통과시키고, 패자는 0행 → 재조회 → `already_linked_other`. UNIQUE 인덱스(`resident_no_hash`)가 1행 보장이라 "모호(2+ 매칭)"는 구조적으로 불가 — 에픽의 "모호 시 폴백"은 **성명 불일치/이미연결**로 현실화(§AC3).
- **1 계정 = 1 환자 불변식:** UPDATE 전 `auth_uid=$sub` 선점 검사(Task 4.3 step 2)로 한 계정이 둘째 환자에 연결되는 것 차단(`account_already_linked`). DB partial unique index(`unique (auth_uid) where auth_uid is not null`)는 더 강한 보증이나 **신규 마이그레이션이라 미도입** — 앱 로직 가드 + (희박한)동시 2-RRN 레이스는 §Open Q/하드닝 이월.
- **멱등:** 같은 환자가 self-link를 두 번 호출(중복 제출·재시도) → 둘째는 `already_linked`(200, 동일 요약). 네트워크 재시도·뒤로가기 안전.
- **감사:** UPDATE는 0009 `trg_patients_audit`가 자동 기록(action='update', actor=`app.actor_id`=본인 sub). **앱이 audit_logs 직접 INSERT 금지.** 스냅샷에 name/phone 평문 PII 적재 = A-3 교차절단(deferred-work L166 통합 추적, 신설 금지).

### 본인인증 = 시뮬 seam + 위협 모델 (실제처럼 — 명시)

- **시뮬 범위:** 아키텍처 §시뮬은 본인인증(PASS)을 "연결 가능한 이음매(seam)"로 둔다(실연동 범위 밖). `services/identity.simulate_identity_verification`이 그 자리 — 현재 통과만, 실 PASS/NICE 도입 시 교체점. 과도 구현 금지(외부호출·플래그 분기 없이 seam 표식 + docstring).
- **시뮬 시대의 사칭 위협:** 공개 가입(이메일 미확인) + 시뮬 본인인증이면, 악의 사용자가 **타인의 주민번호를 추측·입력해 그 사람의 진료 레코드를 자기 계정에 연결(탈취)** 시도 가능. 독립 신호는 **성명**뿐(birth_date·sex는 RRN에서 파생→종속, 같은 RRN이면 항상 일치). 따라서 **성명 일치 가드**가 1차 방어선(불일치 → `identity_mismatch`, 연결 거부). **이미 연결된 레코드 재연결 거부**(`already_linked_other`)가 2차선(선점된 본인 데이터 보호).
- **잔여 위험·하드닝 이월:** 성명까지 아는 표적 공격은 시뮬로는 못 막는다(실 PASS가 RRN 소유를 암호학적으로 증명해야 해소) — **명시 갭**. RRN 추측 무차별 대입은 config 레이트리밋(`sign_in_sign_ups`/`token_*`)이 부분 완화하나 self-link 전용 레이트리밋·연속 실패 잠금·이메일 확인 활성·캡차는 **보안 하드닝 스토리로 이월**(은폐 아님). 이 갭은 "실제처럼"을 위해 문서화하되 MVP 시뮬에서 수용.

## Open Questions (개발 착수 전 확인 — 차단 아님)

1. **no-match 폴백 = 안내 vs 신규 레코드 생성 (D-2):** 에픽 AC는 "신규 레코드 생성 **또는** 안내". 본 스토리는 **안내**를 채택(앱 자가 환자 생성은 insurance_type 등 필수 미입력·임상 컨텍스트 부재·patient.create 권위 문제 + Epic 3.1이 환자 생성을 원무 책임으로 프레이밍). 만약 "앱-우선 신규 환자"를 지원하려면 self-signup 전용 환자 생성 경로(insurance_type 등 추가 입력 폼 + `auth_uid=sub` 직접 INSERT, patient.create 비요구 — 본인 레코드 생성 권위)를 별도 Task로 추가. **확인 요망**: 데모 시나리오에 앱-우선 신규 환자가 필요한가?
2. **본인인증 시뮬 강도:** 사칭 방지 1차선 = **성명 일치**(독립 신호는 성명뿐 — birth_date/sex는 RRN 파생). 더 강하게 하려면 등록 시 저장된 `phone` 부분 일치(있을 때만)를 보조 가드로 추가 가능. 표적 공격(성명까지 아는 경우)은 시뮬로 미해소 — 실 PASS 연동에서 해소(seam 교체). **확인**: 성명 단독 가드로 충분(MVP 시뮬)한가, phone 보조 가드 추가할까?
3. **`GET /patients/self` 포함 여부:** onboarding 진입 시 "이미 연결됨" 분기를 위한 조회. self-link 멱등성(같은 호출 두 번=200)으로 대체 가능해 **스코프 최소화 시 생략** 가능. 환자 portal(Epic 8)이 본인 데이터를 RLS 직접조회하므로 중복 우려도 있음. **확인**: 3.4에 포함할까, Epic 8로 미룰까?(현재 Task 6.3·5.3·4.4를 선택으로 표기)
4. **1 계정 = 1 환자 DB 강제:** 현재 앱 로직 가드(`account_already_linked`)로 차단, DB partial unique index(`unique(auth_uid) where auth_uid is not null`)는 신규 마이그레이션이라 미도입. 동시 2-RRN 연결 레이스(희박)는 앱 로직으로만 방어. 강한 보증이 필요하면 Epic 4 마이그레이션(0010)에 partial unique index를 묶어 추가 검토(번호 드리프트 회피).
5. **공개 가입 보안 하드닝:** `enable_signup=true` 재활성 = throwaway 계정·RRN 추측 공격면. config 기본 레이트리밋만 의존, 이메일 확인·self-link 전용 레이트리밋·연속 실패 잠금·캡차는 보안 하드닝 스토리로 이월. **확인**: 데모 범위에서 수용 가능한가(권장: 수용 + 갭 문서화)?
