> ① **반복 이월 "오더-by-내원상태 게이트" = 7.10에서 닫는다** — 종결(completed/cancelled/no_show)·비-in_progress·soft-deleted 내원에 오더 **생성·수행** 차단. `assert_encounter_orderable` 가드 + 오더 생성 BEFORE INSERT 트리거(examinations/treatment_orders/prescriptions) + 수행 RPC(perform_examination/perform_treatment_order) 재정의. deferred-work **L277·L290·L333(treatment perform)·L346** 청산. **이것이 7.10의 실질 작업** — "수행분만 정산"은 이미 구조적 보장(아래 ③).
> ② **미수행 오더 처분 = 그대로 둔다** — 부분수행 내원 완료(정산) 시 남는 미수행(`status='ordered'`) 오더는 `ordered` 상태로 잔존. 청구는 `fee_items` 부재로 **구조적 제외**(자동). 오더 cancel 메커니즘 **추가 안 함**(forward-only 상태머신 유지·신규 RPC 0·7.9 "구조적 보장" 선례).
> ③ **부분수행 가시성 = 카운트 요약 배지** — billing-detail에 "미수행 오더 N건(청구 제외)" 주의 배지. reception은 **order.read 미보유(최소권한)** → 임상 상세 없이 **payment.read 경로**로 카운트만 노출(`_PAYMENT_HEADER_SELECT` 상관 서브쿼리·신규 권한 0). 직원이 finalize 전 "부분 정산"임을 인지.

## Story

As a 원무 직원,
I want 일부 오더만 수행 후 환자가 이탈한 in_progress 내원도 수행된 항목까지 정산하고, 종결·취소된 내원에는 새 수가가 적재되지 않도록 보장하기를,
so that 부분 진료도 정확히(수행분만) 청구되고, 한번 정산·종결된 내원의 청구액이 사후에 변조되지 않는다.

## Acceptance Criteria

> 출처: `epics.md` Story 7.10(L1341-1351) — **Given** 일부 오더만 수행된 내원에 대해 **When** 정산을 수행하면 **Then** 수행된 항목까지만 수납·정산된다(FR-119, 이월 갭 ⑥ 정산 경로).
> PRD FR-119: 부분 수행(일부 오더만 수행 후 이탈) 내원도 수행된 항목까지 수납·정산. 이월 스키마 갭 ⑥(취소·노쇼·**부분수행** 정산 경로·아키텍처 소유). 7.9가 취소·노쇼를 닫았고 **7.10이 부분수행으로 갭 ⑥을 완결**한다.
> 마이그레이션 = **`0053_order_encounter_gate.sql`**(Epic 7 블록 0045~0059 — 0052 다음).

**AC1 (FR-119 수행분만 정산 — 구조적·정산 경로 = 7.4 재사용):**
**Given** 일부 오더만 수행된(진찰 시작 + 일부 검사/처치 perform + 나머지 `ordered` 잔존) in_progress 내원에 대해
**When** 원무가 `POST /v1/encounters/{id}/payment/finalize`(7.4)로 정산하면
**Then** `fee_items`가 **수행된 오더 + 진찰료에만** 적재돼 있어(perform 시점에만 fee 발생·5.10) 수행분만 청구되고, 미수행(`ordered`) 오더는 `fee_item`이 없어 **구조적으로 청구에서 제외**된다. **신규 정산 경로/RPC 0** — 기존 `build_payment`→`price_payment`→`finalize_payment`→`complete_encounter` 트랜잭션(7.4)이 그대로 부분수행을 정산한다. (본 AC는 **검증·테스트**로 충족 — 동작은 이미 존재.)

**AC2 (오더-by-내원상태 게이트 — 가드 함수, 설계 결정 ①):**
**Given** `0053`에서
**When** `assert_encounter_orderable(p_encounter_id uuid) returns void`(SECURITY DEFINER·`set search_path=public`)를 정의하면
**Then** ① 내원 미존재 → `raise ... using errcode='PT404'` ② `is_active=false`(soft-deleted) → `raise ... using errcode='PT409'` ③ `status in ('completed','cancelled','no_show')`(종결) → `raise ... using errcode='PT409'`. **scheduled/registered/in_progress(active)=통과**(registered 내원 워크리스트 플로우 보존 — §Completion Notes 게이트 의미 정밀화 참조). SECURITY DEFINER로 RLS 우회 조회(트리거·RPC 모두에서 actor 무관 평가). `_map_pg_sqlstate`(0010) 기존 매핑(PT409→409·PT404→404) 재사용·추가 매핑 불요.

**AC3 (오더 생성 게이트 — BEFORE INSERT 트리거, deferred L277·L290 청산):**
**Given** `examinations`·`treatment_orders`·`prescriptions` 신규 행 INSERT에 대해
**When** 각 테이블에 `before insert` 트리거(`trg_*_encounter_gate`)를 부착해 `assert_encounter_orderable(new.encounter_id)`를 호출하면
**Then** 비-in_progress·soft-deleted 내원에 오더 생성이 차단된다(직접 API·stale 탭 포함). 정상 경로(진료 허브가 in_progress에서만 오더 패널 렌더 → start_consult 후 발행)는 무영향. deferred-work **L277**(`insert_prescription`)·**L290**(`insert_examination`)+treatment 생성 동일 클래스 청산. ⚠️ **테스트 픽스처**: 오더 INSERT 전 내원이 `in_progress`여야 함(실 플로우 정합 — start_consult 선행) → 비-in_progress 픽스처 일괄 수정(AC9).

**AC4 (오더 수행 게이트 — perform RPC 재정의, deferred L333·L346 청산·정산액 보호):**
**Given** `perform_examination`·`perform_treatment_order`(0015 RPC)에 대해
**When** `0053`에서 두 RPC를 `create or replace`로 재정의해 **오더 행 룩업(`for update`) 직후·상태 전이 전**에 `perform assert_encounter_orderable(v_row.encounter_id);`를 추가하면
**Then** 종결(completed/cancelled)·취소 내원의 `ordered` 오더 수행이 차단되어 → **`fee_items`가 적재되지 않는다**(deferred-work **L346** "종결 내원 수행 시 수가 적재" 청산 = **정산 종료된 내원의 청구액 사후 변조 방지**·Epic 7 정산액 무결성). 정상 경로(in_progress 중 수행=fee 발생)는 무영향. **기존 권한 검사(`has_permission('examination.perform')`/`treatment.perform`)·소스상태 검사(`status<>'ordered'` PT409·재수행 차단 FR-093)·전이 로직은 그대로 보존**(게이트 1줄 추가만). `complete_examination`(판독·performed→completed)은 **게이트 안 함**(종결 내원 사후 판독은 정당·fee 미발생·deferred L546 판독큐 정합).

**AC5 (미수행 오더 카운트 — payment.read 경로, 설계 결정 ③):**
**Given** 수납 헤더 조회(`_PAYMENT_HEADER_SELECT`·db.py)에 대해
**When** `patient_name`/`chart_no` 상관 서브쿼리 패턴을 미러해 미수행 오더 카운트 서브쿼리를 추가하면
**Then** `(select count(*) from public.examinations ex where ex.encounter_id = payments.encounter_id and ex.status='ordered' and ex.is_active) + (select count(*) from public.treatment_orders tr where tr.encounter_id = payments.encounter_id and tr.status='ordered' and tr.is_active) as pending_orders_count`가 헤더에 노출된다. **`prescriptions` 제외**(issued/dispensed — 수행 개념 없음·fee_item 0·원외 약제비 스코프아웃 → "issued"는 임상 완결 상태). `PaymentResponse.pending_orders_count: int = 0` 추가(web `Payment` 거울 동반). **order.read grant 0**(임상 상세 미노출·카운트만·최소권한 유지·reception payment.read 기보유). 모든 헤더 소비처(build/finalize/prepay/settle/fetch)에 컬럼 1개 노출만(기존 값 무영향).

**AC6 (부분수행 배지 — billing-detail·UX-DR14 색비의존):**
**Given** draft 수납 건 상세(billing-detail)에서
**When** `payment.status === 'draft' && payment.pending_orders_count > 0`이면
**Then** 진료비 항목/결제 섹션 인근에 **"부분 수행 — 미수행 오더 {N}건은 청구에서 제외됩니다"** 주의 배지가 표시되어, 원무가 finalize 전 "부분 정산"임을 인지한다. A3 색비의존(앰버/info 톤 + 아이콘 글리프 + 라벨 병기). `pending_orders_count===0`이면 미표시(전체 수행=일반 정산). **finalize 동작·결제수단 토글·신원 confirm은 무변경**(7.4 그대로). cancelled/finalized 상태에선 미표시(종결).

**AC7 (미수행 오더 처분 = 그대로 둠·경계, 설계 결정 ②):**
**Given** 부분수행 내원이 finalize되어 completed로 전이될 때
**When** 미수행(`ordered`) 오더 처리를 평가하면
**Then** 미수행 오더는 **`ordered` 상태로 잔존**(자동 cancel·삭제·상태 전환 **안 함**). 청구 제외는 `fee_items` 부재로 구조적 보장(능동 처리 코드 0). **오더 cancel 전이/RPC 추가 금지**(forward-only 상태머신 유지·스코프 누수 방지). 단, AC4 게이트로 인해 completed 이후엔 그 `ordered` 오더를 **수행할 수 없다**(perform→PT409) → 잔존 오더는 영구 `ordered`(불수행 확정·청구 0).

**AC8 (권한·경계·변경 규모):**
**Given** 7.10 변경 표면에 대해
**When** 권한·스키마를 평가하면
**Then** **신규 권한 0·신규 테이블 0·신규 컬럼 0·신규 엔드포인트 0·신규 RPC 1개(`assert_encounter_orderable` 가드만)**. 게이트는 기존 오더 생성/수행 권한 체계를 **무변경**(perform 권한 검사 그대로·앞단에 내원상태 가드만 추가). pending count는 기존 `payment.read`(reception 기보유). 정산은 기존 `payment.manage`+`encounter.complete`(7.2/7.4). **신규 마이그 1건(0053)**. nurse·doctor·reception baseline 권한 불변.

**AC9 (무회귀 — 게이트 blast radius 관리):**
**Given** 게이트 도입 후
**When** 전체 테스트를 실행하면
**Then** ① **정상 오더 플로우**(in_progress 내원에 생성·수행) 무손상 ② **7.4 finalize·7.8 선후수납·7.9 취소환급·7.5~7.7 문서** 회귀 무손상 ③ **Epic 5 오더 도메인**(5.2~5.10) 회귀 무손상. ⚠️ **테스트 픽스처 일괄 점검 필수**: 오더 생성/수행을 직접 INSERT/RPC로 하는 픽스처가 내원을 `in_progress`로 전이하지 않았다면 게이트에 막힘(PT409) → start_consult 선행 또는 in_progress 셋업으로 일괄 수정(`test-then-fix-batch` 방식). 게이트가 **기존 테스트의 잘못된 셋업(비실제 플로우)을 드러내는 것**이지 회귀가 아님 — 수정은 실 플로우 정합.

**AC10 (테스트):**
**Given** 변경 표면에 대해
**When** 테스트를 실행하면
**Then** ① **게이트 DB 레벨**(종결/취소/registered/soft-deleted 내원에 오더 생성 INSERT 거부·perform RPC 거부 PT409·미존재 PT404·정상 in_progress 통과·complete_examination는 게이트 없음 통과) ② **부분수행 정산 통합**(in_progress + 일부 perform + 일부 ordered → finalize → 수행분+진찰료만 청구·미수행 제외·내원 completed·`pending_orders_count` 정확) ③ **무회귀**(정상 오더·7.4/7.8/7.9·5.x) ④ **web**(부분수행 배지 렌더 조건·카운트 표시·전체수행 시 미표시)가 통과한다.

### 핵심 통찰 — 왜 "수행분만 정산"이 이미 구조적인가 (7.9 "수가 미발생" 구조성의 쌍대)

```
오더 상태머신(0015·forward-only·취소 RPC 없음):
  examinations/treatment_orders: ordered → performed → completed
  prescriptions:                 issued → dispensed (수행 개념 없음·fee 0·원외)
fee_items 발생 시점(0021 트리거):
  진찰료    = registered→in_progress (fee_on_encounter_start)
  검사·처치료 = ordered→performed     (fee_on_examination/treatment_performed)
  처방료    = 없음(원외 약제비 스코프아웃)
내원 상태머신: ... registered → in_progress → completed(only)   ← 부분수행도 completed 후 정산
```

- **수행분만 청구 = fee_items가 perform 시점에만 발생 → 미수행(`ordered`) 오더는 fee_item 0 → build_payment(fee_items 집계)가 자동으로 제외.** 7.9의 "취소=수가 미발생(in_progress 진입 전이라 fee 0)"과 **동일한 구조적 보장의 쌍대**(7.9=진입 전 0, 7.10=수행분만). **fee_item 삭제/필터 코드 작성 금지**(존재하지 않는 것을 제외하려 하지 말 것).
- **부분수행 정산 진입점 = 이미 존재.** in_progress + total>0(진찰료 ≥1) 내원은 billing-worklist(in_progress 노출·7.2/7.8)에 떠 있고, billing-detail에서 "결제·내원 완료" 버튼(7.4 finalize)이 이미 노출된다(7.9 메모: "in_progress(total>0)는 완료만·부분수행=7.10"). → **7.10은 새 정산 UI/경로를 만들지 않는다.** finalize가 그대로 부분수행을 정산하고 내원을 completed로 닫는다.
- **따라서 7.10의 실질 능동 작업 = ① 오더-by-내원상태 게이트(정산 종료/종결 내원에 신규 수가 변조 차단 — 무결성)** ② pending count 가시성 ③ 부분수행 정산 검증 테스트. ①이 핵심(deferred 5건이 FR-119를 지목).
- **게이트와 부분수행 정산의 관계(비자명):** 게이트는 부분수행 *정상 경로*를 막지 않는다. 정상 경로 = `in_progress 중 perform`(게이트 통과·fee 발생) → `finalize`(complete_encounter로 in_progress→completed). perform은 항상 completed *이전*(in_progress)에 일어남. 게이트가 막는 건 **completed/cancelled 이후의 perform**(정산액 사후 변조) — 이건 정상 경로에 없는 변조 시도다. ⇒ 게이트는 정산 무결성을 보호하면서 부분수행 정산을 깨지 않는다.
