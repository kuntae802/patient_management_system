# Story 8.2: 환자 포털 — 처방 · 검사 결과

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a **환자**,
I want **본인의 처방(복약 안내)과 검사 결과를 쉬운 말로 보기를**,
so that **복약법과 검사 결과를 스스로 이해한다**.

> **Story 8.1의 직속 후속 — "내 기록" 카드를 펼치면 나오는 상세를 채운다.** 8.1이 "본인 내원 카드 리스트(표시 전용) + 펼침 상세는 8.2"로 명시 이월했고, 컴포넌트를 **detail 블록을 끼울 구조**로 남겨 두었다. 8.2는 그 펼침 상세에 **처방(복약 쉬운 말) + 검사 결과 요약(정상/주의 플래그)** 을 올린다. 여전히 **읽기 전용 · 세션 uid 스코프(본인 외 0건) · 새 임상 데이터 생성 0**. 누적된 처방·검사 데이터를 환자 언어로 비춘다.

## Acceptance Criteria

**AC1 — 카드 펼침 → 처방·검사 상세 (FR-121, UX-DR17)**
**Given** 환자 세션에서 "내 기록" 탭의 내원 카드를
**When** 펼치면(탭/셰브런 토글)
**Then** 그 내원의 **처방 섹션(💊)** 과 **검사 결과 섹션(🧪)** 이 상세로 표시되고, 다시 누르면 접힌다(`aria-expanded`/`aria-controls`, ≥44px 터치 타깃, 색 비의존 글리프 `›`/`⌄`).

**AC2 — 복약 안내 쉬운 말 (FR-121)**
**Given** 펼친 내원에 발행 처방이 있을 때
**When** 처방을 렌더하면
**Then** 각 약마다 **약품명 · 복약 안내(쉬운 말) · 복용 기간(일수 칩)** 이 표시된다 — 복약 안내는 저장된 구조화 필드(`frequency`·`usage_instruction`·`dose`+`drugs.unit`)를 환자 문장으로 조립한다(예: "1일 3회, 매 식후 30분, 1정"). 임상 코드를 그대로 노출하지 않는다.

**AC3 — 검사 결과 요약 + 정상/주의 플래그 (FR-121, UX-DR20)**
**Given** 펼친 내원에 **완료된**(status='completed') 검사가 있을 때
**When** 검사 결과를 렌더하면
**Then** 각 검사마다 **검사명 · 쉬운 말 결과 요약 · 정상/주의 플래그**가 표시된다. 플래그는 **색 + 글리프 + 라벨**로 중복 인코딩한다(정상=그린·주의=앰버, 음영 단독 금지). 결과 요약·플래그가 없는(NULL) 검사는 안내문으로 우아하게 폴백하고, **판독 소견 원문(`findings`/`reading_conclusion`)은 환자에게 노출하지 않는다**(임상 서사·감사 마스킹 대상).

**AC4 — 임상 용어 환자용 풀어쓰기 (UX-DR23)**
**Given** 상세에 임상 정보가 있을 때
**When** 표시하면
**Then** 환자 언어로 풀어쓴다 — 진단 쉬운 말 부연(8.1 카드 헤더에 이미 표시) + 복약 안내(쉬운 말) + 검사 결과 요약(쉬운 말). 직원용 코드/약어/판독 서사를 그대로 보여주지 않는다.

**AC5 — self-scope 경계 · 빈/예외 상태 · 12시간**
**Given** RLS·세션 uid 경계가 강제되는 상태에서
**When** 상세를 받으면
**Then** **본인 내원의 처방·검사만** 반환되고(타인 내원 id 요청 시 404 — 존재/소유 구분 없음), 처방·검사가 **0건**이면 섹션을 생략하거나 "처방/검사 내역이 없어요" 안내를 보여준다. 시간은 12시간(오후 2:30)·KST·쉬운 말이 일관 적용되고, 신뢰 노트(8.1)는 상시 유지된다.

### 이 스토리의 본질 (8.1 위에 한 겹)
데이터·RLS·환자 셸이 **이미 다 깔려 있다**(8.1이 self-read 패턴을 확립). 8.2 = "본인 내원의 처방·검사를 환자 언어로 비추는 펼침 상세":
- **읽기 경계(self-scope)** = 기존 `prescriptions_select_self`/`prescription_details_select_self`/`examinations_select_self`(0015:345~374) RLS + `get_current_patient` 게이트 + `where p.auth_uid = $1 and e.id = $2` WHERE. 새 보안 메커니즘 0.
- **처방 데이터** = 전부 존재(`prescription_details`: dose·frequency·duration_days·usage_instruction + `drugs`: name·unit·coverage_type). **신규 컬럼 0** — 저장값으로 쉬운 말 조립.
- **검사 데이터** = `examinations`·`fee_schedules` 존재. **신규 = `patient_result_summary`/`patient_result_flag` 컬럼 2개**(8.1 진단 쉬운 말 컬럼과 동형 — 정상/주의 플래그의 구조적 원천).
- **실작업** = ① 검사 환자결과 컬럼+데모시드 ② self 내원상세 엔드포인트 ③ 카드 펼침 + 처방·검사 블록.

### 🔒 보안 경계 — service_role 현실 + 환자 비노출 데이터 (가장 중요)
**FastAPI는 service_role(로컬 postgres) 연결 → RLS를 우회**(`api/app/core/db.py:3` 주석). `_run_authed(sub)` 는 `auth.uid()` 만 주입할 뿐 RLS 필터를 적용하지 않는다.
- ⇒ 본인 스코프는 **반드시 쿼리의 `where p.auth_uid = $1`(=sub) 로 명시**. 추가로 **소유 검증** `and e.id = $2`(요청 encounter_id 가 본인 것인지) — 없으면 환자가 타인 encounter_id로 처방을 읽을 수 있다. 소유 미일치 → **404**(존재/비소유 구분 노출 금지).
- ⇒ **환자에게 절대 투영 금지(자유 임상 서사)**: `examinations.findings`·`reading_conclusion` = 판독 소견 원문(예 "HbA1c 7.8% — 목표 미달, 약물 조절 필요"). **감사 스냅샷 마스킹 대상**(`api/app/services/audit.py:_SENSITIVE_KEY` + `web/src/lib/admin/audit.ts:SENSITIVE_KEY` 양쪽 등록·0020 주석). 환자는 큐레이션된 `patient_result_summary` 만 본다. 스키마에서 `findings`/`reading_conclusion` 을 **응답 모델에 넣지 않는 것**으로 구조적 차단.
- **안티패턴 금지**: 클라이언트 Supabase 직접조회로 처방/검사를 읽지 말 것(컬럼 투영 통제 불가 → `findings` 같은 민감 컬럼이 새고, 약품명 조인이 RLS에 막힐 수 있음). 환자 임상 데이터의 큐레이션 투영은 FastAPI가 소유(8.1 동일).

### ⚠️ 데이터 함정 — frequency 는 코드가 아니라 한국어 (반드시 확인)
`0015_orders.sql:67` 컬럼 주석은 `frequency` 예시를 `'TID'`(=1일3회)로 적지만, **실제 데모 데이터(`demo_seed.sql:264~274`)는 이미 한국어 산문**이다:
- `frequency`: `'1일 3회'`·`'1일 2회'`·`'1일 1회'`·`'필요시'`
- `usage_instruction`: `'매 식후 30분'`·`'아침·저녁 식후'`·`'취침 전'`·`'아침 식전'`·`'발열 시'`
`frequency` 는 `text`(자유)·5.2 처방 패널이 입력하는 임의 문자열이다. ⇒ **TID→한국어 매핑 테이블을 만들지 말 것**(데이터와 불일치). 저장값을 **그대로 결합**해 복약 문장을 만든다: `formatDosage` = `[frequency, usage_instruction, dose+unit].join(', ')`(빈 필드 생략). 방어적으로 코드가 들어와도 원문 표시(조작·허위 금지). 목업의 "하루 1번, 아침 식사 후 한 알"은 더 구어적이지만 의미는 저장 필드가 이미 담고 있다.

### ⚠️ 데이터 현실 — 검사 수치·참조범위는 DB에 없다 (허위 금지)
목업(`key-patient-app.html:552~567`)은 검사 행에 **수치(142/90·185) + 정상/주의 + 요약**을 보여주지만, **DB에는 구조화된 검사 수치값·참조범위·정상/비정상 플래그가 없다**(`findings` 자유텍스트만). ⇒ **수치를 별도 구조 컬럼으로 만들지도, free-text에서 파싱하지도 말 것**(취약·허위 위험). 환자는 **검사명 + `patient_result_summary`(쉬운 말, 수치는 판독의가 요약문에 녹여 작성 가능) + `patient_result_flag`(정상/주의)** 를 본다. 구조화 수치의 부재는 하단 안내 노트("자세한 검사 수치…의원에 보관")가 덮는다(목업 L570). 이게 정직한 매핑이다.
