## Synthesis

A genuinely strong spine pair on the rubric axis: flow coverage, component pairing, state coverage, and inheritance discipline are near-airtight, and the spine/source/decision-log trace is exemplary — a consumer can source-extract the clinical surface cleanly. Two issues block a *clean* extract: the contrast deferral (EXPERIENCE.md punts to DESIGN.md, which states no WCAG target anywhere) and aggregate-nickname mockup linking instead of per-section filenames. On the rubric alone, everything else is polish.

The two adversarial reviewers shift that picture and the "polish" framing does not survive them. **Accessibility is THIN, not done**: the team solved colorblindness + cheap monitors thoroughly and declared accessibility complete, but there is no screen-reader/ARIA/semantics layer anywhere (the safety-critical allergy alert, 409 toasts, and the audited RRN reveal are all unannounced), keyboard operability is asserted but unspecified and contradicted by the mockups, and load-bearing text fails WCAG AA in several places — 4 criticals. **Clinical-safety is ADEQUATE but with 2 criticals**: most damningly, the flagship encounter-hub mock flags 김영희 as 아스피린-allergic while the patient app shows her a completed 아스피린 100mg prescription, and the spine has no allergy-vs-order cross-check (only FR-052 same-ingredient duplicate) — the flagship artifact demonstrates the exact accident the system is meant to prevent. The second clinical critical is printed-RRN on statutory forms bypassing the decrypt/audit gate with no print-PII policy. Net: extractable as a contract, not yet buildable as a safe, accessible clinical product until the six criticals and nineteen highs below are closed.

## CRITICAL (6)

- **[Accessibility]** — No ARIA / semantics / AT story anywhere (§5; all 6 mockups + EXPERIENCE.md Accessibility Floor). The only `aria-*` in any mockup is `aria-hidden` on decorative sparklines; the spine never mentions screen readers, AT, ARIA, or semantics. For an RRN/allergy/prescription/audit system this is the highest-impact failure. *Fix:* add a screen-reader/semantics layer to the spine; require real roles, labels, and live regions before any accessibility claim.
- **[Accessibility]** — Safety-critical allergy alert is not announced (§5; key-encounter-hub.html L626). The alert-row has no `role="alert"`/`aria-live` — the single most safety-critical element is silent to AT. *Fix:* allergy/safety alert must be `role="alert"` or a live region; meaningful status changes need polite live regions.
- **[Accessibility]** — 409 conflict toasts (and autosave) not announced to AT (§5; key-encounter-hub.html L985). The transition-conflict toast is a positioned `<div>` with no `role`/`aria-live`; a SR/low-vision user thinks the failed action worked. *Fix:* toasts → live region; 409 = assertive.
- **[Accessibility]** — Masked RRN reveal: control unlabeled + revealed value + audit event not announced (§5; reveal-btn, key-encounter-hub.html L657). No `aria-label` on the audited reveal control; revealed value never announced; second reveal (연락처) has no audit sub-label. *Fix:* `aria-label` ("주민번호 표시 — 조회 시 감사 로그 기록됨"); announce the revealed value via live region; keep the audit warning in the accessible name.
- **[Clinical-safety]** — Mockups prescribe a drug the same patient is flagged allergic to — no allergy↔order cross-check (§1; key-encounter-hub.html L631,746 + key-patient-app.html L540–546). 김영희 is 아스피린-allergic in the hub yet shown a completed 아스피린 100mg prescription; the only drug check named is FR-052 same-ingredient duplicate. The flagship artifact demonstrates the accident. *Fix:* add an order-add safety check (evaluate vs recorded allergies + current-med interactions, hard-stop / override-with-reason) as a spine pattern; fix the demo data.
- **[Clinical-safety]** — Printed 영수증 shows RRN — statutory print path bypasses the reveal/audit gate; no print-PII policy (§4; key-billing.html L704, L667). Full/masked RRN on a printed statutory form is either an un-audited reveal outside the "표시 + 감사기록" gate or a non-conformant document; "PDF 저장" is a leak vector. *Fix:* spec the print policy per statutory form; treat full-RRN rendering into a document as an audited, permission-gated reveal; watermark/limited-fields + warn on PDF-save.

## HIGH (19)

- **[Token completeness / Rubric]** — Contrast targets absent from the load-bearing color layer (§2; EXPERIENCE.md:143,149 → DESIGN.md Colors). EXPERIENCE.md defers contrast numbers to DESIGN.md, which states no WCAG level or ratio anywhere — a dangling promise on marginal brand-on-white combos. *Fix:* add to DESIGN.md Colors: "텍스트/상태 라벨은 배경 대비 WCAG 2.2 AA(본문 4.5:1, 큰 텍스트/UI 3:1); 검증 조합 = status-received-ink·text-muted·primary."
- **[Visual reference coverage / Rubric]** — Mockup linking is one aggregate nickname reference, not per-section by filename (§5; EXPERIENCE.md:69, DESIGN.md none). Six nicknames, zero filenames, not placed at each section; DESIGN.md has no mockup links at all. *Fix:* append the filename on each EXPERIENCE Component Patterns row and DESIGN Components row.
- **[Accessibility]** — Placeholder & disabled text fails contrast everywhere (§2; --text-disabled #97A9AA = 2.45:1 / 2.22:1). Used for search/picker placeholders, the 8px RRN "감사기록" sub-label, 마감 tags, disabled time text, captions — all below the 3:1 large-text floor. *Fix:* never use #97A9AA for meaningful text; floor placeholders at text-muted #54686A (5.89:1).
- **[Accessibility]** — Raw amber #BC7E12 as text/glyph fails at 3.43:1 (§2; DESIGN.md status-received token). The token contract invites an implementer to use non-ink amber for labels and ship a 3.43:1 label. *Fix:* forbid #BC7E12 as a text/glyph color on light surfaces; only `-ink` may touch text.
- **[Accessibility]** — Mockups are not keyboard-operable as built (§3; all mockups — nav/RBAC/slots/SOAP). Nav `<a>` with no href, RBAC checkboxes are `<span>`, slots are `<div>`, SOAP rows are non-editable `<div>`s — the promoted reference models an inaccessible structure. *Fix:* spine must state these become real `<button>`/`<input>`/role-bearing widgets; don't cite mockups as interaction truth.
- **[Accessibility]** — No tab order, no focus-visible spec, no grid keyboard model (§3; EXPERIENCE.md L138). The 22×5 RBAC matrix, slot grid, and SOAP ledger all need defined roving-tabindex/arrow-key models — unspecified. *Fix:* add a keyboard-interaction spec per complex component.
- **[Accessibility]** — Icon-only buttons have no accessible name (§5; tb-bell, sidebar toggle, collapsed nav, datepick arrows). Several rely on weak `title=` or hover-only CSS tooltips; many have nothing. *Fix:* `aria-label` on every icon-only control.
- **[Accessibility]** — RBAC matrix has no table semantics for AT (§5; key-rbac-matrix.html). No `<th scope>`, no header association, no per-cell accessible name — a 110-cell grid unusable with a screen reader. *Fix:* require `<table>` header scoping + per-checkbox name composed of row+column.
- **[Accessibility]** — Reduced motion entirely unaddressed and explicitly punted (§6; EXPERIENCE.md L231 `[OPEN] 모션`). Live row updates, sliding toasts, glowing live-dot, shimmering skeletons — none reference `prefers-reduced-motion`. *Fix:* commit now to honoring `prefers-reduced-motion`; don't defer past v1.
- **[Accessibility]** — 422 validation not associated to fields for AT (§6; key-encounter-hub.html L395,L1007). Good visual behavior, but no `aria-invalid`/`aria-describedby`/`aria-required`; error text won't be announced. *Fix:* require `aria-invalid` + `aria-describedby` on errored fields; announce the summary via live region.
- **[Accessibility]** — Patient app: no font-size floor, no zoom/OS-large-text honoring (§7; key-patient-app.html L84–87). Fixed-px body + fixed-height phone frame breaks reflow when an elderly user cranks zoom/OS text — the one thing the 58yo+ persona needs most. *Fix:* commit to relative units, reflow at 200% zoom, OS dynamic-type, a stated minimum body size.
- **[Clinical-safety]** — Allergy alert has no overflow/truncation behavior (§1; key-encounter-hub.html L634). A single flex row with a "상세 · 변경 이력" more-link silently degrades "can't-miss" to "the first few you can fit." *Fix:* never hide an allergen behind a more-link; wrap-all or danger-styled count chip that expands inline; severity-ranked first; collapsing forbidden.
- **[Clinical-safety]** — Order → 수행 tracking can lose an order silently — no overdue/unperformed detector (§1; DESIGN.md:277, key-encounter-hub.html L911,920,929). Nothing answers "which of today's orders are still un-performed?" — "누락 0" has no detector. *Fix:* add an outstanding/overdue 수행 indicator on the hub + worklists with an age threshold.
- **[Clinical-safety]** — No identity confirmation before high-risk writes (§2; key-encounter-hub.html L980, key-billing.html L636). 완료/처방/수납 fire with no "you are acting on 김영희, chart 00123 — confirm?"; the new-employee persona is exactly who acts in the wrong open tab. *Fix:* identity-bearing confirm (name + chart no) on 완료/처방 발행/수납/환자 삭제.
- **[Clinical-safety]** — Multiple-encounter / stale-tab safety is undefined (§2; EXPERIENCE.md IA L59, key-encounter-hub.html L820,987). Two encounter tabs; autosave writes silently to whichever fired last; 409 covers transitions, not stale-tab autosave into the wrong patient. *Fix:* a tab that lost the active-encounter lock must show a blocking banner and refuse autosave; consider single-active-encounter enforcement. Add to State Patterns.
- **[Clinical-safety]** — Concurrent performance/settlement race under-specified beyond "buttons disable" (§3; EXPERIENCE.md L129). "safe" for optimistic updates is undefined; under realtime lag a second nurse can act on a stale un-performed row. *Fix:* perform/settle = non-optimistic, server re-validate on submit, row locked on realtime receipt; tie to FR-093.
- **[Clinical-safety]** — Stale-realtime + critical action: spine recommends refresh but doesn't enforce it (§3; EXPERIENCE.md L124). "권장" is not a guard; a new employee will call/perform off stale data. *Fix:* when the realtime channel is stale, disable critical mutation buttons until reconnect/refresh.
- **[Clinical-safety]** — Phone reveal button has no audit label, unlike RRN (§4; key-encounter-hub.html L654,657, key-billing.html L505,508). Inconsistent PII discipline — phone reveal is either unaudited or unlabeled. *Fix:* audit contact/address/보험/보호자 reveals too and label consistently; state the rule once in PII 패턴.
- **[Clinical-safety]** — No spec forbids RRN/PII in URLs, deep links, realtime payloads, or PDF filenames (§4; EXPERIENCE.md L78,174). Forbidden only in logs/toasts/errors; silent on query strings, `postgres_changes` payload columns (RLS scopes rows not columns), query cache, and document filenames. *Fix:* PII never in URLs / realtime broadcast columns / filenames; confirm RLS column-level exclusion on the Supabase-direct read path.
- **[Clinical-safety]** — RBAC matrix applies sensitive permissions immediately with no confirmation (§5; DESIGN.md:280, EXPERIENCE.md L112, rbac mock L418–422,670). One stray click in 주민번호 표시 / 환자 삭제 / 권한 관리 / 감사 로그 조회 instantly grants it to a whole role; the "민감" pill marks but doesn't gate. *Fix:* sensitive toggles require a confirm step (ideally re-auth) naming the permission + role; immediate-apply only for non-sensitive cells.

## MEDIUM (23)

- **[Token completeness / Rubric]** — soap-ledger.badge-colors hardcodes hexes that are already tokens (DESIGN.md:137, prose 276). `S=#4F46C7 · O=#0E7C8E · A=#8A5D09 · P=#2C8466` = status-inprogress/primary/status-received-ink/status-done; literals bypass the resolver and desync on a later token edit. *Fix:* use `{colors.*}` refs.
- **[Component coverage / Rubric]** — button-ghost (the #1 next-action pattern) has no behavioral row (EXPERIENCE.md:106 mention only). Visual spec exists; behavioral row (disabled-during-mutation, 1 vs 2 buttons, transition gating) does not. *Fix:* add a `next-action button (다음-액션)` Component Patterns row.
- **[Accessibility]** — KPI strip dots are color-only, no glyph (§1; key-waiting-board.html L233–243). Inconsistent with the glyph-bearing A3 dots. *Fix:* add ○●◐✓✕ to KPI dots or drop the dot and lean on the label.
- **[Accessibility]** — Vitals "flag" abnormal state is color-only (§1; key-encounter-hub.html L286,689). A clinical signal riding on amber-ink text alone on the doctor's primary screen. *Fix:* add a glyph/label ("주의"/"↑") to flagged vitals.
- **[Accessibility]** — SOAP placeholder-vs-filled distinction is color-only (§1; key-encounter-hub.html L391). Empty section reads only as muted text. *Fix:* mark empty sections with a glyph/label ("비어 있음"), not just lighter text.
- **[Accessibility]** — Green 급여 pay-chip text on tint = 4.08:1; nav-active teal on tint = 4.29:1 (§2). Both sub-AA on a money/coverage screen and nav. *Fix:* darken the on-tint inks or move text onto white.
- **[Accessibility]** — Focus ring is weak (§2; --ring #2C9FB0). 3.14:1 vs white, 2.43:1 vs the adjacent border it sits against — may be indistinguishable from the resting border (WCAG 2.4.13). *Fix:* thicken (≥2px) and/or darken toward #0E7C8E.
- **[Accessibility]** — DESIGN.md states no explicit contrast ratios (§2). Circular cross-reference — each doc says the numbers live in the other; no AA claim, no per-pair table. *Fix:* add an explicit contrast table + AA conformance statement to DESIGN.md, flagging the failing pairs.
- **[Accessibility]** — Command palette (Ctrl K) is the one bright spot but over-leaned-on (§3). `aria-live` results are good, but it doesn't substitute for making the matrix/grid/ledger directly operable. *Fix:* keep it; still make complex components directly operable.
- **[Accessibility]** — Reveal / icon-compound controls need keyboard activation + focus-return spec (§3). After reveal, where does focus go; is the RRN announced? Unspecified. *Fix:* define keyboard activation + focus-return + announcement.
- **[Accessibility]** — Staff desktop targets are genuinely small (§4; key-waiting-board.html L297–299, etc.). Row-action ghost buttons ~24px, status dots 8px, reveal ~16px, RBAC cells 22px — at/below WCAG 2.5.8 24×24px. *Fix:* floor interactive desktop targets at 24×24px effective hit area; enlarge reveal + row actions.
- **[Accessibility]** — Required fields marked with a color-only asterisk (§6; dx-block L332, booking-peek L377). `.req{color:danger}` "*" is not `aria-required` and rides on color. *Fix:* add `aria-required` and an explicit "(필수)" floor.
- **[Accessibility]** — 403 disabled controls hide their reason from AT (§6; key-encounter-hub.html L452). Genuinely `disabled` buttons aren't focusable, so the hover-only `.ptip` reason is unreachable by AT. *Fix:* use `aria-disabled` (focusable) + reason in accessible name/`aria-describedby`, not a hover tooltip.
- **[Accessibility]** — `lang="ko"` set but mixed-language tokens lack pronunciation hints (§5; KCD codes, SpO₂, HbA1c, serif legal form). Minor. *Fix:* add lang hints where mixed.
- **[Clinical-safety]** — Order re-do lock (FR-093) stated for the encounter transition but not for individual order 수행 (§1; key-encounter-hub.html L905,914,923). Mockups still show an active "×" delete on every order item. *Fix:* show the performed/locked order state; gate delete once 수행완료 (deletion itself audited/blocked).
- **[Clinical-safety]** — Ctrl-K wrong-patient navigation has no disambiguation guardrail (§2; EXPERIENCE.md L60). Two 김영희s / a mistyped chart number lands on the wrong banner; compounds with no identity-confirm. *Fix:* search results show DOB / masked-RRN tail / last visit; destination re-states identity.
- **[Clinical-safety]** — 409/422/403 user-facing behavior is a genuine strength — no change needed (§3; EXPERIENCE.md L120–127,179–183). Logged as a strength for the record. *Fix:* none to the mapping itself.
- **[Clinical-safety]** — list/grid/search RRN masking unverified (§4). "마스킹 by DEFAULT on every surface" asserted but only demonstrated on detail banners; no list/grid/search mockup. *Fix:* state masking applies to all list/grid/search renders; lists never carry a per-row reveal.
- **[Clinical-safety]** — Patient-portal reads should be uid-scoped, not client-supplied id (§4b). RLS is the only backstop if the portal ever queries by a client-supplied patient_id. *Fix:* state portal reads are always scoped by session uid; belt-and-suspenders with RLS.
- **[Clinical-safety]** — 환자 삭제 / 수가 조정 have no confirmation/audit UX defined (§5; rbac L489,617). Permissions exist (soft-delete FR-203) but no confirm + reason-capture + audit flow. *Fix:* spec a destructive-action pattern (confirm + reason + audit before/after snapshot).
- **[Clinical-safety]** — Admin column locked all-on removes separation-of-duties (§5; rbac L454,671). A super-admin can grant permissions and read audit with no separation; append-only audit mitigates. *Fix:* confirm admin's own sensitive reveals/reads are themselves audited.
- **[Clinical-safety]** — Audit deterrence is inconsistent (§6). The "이 동작은 기록됨" affordance appears only on RRN reveal, not on phone reveal, print/PDF, or 완료/수납/삭제. *Fix:* make "this action is recorded" a consistent affordance on every sensitive/destructive action; surface a per-patient access history.
- **[Accessibility]** — Patient-app bottom-tab labels/icons are CSS-glyph `<div>`s, not real tabs (§7; gl-cal/gl-doc/gl-user). Same keyboard/semantics problem on the surface most likely to have AT users. *Fix:* real tab semantics + accessible names.
