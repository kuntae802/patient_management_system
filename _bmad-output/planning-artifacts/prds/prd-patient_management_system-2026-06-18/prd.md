# PRD: 환자 관리 시스템 (Patient Management System)

> 작성 완료(코칭 방식, 여정 주도). Finalize 검토 반영본.

## 1. 개요 (Overview)

**배경.** 수십 년 업력의 지역 중소병원이 **세대교체** 국면에 들어섰다. 외래 운영을 떠받쳐 온 베테랑들의 워크플로우(접수 순번·오더 인계·수가 코드·예약 조정)는 어디에도 적혀 있지 않고 그들의 머릿속과 손버릇 — **암묵지** — 위에서만 돌아갔다. 고참이 빠지고 신규 인력이 채워지자 접수·처치·수납·예약 전반에서 체계가 삐끗하기 시작했다. 클라이언트의 요청은 "EMR 하나 깔아 달라"가 아니라, **흩어진 암묵적 워크플로우를 사람이 바뀌어도 무너지지 않는 명시적·표준 운영 체계로 굳혀 달라**는 것이다.

**제품.** 환자 관리 시스템(PMS)은 환자의 한 번의 내원(Encounter)을 — 예약부터 수납까지 — 시스템이 단계별로 안내하는 **명시적·역할별 파이프라인**으로 만드는 외래 운영 관리 앱이다. 워크플로우가 베테랑의 기억이 아니라 시스템 안에 살게 하여, 사람이 바뀌어도 절차가 그대로 굴러가게 한다. 진료 상태머신이 다음 할 일을 안내하고, 6개 역할 화면이 각자의 단계를 규정하며, 오더→수행 추적·표준 마스터·SOAP 기록·RBAC/RLS/감사로그가 절차·책임·안전·신뢰를 시스템에 새긴다.

**설계의 중심 인물.** 아직 암묵지를 체득하지 못한 **신규 직원**이다. 모든 화면은 "이 사람이 선배 없이도 제대로 일할 수 있는가"를 기준으로 만든다.

**범위 한정.** 외래(Outpatient) 한 줄기를 끝까지 완성한다(외래 도메인 약 26개 테이블). 입원(Inpatient)은 차기 단계로 분리. 6역할(직원 5: 원무·의사·간호사·방사선사·관리자 + 환자 1). 직원은 데스크톱 웹, 환자는 모바일 앱(APK), 백엔드·인증은 Supabase.

**성공 합격선.** 6역할을 가로지르는 **골든 패스**(§3 UJ-1)가 end-to-end로 끊김 없이 돌고, 신규 직원이 선배 없이 핵심 업무를 완주한다.

## 2. 용어집 (Glossary)

| 용어 | 정의 |
|---|---|
| **내원 (Encounter)** | 환자의 한 번의 방문 단위. 예약→접수→진행중→완료의 상태를 가지는 워크플로우의 허브. 파이프라인의 중심 엔티티. |
| **진찰** | 의사가 내원 중 수행하는 진료 행위(문진·검사·평가). 내원의 한 단계이지 내원 자체가 아니다. |
| **진료기록 (SOAP)** | 진찰 결과로 작성되는 임상 기록. 한 내원에 1:N. 내원과 별개 엔티티. |
| **진료과 / 진료실** | 조직 단위(정형외과 등) / 물리적 공간. 마스터 데이터. |
| **진료비 / 수납** | 내원에서 발생한 수가의 정산. |
| **오더** | 의사의 지시(처방·검사·처치). 지시 → 수행(직역) → 완료의 하위 생명주기를 가진다. |
| **수가 자동발생** | 진찰·수행된 오더에 대응하는 수가 항목이 수납에 자동 적재되는 것(§FR-116). |
| **마스터 데이터** | 진료과·진료실·진단(KCD)·수가(EDI)·약품 등 표준 코드의 단일 진실 공급원. |

> "진료"는 문맥에 따라 내원/진찰/진료기록을 가리킬 수 있어, 본 PRD는 위 구분 용어를 사용한다.

## 3. 사용자 여정 (User Journey)

> 골든 패스(합격선)를 이름표 주인공이 등장하는 하나의 서술로 적는다. 각본이 아니라, 다운스트림(아키텍처·스토리)이 "화면이 이 순서로 이어지는가"를 읽어내는 설계 기준이다.

**UJ-1 — 한 번의 내원, 6역할을 가로질러.**

등장인물: 환자 **이수진**(32, 어깨 통증), 신규 원무 **정해린**(입사 2주차), 정형외과 **김도현** 과장, 신규 간호사 **한지우**(입사 3주차), 방사선사 **오민재**, 관리자 **최원장**.

1. **예약(환자).** 수진은 퇴근길에 앱에서 정형외과 김도현 과장의 빈 슬롯을 골라 예약한다. 3일 전·1일 전 SMS 리마인더가 온다.
2. **접수(신규 원무).** 내원일, 수진이 도착하자 해린은 예약 목록에서 수진을 찾아 접수한다 — 내원이 '접수' 상태가 되고 정형외과 대기열에 오른다. 해린은 순번을 외울 필요 없이, 시스템이 알려주는 다음 호출 환자만 부른다.
3. **진료·오더(의사).** 호출된 수진이 진료실로. 김 과장은 과거 이력과 (간호가 미리 잰) 활력징후·환자 임상 프로필을 한 화면에서 보고, SOAP로 기록하며 평가(A)에 진단(KCD)을 붙인다. 어깨 X-ray, 소염제 처방, 물리치료 처치를 오더한다 — 각 오더가 '지시' 상태로 해당 직역에 전달된다.
4. **수행(신규 간호·방사선).** 한지우는 처치 워크리스트에서 물리치료 오더를 받아 수행 기록을 남긴다(이미 수행된 건 다시 못 누른다 → 중복 방지). 오민재는 촬영 워크리스트의 X-ray를 찍고, 영상은 스토리지에 저장(DB엔 URL), 김 과장이 판독 소견을 적어 검사 오더가 완료된다.
5. **수납(원무).** 진찰·수행이 끝나면 수가가 자동 적재돼 수납 건이 생긴다. 해린은 후수납으로 정산하고 표준 「진료비 계산서·영수증」·세부산정내역서를 출력, 원외처방전을 발급한다 — 베테랑만 알던 코드를 몰라도 시스템이 항목을 채워준다. 내원이 '완료'된다.
6. **운영(관리자).** 최원장은 관리자 화면에서 한지우에게 권한을 부여하고, 오늘의 내원·매출·노쇼 현황을 보며, 민감정보 접근을 감사로그로 확인한다.

수진은 본인 앱에서 진료·처방·영수증 내역을 조회한다(본인 데이터만 — RLS 강제).

## 4. 내원 파이프라인 (상태 흐름)

> 워크플로우를 사람의 기억이 아니라 시스템에 새기는 척추. 모든 외래 업무는 하나의 내원(Encounter)이 정해진 상태를 따라 흐르는 것으로 모델링된다.

**내원 상태:** `예약(Scheduled)` → `접수(Registered)` → `진행중(In-progress)` → `완료(Completed)` · 예외: `취소(Cancelled)` / `노쇼(No-show)`

- **예약** — 환자(앱) 또는 원무가 슬롯을 예약.
- **접수** — 내원일 도착·접수, 진료과 대기열 등록.
- **진행중** — 의사 진찰 시작. 이 구간에서 오더가 지시 → 수행으로 흐른다.
- **완료** — 모든 수행·수납 종료(후수납 기본).
- **취소 / 노쇼** — 예약 취소, 미방문. 수가 미발생으로 종결(§FR-118).

**오더 하위 생명주기(유형별).**
- **처방(약):** 발행 → 발급(원외 약국이 조제, 시스템 내 수행자 없음).
- **검사·영상:** 지시 → 수행 → 판독/결과 기록 → 완료.
- **처치:** 지시 → 수행 → 완료.

확정된 진단·수행 오더는 수가 자동발생의 근거가 되어 수납으로 이어진다(§FR-116).

## 5. 기능 요구사항 (Functional Requirements)

> 기능별로 묶고, 전역 고유 번호(FR-NNN)를 부여한다. 번호는 안정적이며 중간 번호는 향후 삽입을 위해 비워 둔다.

**추적성 — 고통 → 7기둥 → FR.** 각 FR 그룹은 브리프의 문제(고통) 하나와 솔루션 기둥에 대응한다.

| 고통(문제) | 기둥 | 대응 FR / 섹션 |
|---|---|---|
| 접수·대기 혼선 | ① 명시적 진료 파이프라인 | §4, FR-020~023 |
| 역할 혼선·구두 의존 | ② 역할별 화면 + RBAC | FR-210~215, 각 역할 FR 그룹 |
| 처치 중복·누락(환자 안전) | ③ 오더→수행 추적 | FR-060~061, 070, 080~081, 090~094, 100~103 |
| 수납·코드 오류 | ④ 단일 진실 마스터 | FR-200~203 |
| 기록 분실·추적 불가 | ⑤ 표준 기록 자동화 | FR-040~042, 113~114 |
| 예약 혼선·원무 과부하 | ⑥ 멀티플랫폼 + 환자 포털 | FR-001, 010~016, 120~122 |
| 민감정보 보안 | ⑦ 보안·신뢰 | FR-240~243 |

### 기능: 환자 신원·등록

- **FR-001** 환자는 모바일 앱에서 직접 회원가입(본인인증)하여 환자 계정을 생성할 수 있다.
- **FR-002** 원무 직원은 앱 미사용 환자(전화·방문·고령자)의 환자 레코드를 직접 생성할 수 있다(`auth_uid` 미설정).
- **FR-003** 원무가 생성한 환자가 이후 앱 가입 시, 본인인증으로 기존 레코드와 자동 연결된다(중복 환자 방지).
- **FR-004** 원무·의사는 환자의 임상 프로필(혈액형·알레르기·기저질환·복용약·특이사항)을 입력·갱신한다.
- **FR-005** 의사는 진료 화면에서 환자 임상 프로필을 조회한다(처방·처치 안전 참조).
- **FR-006** 시스템은 환자 보호자 정보(성명·연락처·관계)를 기록한다.

### 기능: 예약

- **FR-010** 환자는 앱에서 진료과·의사·가능 시간 슬롯을 조회하고 예약할 수 있다.
- **FR-011** 원무 직원은 전화·방문 환자를 대신해 예약을 생성·변경·취소할 수 있다.
- **FR-012** 시스템은 의사 근무표·휴진 예외를 반영해 예약 가능 슬롯만 노출한다.
- **FR-013** 시스템은 동일 슬롯의 더블부킹을 차단한다(오버부킹은 정책 설정 시 허용).
- **FR-014** 시스템은 예약 3일 전·1일 전 SMS 리마인더를 발송하고 발송 이력을 기록한다.
- **FR-015** 시스템은 환자별 노쇼 횟수를 기록하고, 임계치(기본 2회) 초과 시 예약을 제한한다.
- **FR-016** 의사 휴진 등록 시, 영향받는 예약을 표시하고 재배정·안내를 지원한다.

### 기능: 접수·대기

- **FR-020** 원무 직원은 도착한 예약 환자를 접수하여 내원(Encounter)을 생성하고 '접수' 상태로 전환, 진료과 대기열에 등록한다.
- **FR-021** 예약 없이 방문한 환자(walk-in)도 원무가 즉석 접수할 수 있다.
- **FR-022** 시스템은 진료과·진료실별 실시간 대기 현황과 순번을 제공한다.
- **FR-023** 시스템은 순번에 따라 "다음 호출 환자"를 안내하고 호출 상태를 기록하여, 중복 호출·누락을 방지한다.

### 기능: 진찰

- **FR-030** 의사는 접수 완료된 환자의 진료 대기열을 조회하고, 진찰을 시작하여 내원을 '진행중' 상태로 전환한다.
- **FR-031** 의사는 환자의 과거 내원·진단·처방·검사결과 이력을 타임라인/요약으로 한 화면에서 조회한다.
- **FR-032** 의사는 해당 내원의 간호 활력징후 등 사전 입력 데이터를 진료 화면에서 확인한다.

### 기능: SOAP 진료기록·진단

- **FR-040** 의사는 진료기록을 SOAP(주관적·객관적·평가·계획) 형식으로 작성·저장한다.
- **FR-041** 하나의 내원에 복수의 진료기록(1:N)을 남길 수 있다.
- **FR-042** 의사는 평가(A)에 진단을 KCD 진단코드 마스터에서 선택해 부착하고, 주진단/부진단을 구분한다.

### 기능: 처방 오더(약)

- **FR-050** 의사는 약품 마스터에서 약을 선택해 처방전을 발행한다(처방전 헤더 + 처방상세 라인: 약품·용량·횟수·일수·용법).
- **FR-051** 처방은 해당 진단(A)과 연결되어 근거가 기록된다.
- **FR-052** 시스템은 동일 성분 중복 처방 시 경고를 제공한다(약물 상호작용 DB 등 그 이상은 범위 밖).

### 기능: 검사·영상 오더

- **FR-060** 의사는 진단검사·영상검사를 오더한다. 오더는 지시 의사를 기록하고 '지시' 상태로 생성된다.
- **FR-061** 영상검사 오더는 방사선사 워크리스트로, 진단검사(검체) 오더는 간호 워크리스트(채취 후 외부 의뢰)로 전달된다. 외부 의뢰 결과는 기록으로 반영한다.

### 기능: 처치 오더

- **FR-070** 의사는 처치를 오더한다. 오더는 지시 의사를 기록하고 '지시' 상태로 생성되어 간호 워크리스트로 전달된다.

### 기능: 오더 공통(지시→수행)

- **FR-080** 모든 오더는 유형별 생명주기를 따른다 — 처방: 발행→발급(원외 약국, 시스템 내 수행자 없음); 검사·영상: 지시→수행→판독/완료; 처치: 지시→수행→완료. 모든 오더는 지시자·수행자·시각이 기록된다.
- **FR-081** 확정된 진단·수행 오더는 수가 자동발생의 근거가 되어 수납 정산에 반영된다.

### 기능: 간호 — 활력징후·처치 수행

- **FR-090** 간호사는 자신의 처치 워크리스트(지시된 처치 오더)를 조회한다.
- **FR-091** 간호사는 활력징후(혈압·맥박·체온·호흡수·SpO2 등)를 측정·기록한다(활력징후 전용 기록).
- **FR-092** 간호사는 지시된 처치 오더를 수행 처리하고 처치기록(수행자·시각·내용)을 남긴다 → 해당 오더가 '수행' 상태로 전환된다.
- **FR-093** 시스템은 이미 수행된 오더의 재수행을 막아 처치 중복·누락을 방지한다.
- **FR-094** 간호사는 오더 없이도 일상 간호기록을 남길 수 있다(처치 오더 연결은 선택).

### 기능: 방사선 — 영상검사 촬영·판독

- **FR-100** 방사선사는 촬영 워크리스트(지시된 영상검사 오더)와 대기 목록을 조회한다.
- **FR-101** 방사선사는 촬영을 수행 처리하고, 영상 자료는 스토리지에 저장한 뒤 URL만 DB에 연결한다.
- **FR-102** 의사(판독의)는 영상 판독 소견을 기록하고, 해당 검사 오더가 완료 처리된다.
- **FR-103** 시스템은 검사장비 목록·상태를 표시하여 촬영 배정·가용성 확인에 활용한다.

### 기능: 수납·정산

- **FR-110** 시스템은 진찰·오더에서 자동발생한 수가 항목을 집계하여 수납 건을 생성한다(수납 헤더 + 수납상세 라인: 수가 항목·횟수·일수·금액).
- **FR-111** 시스템은 급여(본인부담/공단부담)·비급여를 구분하여 본인부담금을 산정한다.
- **FR-112** 원무 직원은 수납을 처리하고(결제 수단·금액 기록) 내원을 '완료' 상태로 전환한다.
- **FR-113** 시스템은 표준 「진료비 계산서·영수증」을 출력한다(대분류 항목, 급여/비급여 구분, 본인부담총액·이미 납부한 금액·납부할 금액 합계 3행).
- **FR-114** 시스템은 「진료비 세부산정내역서」를 출력한다(라인: 항목분류·일자·코드·명칭·단가·횟수·일수·금액·본인부담·공단부담).
- **FR-115** 원무 직원은 원외처방전을 출력·발급한다.
- **FR-116** 시스템은 수가 자동발생 규칙에 따라 항목을 적재한다 — 진찰료는 진찰 시, 검사·처치·영상 수가는 해당 오더 수행 완료 시, 약제비는 처방 발행 시 발생한다. (행위·진단별 수가 코드 상세 매핑은 마스터·시드로 정의)
- **FR-117** 시스템은 수납 정책(후수납 기본 / 선수납 옵션)을 지원한다. 선수납 설정 시 접수 후 선결제, 진료 후 차액 정산.
- **FR-118** 취소·노쇼로 종결된 내원은 수가 미발생으로 처리한다(노쇼 수수료 정책 적용 시 별도 항목으로 부과 가능).
- **FR-119** 부분 수행(일부 오더만 수행 후 이탈) 내원도 수행된 항목까지 수납·정산할 수 있다.

### 기능: 환자 포털 — 본인 내역 조회

- **FR-120** 환자는 앱에서 본인의 내원 이력(예약·진찰·진단)을 조회한다.
- **FR-121** 환자는 본인의 처방·검사 결과 내역을 조회한다.
- **FR-122** 환자는 본인의 수납 내역·영수증을 조회한다.

## 6. 횡단 역량 · 운영 관리

> 한 번의 내원 흐름이 아니라 시스템 전체를 가로지르는 역량과, 이를 운영하는 관리자 기능.

### 기능: 마스터 데이터 관리

- **FR-200** 관리자는 진료과·진료실 마스터를 관리(생성·수정·비활성)한다.
- **FR-201** 관리자는 진단(KCD)·수가(EDI 행위)·약품 마스터를 관리하며, 각 코드는 버전·유효기간(발효/만료)을 갖는다. _[스키마 보강 필요: 발효/만료 컬럼 — 다운스트림]_
- **FR-202** 모든 임상·정산 입력은 마스터에서 선택하며, 비표준 자유 입력을 제한하여 단일 진실(single source of truth)을 강제한다.
- **FR-203** 마스터 데이터는 비활성(soft delete)으로 처리하여, 과거 기록이 참조하는 코드의 무결성을 유지한다.

### 기능: 역할·권한 관리 (RBAC)

- **FR-210** 시스템은 역할과 권한(`리소스.동작` 코드)을 N:M으로 관리한다.
- **FR-211** 관리자는 관리자 화면에서 역할별 권한을 체크박스로 토글한다(코드 수정 없이 즉시 반영).
- **FR-212** 직원·환자 신원은 분리 프로필로 구분되며(직원=`사용자`(id=auth uid), 환자=`환자.auth_uid`), 로그인 시 uid 소속 테이블로 직원/환자를 분기한다.
- **FR-213** 화면·기능·데이터 접근은 사용자의 역할 권한에 따라 제어된다(권한 없는 기능은 비노출 또는 거부).
- **FR-214** 관리자는 직원 계정을 생성하고 역할·소속 진료과·면허번호 등 프로필을 관리한다.
- **FR-215** 관리자는 직원의 재직상태(재직·휴직·퇴사)를 관리하며, 휴직·퇴사 시 접근을 차단한다.

### 기능: 근무표·스케줄 관리

- **FR-220** 관리자는 의사별 근무표(요일·시간대·진료실)를 등록·관리한다.
- **FR-221** 관리자는 휴진·예외(휴가·학회 등)를 등록하며, 이는 예약 가능 슬롯(FR-012)·휴진 재배정(FR-016)의 근거가 된다.

### 기능: 운영 통계·대시보드

- **FR-230** 관리자는 운영 현황(일별 내원·대기·매출·노쇼율 등) 대시보드·통계를 조회한다.

### 기능: 도움말·온보딩 가이드

> 신규 직원이 자기 업무 화면 안에서 선배 없이 기능을 익히도록 돕는 온보딩 보조. 제품 핵심(외래 운영)이 아닌 보조 역량. (Epic 1~8 구현 완료 후 추가 — 2026-06-25)

- **FR-250** 로그인한 사용자는 자신의 셸 안에서 도움말 페이지에 접근한다(인증 필요·공개 아님).
- **FR-251** 도움말은 현재 로그인 계정에 **실제로 표시되는 메뉴만** 안내한다(권한·역할로 가려진 다른 메뉴는 미노출). 상단 고정 네비게이션바가 그 계정의 메뉴 인덱스가 되어, 선택 시 해당 메뉴 안내로 이동한다.
- **FR-252** 각 메뉴 안내는 실제 화면 스크린샷·동작 요소 번호 하이라이트·단계별 설명으로 구성된다.
- **FR-253** 현재 계정이 사용하는 메뉴를 빠짐없이 수록하여, 스크롤만으로 전체 기능을 파악할 수 있다.

### 기능: 보안·신뢰 (횡단 강제)

- **FR-240** 행 수준 보안(RLS): 환자는 본인 데이터만, 직원은 역할별 접근만 가능하도록 데이터베이스가 직접 강제한다.
- **FR-241** 주민등록번호 등 민감정보는 pgcrypto로 암호화 저장하며, 키는 Supabase Vault에 보관한다(코드·DB에 키 미보관).
- **FR-242** 주요 작업(생성·수정·삭제 및 민감정보 조회 등)은 행위자·시각·대상·동작과 함께 감사로그로 기록된다.
- **FR-243** 관리자는 감사로그를 조회·필터링(행위자·기간·대상별)한다.

## 7. 비기능 요구사항 (Non-Functional Requirements)

> 시스템이 *얼마나 잘·안전하게* 동작하는가. 보안 강제 동작 자체는 FR(240~243)에 명시했고, 여기서는 품질·제약 측면을 모은다.

### 성능
- **NFR-001** 주요 화면(대기열·진료·수납)의 일반 조회 응답은 통상 2초 이내를 목표로 한다. _[데모 환경 기준 목표치]_
- **NFR-002** 대기 현황·워크리스트는 5초 이내로 갱신된다(실시간 구독 또는 폴링).

### 플랫폼·호환성
- **NFR-010** 직원용 클라이언트는 데스크톱 웹(최신 Chromium 계열 브라우저)에서 동작한다.
- **NFR-011** 환자용 클라이언트는 Android 모바일 앱(APK)으로 배포한다.
- **NFR-012** 백엔드·인증·스토리지는 Supabase(PostgreSQL)를 사용한다.

### 보안·프라이버시
- **NFR-020** 모든 클라이언트–서버 통신은 TLS(HTTPS)로 암호화한다.
- **NFR-021** 인증은 Supabase Auth를 사용하며, 최소권한 원칙(RBAC)에 따라 접근을 제한한다.
- **NFR-022** 개인정보보호 관련 표준의 "형태"(암호화·접근통제·감사)는 충실히 모사하되, 공식 인증(EMR 인증·개인정보보호법 세부 규제 완결성)은 범위 밖임을 명시한다.

### 가용성·신뢰성
- **NFR-030** 데이터는 Supabase 관리형 백업에 의존하며, 정기 백업을 전제한다.
- **NFR-031** _[ASSUMPTION]_ 과제·데모 환경 기준 별도 SLA를 두지 않는다(상용 배포 시 정의).

### 데이터 무결성
- **NFR-040** 내원·오더의 상태 전이는 정의된 규칙(상태머신)만 허용한다(역행·건너뛰기 방지). 취소·노쇼·부분수행 경로는 FR-118~119에 정의된 바를 따른다.
- **NFR-041** 임상·정산 데이터는 마스터 참조 무결성을 유지하며, 다단계 작업(수납 생성 등)은 트랜잭션으로 원자성을 보장한다.
- **NFR-042** 감사로그는 추가 전용(append-only)으로, 사후 변조가 어렵도록 한다.

### 사용성 (신규 직원 친화성 = 설계 원리)
- **NFR-050** 각 진료 단계 화면은 현재 내원에 대해 다음 수행 가능한 작업을 명시적으로 제시한다(다음 단계 안내/버튼).
- **NFR-051** 각 역할 사용자는 자신의 권한 범위 내 기능만 화면에서 접근하며, 핵심 업무 플로우는 역할별 화면에서 외부 안내 없이 완결할 수 있다.
- **NFR-052** 모든 UI와 표준 출력 문서는 한국어로 제공한다.

### 확장성
- **NFR-060** 데이터 모델은 진료(Encounter)를 허브로 하여, 향후 입원(Inpatient) 갈래 확장을 수용할 수 있도록 설계한다(진료/진료기록 1:N 등).

## 8. 성공 지표 · 범위 · 미해결 질문

### 8.1 성공 지표

**① 제품 성공 신호 (배포 시 측정)** — 각 신호는 문제의 고통 하나에 대응.

| 신호 | 측정 | 겨냥하는 고통 |
|---|---|---|
| 신규 직원이 선배 없이 첫 주 내 핵심 업무 수행 | 온보딩 소요 시간 ↓ | 암묵지 의존 |
| 처치 중복·누락 0 | 오더 대비 수행 불일치 건수 | 환자 안전 |
| 수납 정산 오류 ↓ | 정정 건수 / 청구 불일치율 | 코드 오류 |
| 더블부킹·노쇼 누락 0 | 예약 충돌 건수 | 예약 혼선 |
| 환자 본인 외 데이터 접근 0 | RLS 위반 차단 | 보안 |
| 베테랑 부재 시 업무 지속 | 버스 팩터 > 1 | 조직 리스크 |

**② 과제 성공 신호 (평가 포인트 직결)** — ①요청 구체화 ②시스템 구조 변환(고통↔FR/스키마 매핑 — §5 추적성 표) ③구현(골든 패스 UJ-1 end-to-end 동작) ④사용자 친화(신규 사용자가 가이드만으로 완주).

**카운터 지표 (지표를 좇다 잃지 말아야 할 것).**
- 파이프라인 강제가 베테랑의 건당 처리 속도를 떨어뜨리지 않는가(건당 처리시간).
- 마스터 강제가 예외 항목 처리를 막아 우회·자유 입력 요구를 늘리지 않는가.
- 입력 부담 증가로 기록 누락·형식적 입력이 늘지 않는가.

### 8.2 범위

**포함 (외래 Outpatient 전체).** 환자·내원(접수 흡수)·SOAP 진료기록·진단, 처방(처방전·상세·약품마스터), 검사(검사·장비), 간호(처치오더·기록·활력징후), 예약·근무/예외 스케줄, 수납(수납·상세·수가마스터 + 표준 진료비 문서), 마스터(진료과·진료실·약품·진단·수가), 감사로그 — 외래 도메인 약 26개 테이블. 6역할 화면 + RBAC + RLS + pgcrypto. 직원 데스크톱 웹 + 환자 모바일 앱(APK), 백엔드 Supabase.

**제외 (1차 밖, 차기 후보).**
- **입원(Inpatient) 관리** — 병동·병상·재원·회진·입원료(별도 허브 필요, 외래 안정화 후).
- **외부 실연동** — 보험청구(EDI) 실제 전송, 처방전 약국 전송, 검사 외부 의뢰 자동화.
- **인증·규제 완결성** — EMR 공식 인증, 개인정보보호법 세부 준수는 "형태"만 모사.
- **외래 외 진료 형태** — 응급·건강검진 등.

## 9. 솔직한 한계

- **소프트웨어는 만병통치가 아니다.** 워크플로우를 시스템으로 강제하면 현장의 유연성이 일부 줄고, 신규 직원 정착에는 여전히 교육·문서화가 함께 필요하다. 이 앱은 그 부담을 줄일 뿐 없애지 않는다.
- **학습용 모사다.** EMR 공식 인증·개인정보보호 세부 규제·보험청구(EDI) 연동까지 완전히 충족하지는 않는다. 표준의 "형태"는 충실히 따르되, 인증 수준의 완결성은 범위 밖이다.
- **폭과 깊이의 트레이드오프.** "정석을 다 넣자"는 야심은 넓은 범위를 만든다. 단일 개발자·과제 일정 안에서는 각 기능의 깊이가 제한될 수 있어, 범위를 외래로 명확히 한정했다(§8.2).

> 한계를 숨기지 않는 것 자체가 프리랜서로서 신뢰를 만드는 방식이다.

## 10. 비전

**가까운 미래 (외래 다음).** 외래가 안정되면 보류해 둔 **입원(Inpatient)** 으로 확장한다 — 병동·병상·재원·회진·입원료. 같은 진료 허브 철학을 입원 갈래로 넓히고, 보험청구(EDI)·약국 처방 전송 같은 외부 연동과 운영 대시보드로 깊이를 더한다.

**이 시스템이 되려는 것.** 궁극의 목표는 병원의 **"운영 기억(operating memory)"** 이 되는 것이다. 베테랑의 머릿속에 흩어져 있던 수십 년치 워크플로우가 시스템에 영구히 새겨지면, 사람이 몇 번 바뀌어도 병원은 첫날처럼 굴러간다. **세대교체가 더 이상 위기가 아니라 그저 또 한 번의 평범한 인수인계가 된다** — 그것이 이 프로젝트가 푸는 문제의 끝이다.
